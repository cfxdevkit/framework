## Context

The `repo-check` CLI tool is currently non-functional due to a combination of lint errors and malformed JSON output. Specifically:

- **Lint errors** prevent successful execution: missing exports in `commands.js`, `context.js`, and `types` cause import failures during lint and check steps.
- **Format step failure** stems from malformed JSON output generated by the `repo check docs --json` command, where an invalid minus-digit sequence (e.g., `-` not followed by a digit) appears in the output stream, breaking JSON parsing.

These issues are tightly coupled: the lint errors block the CLI from running correctly, and the malformed JSON output prevents downstream consumers (e.g., CI pipelines, tooling integrations) from parsing structured results. The current state is `4 error(s), 1 warning(s), 3/8 passed`, with the `format`, `lint`, and `check` steps all failing.

Stakeholders include developers relying on `repo-check` for repository validation, CI/CD pipelines consuming its structured output, and maintainers responsible for CLI reliability.

## Goals / Non-Goals

**Goals:**
- Restore full functionality to the `repo-check` CLI by resolving all lint errors and JSON output issues.
- Ensure the `format` step produces valid JSON that can be parsed by standard tools (e.g., `jq`, `JSON.parse`).
- Fix missing exports in `commands.js`, `context.js`, and `types` to satisfy import requirements.
- Maintain backward compatibility with existing CLI interfaces and output schemas.
- Enable reliable execution of `pnpm run format`, `pnpm run lint`, and `pnpm run check`.

**Non-Goals:**
- Refactoring unrelated modules or introducing new features beyond this remediation.
- Changing the CLI’s public interface or output schema beyond fixing the malformed JSON.
- Adding new dependencies or modifying build tooling (e.g., ESLint, Prettier) unless strictly required to fix the lint errors.

## Decisions

### 1. Prioritize lint fixes to unblock execution before JSON output fixes
**Rationale:** Lint errors prevent the CLI from running at all, making JSON output fixes impossible to validate. By resolving missing exports first, we ensure the CLI can execute and produce output for debugging the JSON issue.

**Implementation approach:**
- Audit `commands.js`, `context.js`, and `types` to identify missing exports (`runStructuredRepoCommand`, `findWorkspaceRoot`, `getGitNexusSnapshot`, `writeJson`, and type definitions).
- Add missing named exports and ensure type definitions are correctly exported (e.g., `export type { RepoCheckHotspotsResult }`).
- Run `pnpm run lint` to confirm all lint errors are resolved before proceeding.

### 2. Fix malformed JSON output by sanitizing minus-digit sequences
**Rationale:** The JSON error (`Minus must be followed by a digit`) suggests an unescaped minus sign (e.g., `-` used as a placeholder or separator) is being emitted in structured output. This is likely in the `docs` check result formatting logic.

**Implementation approach:**
- Locate the code path responsible for generating JSON output in the `docs` check (likely in `docs.js` or a shared formatter).
- Identify where a raw minus sign (e.g., `-` used for bullet points or placeholders) is being included in JSON-serializable data.
- Replace or escape such characters (e.g., use `–` (en-dash) for visual separators, or wrap non-JSON-safe strings in a `text` field with a `raw` fallback).
- Validate output using `pnpm run tooling -- -- repo check docs --json | jq .` to confirm valid JSON.

**Alternative considered:** Using a custom JSON serializer (e.g., `safe-json-stringify`) to handle edge cases. *Rejected* due to added dependency and over-engineering; the issue is localized and fixable via input sanitization.

### 3. Use a single commit to fix lint and JSON issues together
**Rationale:** Since the issues are interdependent, splitting them risks incomplete rollbacks or CI regressions. A combined fix ensures atomic resolution.

**Implementation approach:**
- Create a single PR with:
  - Lint fixes (exports, types).
  - JSON sanitization (minus-digit fix).
  - A new test case for `repo check docs --json` to prevent regression.
- Run `pnpm run check` end-to-end to validate.

### 4. Preserve existing output schema while fixing formatting
**Rationale:** Changing the schema would break consumers; only the *formatting* of existing fields needs correction.

**Implementation approach:**
- Ensure all output fields retain their original names and structure.
- Only modify how values are derived (e.g., replace `"-"` with `"–"` in human-readable fields, not in machine-readable keys).

## Risks / Trade-offs

- **[Risk]** Lint fixes may inadvertently expose new runtime errors (e.g., missing dependencies in newly exported functions).  
  → **Mitigation:** Run `pnpm run check` after lint fixes to catch runtime issues early; use `--no-cache` to avoid stale test results.

- **[Risk]** JSON sanitization may alter expected output semantics (e.g., visual differences in CLI output).  
  → **Mitigation:** Compare before/after JSON with `jq` and validate against schema tests; document any semantic changes in changelog.

- **[Risk]** Combined fix may introduce merge conflicts if branches diverge.  
  → **Mitigation:** Rebase frequently on `main`; use a dedicated branch (`fix/repo-check-lint-json`) for isolation.

## Migration Plan

1. **Branch creation:** Create `fix/repo-check-lint-json` from `main`.
2. **Lint fixes:** Add missing exports in `commands.js`, `context.js`, and `types`; run `pnpm run lint` until clean.
3. **JSON fix:** Locate and sanitize minus-digit sequences in `docs` output; validate with `pnpm run tooling -- -- repo check docs --json | jq .`.
4. **Validation:** Run `pnpm run check` to confirm all 8 checks pass (including `lint`, `format`, `check`).
5. **Testing:** Add regression test for `repo check docs --json` (e.g., `test/json-output.test.ts`).
6. **PR and merge:** Open PR with full diff; merge after CI passes.
7. **Rollback strategy:** If issues persist, revert the branch; no state changes are made (CLI is stateless).
