## Architecture Overview

The docs-site uses Next.js 16 + Nextra with static generation. Content is generated by the `@cfxdevkit/docs-pipeline` package at Docker build time. The pipeline discovers packages, reads their READMEs/APIs, and writes `.mdx` files to `content/packages/`.

```
.github/workflows/build-docs.yml
    └── pnpm sync all (generates content)
        └── docker build (runs next build)
            └── ghcr.io/cfxdevkit/framework/docs-site:latest
                └── deployed to VPS via deploy-docs.yml
```

## Implementation Strategy

### Phase 1: Pipeline Extensions (docs-pipeline)

Add new sync commands to `@cfxdevkit/docs-pipeline`:

```typescript
// New commands in scripts.ts
export type DocsCommandName =
  | /* existing ... */
  | 'sync:releases'      // NEW
  | 'sync:guides'        // NEW
  | 'sync:api-reference' // NEW
  | 'validate:content';  // existing + enhanced
```

#### `sync:releases`
- Read `.changeset/*.md` files (one per changeset entry)
- Group by version (changesets use `---\nversion: x.y.z\n...` frontmatter)
- Render as changelog with "What's Changed" sections
- Link to npmjs.com provenance URLs
- Output: `content/releases.mdx`

```typescript
// Pseudocode
const changesetFiles = await glob('.changeset/*.md');
const entries = changesetFiles.map(f => parseChangesetFile(f));
const grouped = groupByVersion(entries);
const mdx = renderChangelog(grouped);
writeFileSync('content/releases.mdx', mdx);
```

#### `sync:guides`
- Read `docs/guides/*.md` files
- Convert to MDX (add frontmatter, import Nextra components)
- Generate `content/guides/index.mdx` as index page
- Output: `content/guides/*.mdx`

#### `sync:api-reference`
- Aggregate all 27 package exports into a single table
- Group by tier (framework / platform / domain)
- Include sub-path import paths
- Output: `content/api.mdx`

#### `sync:wiki` (enhanced)
- Keep existing sync logic
- Add post-processing step: `condenseWikiContent()`
- Rules:
  1. First heading → keep as-is
  2. First 200 words → summarize to one paragraph
  3. Remove "Integration with Codebase" sections entirely
  4. Remove "The module contains no executable code" disclaimers
  5. Keep "Configuration Files" sections (they're useful)

### Phase 4: Docker Build Hardening

Update `repos/cfx-tools/packages/docs-site/Dockerfile`:

```dockerfile
# After sync, add verification
RUN node scripts/verify-content.mjs
```

Where `scripts/verify-content.mjs` checks:
- `content/releases.mdx` exists (or allow missing during local dev)
- `content/guides/index.mdx` exists
- `content/api.mdx` exists
- `content/quickstart.mdx` exists
- Total package page count >= 27

### Phase 5: CI Coverage Pre-step

Update `.github/workflows/build-docs.yml`:

```yaml
- name: Generate coverage data
  run: pnpm --filter "@cfxdevkit/*" test:coverage -- --concurrency 4
  continue-on-error: true  # Some packages may not have tests yet
```

This runs before `sync all` so that coverage data is available when `sync:coverage` generates the page.

## Model for Wiki Generation (Deferred)

The wiki quality improvement (Phase 1, item 4) handles basic cleanup. For full wiki regeneration with LLM models:

**Deferred to a follow-up change** due to model dependency:
- Target model: `Qwen3.6-35B-A3B-MTP-GGUF-Q8_0` (lemonade/local)
- Requires running `gitnexus wiki --force` with local model
- Would be a separate `cdk docs wiki generate` command
- Depends on model being available on the build machine

This is intentionally left out of scope for the current change because:
1. The model download/install adds significant build time
2. Model quality varies by hardware
3. The post-processing cleanup (Phase 1) provides immediate value
4. Wiki generation can be a manual/optional step
