# Wallet Abstraction & Hardware

# @cfxdevkit/wallet

The `@cfxdevkit/wallet` module provides a unified interface for managing cryptographic keys, interacting with hardware wallets, enforcing transaction policies, and generating ephemeral session keys. It abstracts the underlying key storage (local keystore files or hardware devices) into the standard `Signer` interface defined in `@cfxdevkit/cdk`.

## Architecture

The module is structured around three core concerns: **Key Management**, **Hardware Abstraction**, and **Security Policies**.

## Key Components

### 1. Hardware Wallet Adapters

Hardware wallets are accessed via the `HardwareWalletAdapter` contract. Adapters return a `Signer` compatible with the rest of the SDK. The module supports **Ledger**, **OneKey**, and **Satochip** (on-hold).

#### Adapter Contract
```typescript
interface HardwareWalletAdapter {
  readonly kind: HardwareWalletKind; // 'ledger' | 'onekey' | 'satochip'
  getSigner(path?: string): Promise<Signer>;
}
```

#### OneKey Adapter
The OneKey adapter (`hardware/onekey`) supports both EVM (eSpace) and Conflux Core Space.

*   **SDK Injection:** The `@onekeyfe/hd-common-connect-sdk` is **not** imported directly. Consumers must initialize the SDK and pass the instance to `signerFromOneKey` to avoid bundling conflicts.
*   **Signer Creation:**
    ```typescript
    import HardwareSDK from '@onekeyfe/hd-common-connect-sdk';
    import { signerFromOneKey } from '@cfxdevkit/wallet/hardware/onekey';

    await HardwareSDK.init({ env: 'webusb', debug: false });
    const signer = await signerFromOneKey({
      sdk: HardwareSDK,
      connectId: '...',
      deviceId: '...',
      chainId: 1030,
    });
    ```
*   **Typed Data:** Supports CIP-23 (Conflux equivalent of EIP-712) via `normaliseTypedData` and `computeCip23Hashes`.
*   **Error Handling:** Device errors are mapped to `HardwareWalletError` with codes like `wallet/hardware/onekey/device-error`.

#### Ledger Adapter
The Ledger adapter (`hardware/ledger`) wraps the `@cfxdevkit/services/keystore-ledger` signer.

```typescript
import { createLedgerHardwareAdapter } from '@cfxdevkit/wallet/hardware/ledger';

const adapter = createLedgerHardwareAdapter({
  eth: ledgerEthApp,
  coreTransport: transport,
  path: "m/44'/60'/0'/0/0",
});
const signer = await adapter.getSigner();
```

#### Satochip Adapter (On-Hold)
The Satochip adapter (`hardware/satochip`) is currently **on-hold**. It requires a Python bridge sidecar (`http://127.0.0.1:8397`) for PC/SC communication and is not suitable for browser-based showcases.

### 2. Local Keystore Initialization

Local key management is handled via `initLocalWallet` and `openLocalWallet`. Keys are stored in an encrypted JSON file (`keystore.json`) using Argon2id for key derivation and AES-256-GCM for encryption.

*   **First Boot:** Use `initLocalWallet`. Generates a BIP-39 mnemonic, encrypts it, and returns the mnemonic for backup.
    ```typescript
    const { signer, mnemonic, path } = await initLocalWallet({
      passphrase: 'strong-passphrase',
      label: 'deployer',
    });
    ```
*   **Subsequent Boots:** Use `openLocalWallet`. Requires the existing `SecretRef` (default: `{ service: 'cfxdevkit', account: 'default' }`).
    ```typescript
    const { signer } = await openLocalWallet({
      passphrase: 'strong-passphrase',
    });
    ```
*   **Passphrase Rotation:** `rotateLocalPassphrase` re-encrypts the keystore under a new passphrase.

### 3. Security Policies

The `policies` module enforces constraints on `Signer` operations via `withCapability`. This wraps any signer (hardware or keystore) to validate transactions before signing occurs.

#### Capability Constraints
A `Capability` object defines allowed operations:
*   `chains`: Allowed chain IDs.
*   `contracts`: Allowed recipient addresses.
*   `selectors`: Allowed function selectors (first 4 bytes of `tx.data`).
*   `maxValuePerTx`: Maximum value allowed per transaction.
*   `notAfter`: Expiry timestamp.

#### Usage
```typescript
import { withCapability } from '@cfxdevkit/wallet/policies';

const scopedSigner = withCapability(rootSigner, {
  chains: [1030],
  contracts: ['0x...'],
  selectors: ['0xa9059cbb'], // ERC-20 transfer
  maxValuePerTx: 1000000000000000000n,
  notAfter: Date.now() + 60 * 60_000,
});

await scopedSigner.signTransaction(tx); // Throws SessionKeyError if violated
```

#### Error Codes
Violations throw `SessionKeyError` with specific codes:
*   `wallet/policies/expired`
*   `wallet/policies/chain-denied`
*   `wallet/policies/contract-denied`
*   `wallet/policies/selector-denied`
*   `wallet/policies/value-exceeded`

### 4. Session Keys

Session keys provide ephemeral signing authority scoped by a capability and authorized by a parent key. They are useful for delegation (e.g., account abstraction, paymasters).

*   **Creation:** `createSessionKey` generates a new keypair, signs an attestation with the parent, and wraps the new signer with policies.
*   **Attestation:** The parent signs a canonical JSON message (`canonicalAttestationMessage`) committing to the session address and capability.
*   **Revocation:** `session.revoke()` immediately disables the signer.

```typescript
const session = await createSessionKey({
  parent: rootSigner,
  capability: { chains: [1030], notAfter: Date.now() + 3600000 },
});

// Use session.signer for delegated operations
await session.signer.signTransaction(tx);

// Revoke when done
session.revoke();
```

### 5. Transaction Batching

The `batcher` module allows queuing and executing multiple tasks with controlled concurrency.

```typescript
import { TransactionBatcher } from '@cfxdevkit/wallet/batcher';

const batcher = new TransactionBatcher();
batcher.add(() => signer.signTransaction(tx1));
batcher.add(() => signer.signTransaction(tx2));

const results = await batcher.execute({ stopOnFailure: true });
```

## Error Handling

The module defines two primary error classes extending `CfxError`:

1.  **`SessionKeyError`**: Used for policy violations, session expiry, and signer state issues.
2.  **`HardwareWalletError`**: Used for device communication failures. Codes are vendor-prefixed (e.g., `wallet/hardware/ledger/transport-unavailable`).

### Common Execution Flow (Hardware Signing)
When a hardware signer fails, errors propagate through the adapter layer:

1.  **Adapter Call:** `signerFromOneKey` calls `sdk.evmSignTransaction`.
2.  **Normalization:** Helper functions (`normaliseSignature`, `toCanonicalHex`) process the response.
3.  **Validation:** If signature length or format is invalid, `HardwareWalletError` is thrown (e.g., `wallet/hardware/onekey/bad-response`).
4.  **Abortion:** If an `AbortSignal` is triggered, `throwIfAborted` throws `wallet/hardware/aborted`.

## Contributing

### Adding a New Hardware Adapter
To add support for a new vendor (e.g., Trezor):

1.  Create a new directory under `src/hardware/<vendor>`.
2.  Implement `HardwareWalletAdapter` in `index.ts`.
3.  Ensure the adapter returns a `Signer` conforming to `@cfxdevkit/cdk`.
4.  Map device errors to `HardwareWalletError` with the code `wallet/hardware/<vendor>/<reason>`.
5.  Export the adapter factory from `src/hardware/index.ts`.

### Signature Normalization
Hardware devices often return `(r, s, v)` tuples. Use the helpers in `src/hardware/types.ts` to convert these to the canonical 65-byte hex format required by the SDK:
*   `rawSignatureToHex`: Converts `(r, s, v)` to `0x...` string.
*   `finaliseEip1559Tx`: Combines unsigned EIP-1559 tx with signature components.
*   `toCanonicalHex`: Ensures hex strings are 0x-prefixed and lowercase.
