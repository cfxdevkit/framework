# Repository Layout — cfx-tools

# cfx-tools

**Tier 1 — Developer Experience.** This module contains the tooling, CLI, and IDE integrations for the Conflux development ecosystem. It is designed for a fast release cadence (weekly) and is **never** a runtime dependency of a deployed application.

## Architecture & Boundaries

`cfx-tools` sits outside the core runtime stack. It may depend on `framework/`, `domains/`, or `projects/` for build-time logic, but core packages must not depend on `cfx-tools`.

## CLI (`@cfxdevkit/cli`)

A developer-facing CLI for network probing and account management. It is bundled with `@cfxdevkit/cdk` but exposed as a standalone binary.

### Commands

*   **`cfx status`**: Pings chains for head block, epoch, and latency.
*   **`cfx derive`**: Derives dual-space accounts (EVM `0x…` + Core `cfx[…]:…`) from a mnemonic.
*   **`cfx generate`**: Generates a fresh BIP-39 mnemonic.

### API Reference

The package exports command helpers for programmatic invocation.

```ts
// Entry point
export { run }

// Status
export { runStatus, StatusReport }

// Derivation
export { runDerive, DeriveReport }

// Generation
export { runGenerate, GenerateReport }

// Utilities
export { parseArgs, ParsedArgs }
```

### Usage

```sh
# Check network status
cfx status --chain core-testnet --json

# Derive accounts (default core-network-id: 1029)
cfx derive --mnemonic "..." --count 5 --show-private-keys

# Generate new mnemonic
cfx generate
```

## Scaffold CLI (`@cfxdevkit/create`)

Project generator used via `npx @cfxdevkit/create <template>`. It discovers templates from `../../templates/`, renders files, and runs post-install hooks.

### API Reference

```ts
// Main entry
export { scaffoldProject }
export { parseArgs }
export { validateName }

// Template Management
export { getTemplate, listTemplates }
export { getTemplateFiles }
export { renderFile }

// Types
export { TemplateDefinition, TemplateFile, TemplateTarget }
```

### Template Contract

Templates reside in `templates/` and must be self-contained (no monorepo coupling).

*   **`template.json`**: Metadata (`name`, `description`, `vars`, `hooks`).
*   **`files/`**: Source files with `{{ variable }}` placeholders.
*   **`hooks/`**: Post-scaffold scripts (e.g., `setup-wallet.sh`).

### Execution Flow

1.  **Discovery**: `listTemplates()` scans `templates/` and remote registries.
2.  **Selection**: User selects a template (e.g., `minimal-dapp`).
3.  **Rendering**: `renderFile()` substitutes variables.
4.  **Post-Install**: Runs `pnpm install`, `git init`, and `env-setup`.

## MCP Server (`@cfxdevkit/mcp-server`)

Model Context Protocol server exposing devkit tools to AI agents. It enforces a strict tool allowlist and requires user confirmation for write operations.

### Security Model

*   **No Raw Keys**: Never accepts raw private keys; uses session keys or managed signers.
*   **Confirmation**: All `write` and `admin` tools require confirmation.
*   **Ledger**: Tracks operation history via `OperationLedger`.

### API Reference

```ts
// Server Creation
export { createMcpServer }

// Tool Registry
export { listMcpTools, getMcpTool, defineTool }
export { MCP_TOOL_DEFINITIONS }

// Types
export type McpToolGroup = 'node' | 'accounts' | 'blockchain-read' | 'blockchain-write' | 'compiler' | 'keystore' | 'wallet-utils' | 'scaffold';
export type McpToolMutability = 'read' | 'write' | 'admin';

// Ledger
export declare class OperationLedger {
  record(operation: OperationRecord): void;
  get(id: string): OperationRecord | undefined;
  list(): OperationRecord[];
  clear(): void;
}
```

### Tool Groups

*   **`node`**: Start/stop dev nodes.
*   **`accounts`**: Derivation and management.
*   **`blockchain-read`**: Query state.
*   **`blockchain-write`**: Deploy/Call contracts (requires confirmation).
*   **`keystore`**: Unlock/lock operations.

## VS Code Extension (`cfxdevkit.vscode`)

Integration for managing Conflux infrastructure workflows within the IDE. It relies on `@cfxdevkit/client` for control-plane operations and `@cfxdevkit/devnode-server` for local runtime state.

### Key Commands

*   **Network**: `cfxdevkit.selectNetwork` (local, testnet, mainnet).
*   **Node**: `cfxdevkit.nodeStart`, `cfxdevkit.nodeStop`, `cfxdevkit.nodeWipe`.
*   **Wallet**: `cfxdevkit.addWallet`, `cfxdevkit.unlockKeystore`, `cfxdevkit.selectWallet`.
*   **Contracts**: `cfxdevkit.deployContract`, `cfxdevkit.abiCallWrite`.

### Views

*   `cfxdevkit.networkView`: Active network status.
*   `cfxdevkit.nodeView`: Local node lifecycle.
*   `cfxdevkit.accountsView`: Derived accounts per network.
*   `cfxdevkit.contractsView`: Tracked contracts by network/space.

### Keystore Backends

1.  **`file`**: Encrypted workspace keystore (`.cfxdevkit/keystore.json`). Supports mnemonic-root creation.
2.  **`onekey`**: OneKey hardware signer (eSpace only).
3.  **`satoshi`**: Satochip bridge signer (eSpace only).

### State Persistence

*   **Workspace State**: Network selection, keystore backend, active wallet root.
*   **File System**: `.cfxdevkit/deployments.json`, `.cfxdevkit/keystore.json`.

## DevTools (`devtools/`)

Internal developer-facing tools not published to npm. These power local development and CI.

### Structure

```
devtools/
├── contracts/          # Hardhat workspace for framework-owned contracts
├── devkit-server/      # Express CLI server + lifecycle UI backend
├── devkit-ui/          # Next.js UI embedded in the CLI
└── cfx-keystore/       # Standalone keystore TUI
```

### Components

*   **`contracts/`**: Contains `hardhat.config.ts`, deploy scripts (`scripts/deploy.ts`), and artifact extraction (`scripts/extract-artifacts.ts`).
*   **`devkit-server/`**: Provides routes for `nodes.ts` (start/stop), `compile.ts`, `deploy.ts`, and `keystore.ts`.
*   **`cfx-keystore/`**: TUI for managing file keystores (`init`, `add`, `unlock`, `export`).

## DevContainer (`devcontainer/`)

Reproducible Docker-based development environment. Ensures `git clone && open in container` works without host tooling.

### Features

*   **Base Image**: Node 20, pnpm, moon, foundry, solc, gh, age, sops.
*   **Features**:
    *   `conflux-toolchain`: Installs `xcfx-node`, hardhat helpers.
    *   `cfx-keystore`: Installs file-keystore CLI + age + sops.
    *   `platformio`: ESP32 firmware toolchain (opt-in).
*   **Lifecycle**:
    *   `post-create.sh`: Runs `pnpm install`, `moon setup`.
    *   `post-start.sh`: Runs env checks.

### Security

*   No key material in the image.
*   Keystore is mounted/created at first run.
*   Host keyring forwarded at runtime.

## Security & Reset Protocols

### Keystore Handling

*   **MCP**: Never accepts raw private keys. Writes go through managed signer abstractions.
*   **VS Code**: Wallet roots are mnemonic roots; accounts are derived child indexes.
*   **Backend**: Keystore terminology is shared across MCP, VS Code, and `showcase-local`.

### Reset & Recovery

Destructive reset is **operator-only**.

1.  Stop the local node or backend.
2.  Remove the configured keystore file (e.g., `.cfxdevkit/keystore.json`).
3.  Remove the sibling `.runtime` directory.
4.  Restart to confirm blank, locked, or active-wallet phase via backend status.

**Note**: MCP tools and VS Code commands must not implement passwordless reset mutations that bypass backend safeguards.
