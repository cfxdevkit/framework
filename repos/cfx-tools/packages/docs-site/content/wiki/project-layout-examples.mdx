# Project Layout & Examples

# projects/ — Tier 3: Consumer Applications

The `projects/` directory contains Tier 3 consumer applications. These are end-user facing applications that consume shared infrastructure from `repos/cfx-*` but maintain their own deploy lifecycles, smart contracts (if applicable), and audit records.

## Module Overview

Each project in `projects/` operates as an independent unit with the following characteristics:
- **Deploy Lifecycle**: Managed via `infrastructure/<project>/`.
- **Smart Contracts**: Owned locally under `contracts/` (if any).
- **Audit Compliance**: Must maintain an `AUDITS.md` file tracking contract review and deployment status.
- **Code Ownership**: Project-specific libraries may live in `packages/`, but reusable code shared beyond a single project should reside in `repos/cfx-*`.
- **Dependencies**: Consumes shared packages from `repos/cfx-*` according to architectural tier rules.

## Standard Project Layout

All projects follow this directory structure:

```
projects/<name>/
├── README.md
├── apps/                  # Deployable units (frontend, backend, worker, firmware)
├── packages/              # Project-internal libraries (optional)
├── contracts/             # Solidity sources + deploy scripts (optional)
├── AUDITS.md              # Audit status and contract records
└── e2e/                   # Cross-app end-to-end tests (optional)
```

## Deep Dive: CAS (Conflux Automation Site)

`projects/cas` is the primary reference implementation for a DeFi automation system. It demonstrates the integration of backend services, frontend dashboards, and on-chain contract interactions.

### Architecture

The CAS system consists of a backend API, a frontend dashboard, and an embedded keeper worker.

### Setup & Configuration

#### Guided Setup (Recommended)
The `cas-setup` wizard handles environment generation and keeper registration.

```bash
# 1. Install dependencies (repo root)
pnpm install

# 2. Build the setup wizard
pnpm --filter @cfxdevkit/cas-setup build

# 3. Run the wizard from the projects/cas/ directory
cd projects/cas
pnpm setup
```

The wizard performs the following steps:
1.  Checks Node.js version (≥ 24).
2.  Verifies RPC connectivity for selected network (testnet/mainnet/local).
3.  Deploys or retrieves contract addresses.
4.  Configures keeper mode (optional).
5.  Writes `apps/backend/.env` and `apps/frontend/.env.local`.

#### Manual Setup (No-Keeper Mode)
For read/create jobs without automatic execution:

```bash
# 1. Install dependencies
pnpm install

# 2. Build shared package
pnpm --filter @cfxdevkit/cas-shared build

# 3. Configure backend
cp projects/cas/apps/backend/.env.example projects/cas/apps/backend/.env
# Edit .env (defaults target testnet, keeper disabled)

# 4. Configure frontend
cp projects/cas/apps/frontend/.env.local.example projects/cas/apps/frontend/.env.local

# 5. Start services
pnpm --filter @cfxdevkit/cas-backend dev   # http://127.0.0.1:3011
pnpm --filter @cfxdevkit/cas-frontend dev   # http://127.0.0.1:3010
```

#### Keeper Mode (Automatic Execution)
The keeper worker is embedded in the backend. It requires a signer account registered on the `AutomationManager` contract.

**1. Register Signer (One-time)**
```bash
# Testnet
cast send 0x33e5E5B262e5d8eBC443E1c6c9F14215b020554d \
  "setKeeper(address,bool)" <signerAddress> true \
  --rpc-url https://evmtestnet.confluxrpc.com \
  --private-key <ownerPrivateKey>

# Mainnet
cast send 0x9D5B131e5bA37A238cd1C485E2D9d7c2A68E1d0F \
  "setKeeper(address,bool)" <signerAddress> true \
  --rpc-url https://evm.confluxrpc.com \
  --private-key <ownerPrivateKey>
```

**2. Enable in Backend**
Update `apps/backend/.env`:
```bash
KEEPER_ENABLED=true
SIGNER_PRIVATE_KEY=0xYOUR_FUNDED_TESTNET_SIGNER_KEY
KEEPER_INTERVAL_MS=15000
```

### Backend API Surface

All job routes require a `Bearer <token>` header from `POST /auth/verify`. Admin routes require the caller's address to be in `ADMIN_ADDRESSES`.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/health` | — | Liveness probe |
| POST | `/auth/verify` | — | Verify SIWE message + signature |
| GET | `/jobs` | Bearer | List jobs for authenticated user |
| POST | `/jobs` | Bearer | Create a new automation job |
| POST | `/jobs/:id/cancel` | Bearer | Cancel a job |
| GET | `/sse/jobs` | Bearer | Server-Sent Events stream |
| GET | `/admin/status` | Bearer + Admin | Keeper status |
| POST | `/admin/pause` | Bearer + Admin | Pause keeper execution |
| GET | `/pools` | — | List tradeable token pairs |

## Deep Dive: Examples

`projects/examples` contains Tier 3 example apps that exercise the public API of `@cfxdevkit/*` packages. These apps serve as runnable demos and integration coverage.

### Showcase Applications

| App | Purpose |
|-----|---------|
| `apps/showcase-local/` | Local-runtime showcase. Runs `@cfxdevkit/devnode-server` in-process. Manages wallet-scoped keystore, network profiles, and tracked contracts. |
| `apps/showcase-public/` | Browser-side public SDK showcase. Covers Core/eSpace RPC, browser wallets, hardware wallets, SIWE, and DeFi widgets without local backend dependency. |

### Shared UI Package

`packages/showcase-ui/` (`@cfxdevkit/example-showcase-ui`) provides shared theme, shell primitives, and wallet UI components.

**Key Components:**
- `ConnectWallet`: Wallet connection UI.
- `LogBox`: Logging widget for debug output.
- `WalletPickerModal`: Wallet selection interface.

**Key Helpers:**
- `useCoreWallet`: Hook for Core chain wallet status.
- `useLogList`: Hook for managing log entries.
- `copy`: Clipboard utility.

**Usage:**
```bash
pnpm --filter @cfxdevkit/example-showcase-ui build
```

### Adding New Examples

1.  Create directory: `mkdir -p projects/examples/apps/<name>`
2.  Add `package.json` named `@cfxdevkit/example-<name>`.
3.  Register path in `.moon/workspace.yml`.
4.  Add `moon.yml` (`type: 'application'`, `language: 'typescript'`).
5.  Run `pnpm install` from workspace root.

**Constraint:** All apps in `projects/examples` are Tier 3. They may import from `framework/`, `platform/`, and `domains/`, but no higher-tier code may import from here.

## Audit & Compliance

Every project must maintain an `AUDITS.md` file.

### Requirements
1.  **Current Scope**: State if `contracts/` exists and if audit records are required.
2.  **Deployment Records**: Before contracts deploy, record:
    - Contract package, deployment script, and artifact details.
    - Internal review date, reviewer, commit SHA, findings, and remediation status.
    - External audit report links (required before mainnet deployment).
    - Deployment addresses and verified source references.

### Example Entry
```markdown
## Contract: AutomationManager
- **Address**: 0x9D5B131e5bA37A238cd1C485E2D9d7c2A68E1d0F
- **Review Date**: 2023-10-01
- **Reviewer**: Security Team
- **Status**: Remediated
- **Audit Report**: [Link to PDF]
```

## Development Workflow

### Running Tests
- **CAS Setup Wizard**:
  ```bash
  pnpm --filter @cfxdevkit/cas-setup test
  pnpm --filter @cfxdevkit/cas-setup typecheck
  ```
- **Backend**:
  ```bash
  pnpm --filter @cfxdevkit/cas-backend test
  ```

### Dependency Management
- Use `pnpm` for workspace management.
- Shared code belongs in `repos/cfx-*`.
- Project-specific code belongs in `projects/<name>/packages/`.
- Do not import from `projects/` into `framework/`, `platform/`, or `domains/`.
