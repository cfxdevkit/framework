# LLM Agent Workflows

## Overview

The LLM Agent Workflows module orchestrates automated repository operations using language models. It provides three primary workflow families:

- **Agent Workflows** (`workers/agents/`) - Repository analysis, validation planning, and smoke testing
- **Commit Workflows** (`workers/commit/`) - Full commit lifecycle with quality gates, LLM message generation, and changeset management
- **Documentation Workflows** (`workers/docs/`) - API docs, README, and package page enrichment

All workflows share a common runtime layer (`workers/agents/runtime/`) for corpus collection, report generation, and path utilities.

## Agent Workflows

### `runAll()` - Full Agent Suite

Runs the complete agent suite and generates consolidated reports.

```typescript
import { runAll } from './agents/all.ts';

const report = await runAll();
// report: { generatedAt, mode, fineTuning, executionContext, results }
```

**Flow:**
1. Resolves execution context without LLM (`useLlm: false`)
2. Runs `runReviewAgent()` with silent mode
3. Writes JSON report to `reports/agent-run.json`
4. Writes Markdown report to `reports/agent-run.md`
5. Prints summary to console

### `runReviewAgent()` - Repository Review

Analyzes changed files for common issues and generates validation recommendations.

```typescript
import { runReviewAgent } from './agents/review.ts';

const result = await runReviewAgent({ silent: true });
// result: { agent: 'review', status, changedFiles, findings, executionContext }
```

**Key Checks:**
- **Generated paths** - Warns if artifacts are committed
- **Security-sensitive files** - Flags changes to keystore, wallet, release, workflow paths
- **Code hotspots** - Checks file size budgets (soft: 250 lines, hard: 300 lines)
- **Validation commands** - Suggests appropriate `pnpm run` commands based on changed files

**Calling Pattern:**
```typescript
runReviewAgent()
  → isGeneratedPath() / isSecuritySensitive() (runtime/paths.ts)
  → codeHotspotReport() (runtime/models.ts)
  → suggestValidationCommands() (runtime/checks.ts)
  → writeJsonReport() / writeMarkdownReport() (runtime/reports.ts)
```

### `runAgentCheck()` - Validation Artifact Analysis

Processes validation artifacts and creates actionable plans for fixing issues.

```typescript
import { runAgentCheck, parseAgentCheckFlags } from './agents/check.ts';

const report = await runAgentCheck(['--dry-run', '--create-branch']);
```

**Flags:**
| Flag | Description |
|------|-------------|
| `--dry-run` | Skip artifact materialization and follow-up actions |
| `--create-branch` | Create or switch to the target branch |
| `--draft-pr` | Create draft PR after branch switch |
| `--quick` | Use reduced token budgets |

**Flow:**
1. Parse flags and resolve execution context with LLM (`useLlm: true`)
2. Run repository check command and read validation artifact
3. Collect actionable validation steps
4. Build agent check plan (if actionable steps exist)
5. Materialize OpenSpec changes (unless dry-run)
6. Create branch and draft PR (if requested)
7. Write JSON and Markdown reports

**Calling Pattern:**
```typescript
runAgentCheck()
  → parseAgentCheckFlags()
  → resolveExecutionContext(useLlm: true)
  → runRepoCheckCommand()
  → readValidationArtifact()
  → collectActionableValidationSteps()
  → buildAgentCheckPlan()
  → materializeOpenSpecChanges()
  → maybeCreateBranchAndDraftPr()
  → writeJsonReport() / writeMarkdownReport()
```

### `runAgentSmoke()` - LLM Provider Smoke Tests

Tests configured LLM providers and models for viability.

```typescript
import { runAgentSmoke } from './agents/smoke.ts';

await runAgentSmoke(['--quick', '--no-thinking']);
```

**Probe Sequence:**
1. **Cold probe** - Simple echo, measures initial model load time
2. **Hot probe** - Same prompt, model should already be warm
3. **Task+Thinking** - Tier-appropriate JSON task with thinking enabled
4. **Task-No-Think** - Same task with thinking disabled

**Tier Probes:**
| Tier | Label | Representative Actions |
|------|-------|----------------------|
| 1 | Lightweight / structured output | `validation` |
| 2 | Documentation generation | `docs-api`, `readme-upkeep`, `structure-upkeep` |
| 3 | Reasoning / code review | `review`, `commit`, `changeset`, `ci-cd` |

**Output:** Console table showing model viability, timing, and JSON shape validation.

## Commit Workflows

### `runCommitWorkflow()` - Full Commit Pipeline

Orchestrates the complete commit lifecycle with quality gates, LLM message generation, and changeset management.

```typescript
import { runCommitWorkflow } from './commit/commit.ts';

const result = await runCommitWorkflow(args, {
  approvalMode: 'prompt', // 'defer' | 'prompt' | 'auto-approve'
  modelPolicies: {
    messageGenerationModel: 'gemma-4-26b',
    failureAnalysisModel: 'gemma-4-26b'
  }
});
```

**Workflow Steps:**
1. **Incremental validation sequence** - Run quality gates (lint, typecheck, tests, hotspots, etc.)
2. **Preflight checks** - Ensure gitnexus registration, collect git status/diff signals
3. **Detect changed scopes** - Group files by repository scope
4. **Check release intent** - Generate changeset plan if publishable packages changed
5. **Generate commit message** - LLM generates conventional commit JSON
6. **Approval** - Prompt for confirmation (unless `--yes` or `--dry-run`)
7. **Write changeset** - Create changeset files if needed
8. **Post-checks** - Run quality gates after changeset generation
9. **Commit** - Stage files and create git commit

**Status Values:**
- `blocked` - Failed quality or policy gates
- `approval-required` - Waiting for user confirmation
- `aborted` - User declined confirmation
- `dry-run` - Preview only, no changes made
- `committed` - Successfully created commit

**Calling Pattern:**
```typescript
runCommitWorkflow()
  → parseCommitFlags()
  → resolveExecutionContext(useLlm: true)
  → detectChangedScopes()
  → runQualityGates() / runRepositoryPolicyGates()
  → analyzeGateFailures() (if gates fail)
  → generateChangesetPlan()
  → generateCommitMessage()
  → writeCommitReport()
  → writeChangesetFile()
  → runQualityGates() (post-checks)
  → executeCommit()
```

### `runPrecommitWorkflow()` - Validation-Only Commit

Runs quality gates without generating commit messages or making changes.

```typescript
import { runPrecommitWorkflow } from './commit/precommit.ts';

const result = await runPrecommitWorkflow(args);
// result.status: 'passed' | 'forced' | 'blocked'
```

**Key Differences from `runCommitWorkflow()`:**
- Uses `useLlm: false` (no LLM for failure analysis)
- Skips changeset generation
- Skips commit message generation
- Returns early after quality gates pass/fail

### Quality Gates

Two gate types are supported:

**Quality Gates** (`runQualityGates()`):
- `gitnexus analyze`
- `format`
- `lint`
- `typecheck`
- `test`
- `build`
- `hotspots`
- `kebab-groups`
- `check`

**Repository Policy Gates** (`runRepositoryPolicyGates()`):
- `hotspots` - Code file size budget violations
- `kebab-groups` - Kebab file group analysis
- `check` - Repository check command

**Gate Result Structure:**
```typescript
type GateResult = {
  kind: 'repository-policy' | 'quality';
  id: string;
  label: string;
  command: string;
  required: boolean;
  status: 'ok' | 'warning' | 'error' | 'skipped';
  elapsedMs: number;
  summary: string;
  output: string;
  signalLines: string[];
  hints: string[];
};
```

### Changeset Management

**`generateChangesetPlan()`** - Analyzes changed packages and generates changeset suggestions.

```typescript
import { generateChangesetPlan, writeChangesetFile } from './commit/changeset.ts';

const plan = await generateChangesetPlan(scopes, flags);
// plan: { ok, releaseRelevant, summary, packages, changesets, changedChangesets, risks }

const writtenPaths = await writeChangesetFile(plan);
```

**Changeset Flow:**
1. Collect publishable packages (non-private, not ignored)
2. Detect which packages have changed
3. Check for existing changeset files
4. Generate LLM-based changeset suggestion (with diff excerpts)
5. Fallback to deterministic changeset if LLM returns invalid JSON

**LLM Prompt Schema:**
```json
{
  "summary": "one sentence",
  "changesets": [
    { "packageName": "@scope/name", "bump": "patch|minor|major", "summary": "release note" }
  ],
  "risks": ["risk or empty"]
}
```

### Commit Message Generation

**`generateCommitMessage()`** - Uses LLM to generate conventional commit metadata.

```typescript
import { generateCommitMessage, validateCommitJson } from './commit/message.ts';

const { response, commit } = await generateCommitMessage(preflightCtx, changesetPlan, flags);
// commit: { subject, body, filesToStage, risks }
```

**Commit JSON Schema:**
```json
{
  "subject": "feat(scope): description under 72 chars",
  "bodyLines": ["body line 1", "body line 2"],
  "filesToStage": ["optional", "paths"],
  "risks": ["risk or empty"]
}
```

**Fallback Strategy:**
1. First attempt: LLM with full context (60k tokens)
2. Retry: LLM with error context (30k tokens, no thinking)
3. Fallback: Deterministic message based on changeset plan

## Documentation Workflows

### `runDocsApi()` - API Documentation Pipeline

Generates API.md files with LLM enrichment.

```typescript
import { runDocsApi } from './docs/api.ts';

await runDocsApi(['--package', '@scope/name', '--precheck']);
```

**Steps:**
1. Refresh deterministic API.md skeletons via `docs-pipeline`
2. Discover API targets (packages)
3. Optional: LLM precheck on first package
4. LLM enrichment per package

### `runDocsReadme()` - README Upkeep

Enriches README.md files while preserving deterministic skeletons.

```typescript
import { runDocsReadme } from './docs/readme.ts';

await runDocsReadme(['--package', '@scope/name']);
```

**Skeleton Hash Check:**
- `` footer encodes skeleton hash
- If hash matches current package skeleton, skip LLM call
- Preserves hash footer after enrichment

### `runDocsPackagePages()` - Package Page Enrichment

Enriches docs-site package pages (MDX).

```typescript
import { runDocsPackagePages } from './docs/packages.ts';

await runDocsPackagePages(['--package', '@scope/name']);
```

**Steps:**
1. Sync package MDX stubs via `docs-pipeline`
2. Discover public packages
3. LLM enrich each `content/packages/<slug>.mdx`

## Runtime Utilities

### Execution Context

**`resolveExecutionContext()`** - Resolves LLM provider and model configuration.

```typescript
import { resolveExecutionContext } from '../shared/execution-context.ts';

const ctx = await resolveExecutionContext({
  useLlm: true,
  action: 'commit',
  modelOverride: 'gemma-4-26b'
});
```

**Context Structure:**
```typescript
{
  unit: { name, rootDir } | null,
  llm: {
    status: 'not-used' | 'ready' | 'unavailable',
    provider, model, baseUrl, configPath, error?,
    tokenBudget: { quick?, contextFraction?, cap? }
  }
}
```

### Corpus Collection

**`collectCorpusFiles()`** - Gathers text files for LLM context.

```typescript
import { collectCorpusFiles, chunkFile } from './agents/runtime/corpus.ts';

const files = await collectCorpusFiles();
const chunks = chunkFile(path, content);
```

**File Selection Rules:**
- Only text extensions (`.ts`, `.md`, `.json`, etc.)
- Exclude generated directories (`.git`, `node_modules`, `dist`, etc.)
- Exclude secret-sounding paths (`.env`, `secret`, `private-key`, etc.)
- Max file size: 512 KB
- Chunk size: 80 lines for source files, 60 for docs

### Report Generation

**`writeJsonReport()` / `writeMarkdownReport()`** - Write reports to `artifacts/llm/`.

```typescript
import { writeJsonReport, writeMarkdownReport, renderReview } from './agents/runtime/reports.ts';

await writeJsonReport('reports/agent-run.json', report);
await writeMarkdownReport('reports/agent-run.md', renderAgentRun(report));
```

**Report Types:**
| Report | Path | Renderer |
|--------|------|----------|
| Review | `reports/review.json` / `.md` | `renderReview()` |
| Agent Check | `reports/agent-check.json` / `.md` | `renderAgentCheckReport()` |
| Commit | `reports/llm-commit.md` | `writeCommitReport()` |
| Agent Run | `reports/agent-run.json` / `.md` | `renderAgentRun()` |

## Error Handling & Recovery

### LLM Failure Analysis

**`analyzeGateFailures()`** - Uses LLM to diagnose validation gate failures.

```typescript
import { analyzeGateFailures } from './commit/failure-analysis.ts';

const analysis = await analyzeGateFailures({
  command: 'commit',
  executionContext,
  reports: [qualityReport, policyReport],
  modelOverride: 'gemma-4-26b'
});
// analysis: { attempted, usedLlm, status, content, error? }
```

**Output Sections:**
- Summary
- Root causes
- Minimal fixes
- Next commands

### JSON Validation & Recovery

All LLM responses expect strict JSON. Recovery strategy:

1. Parse JSON directly
2. If invalid, strip markdown fences and retry
3. If still invalid, use fallback deterministic output

**Validation Functions:**
- `validateCommitJson()` - Validates commit metadata JSON
- `validateChangelogJson()` - Validates changelog JSON
- `validateTaskKey()` - Validates task-specific JSON shape
- `parseJsonObject()` - Safe JSON parsing with error recovery

## Common Patterns

### Workflow Pattern

```typescript
export async function runWorkflow(args: string[], flags: Flags) {
  // 1. Parse flags
  const flags = parseFlags(args);
  
  // 2. Resolve execution context
  const ctx = await resolveExecutionContext({ useLlm: true, action: '...' });
  
  // 3. Run validation gates
  const qualityReport = await runQualityGates(flags);
  
  // 4. Analyze failures if needed
  if (!qualityReport.passed && !flags.force) {
    const analysis = await analyzeGateFailures({ ... });
    return { status: 'blocked', analysis };
  }
  
  // 5. Generate content with LLM
  const content = await completeStructuredAgent({ ... });
  
  // 6. Write reports
  await writeJsonReport('reports/workflow.json', report);
  
  // 7. Return result
  return result;
}
```

### Report Rendering Pattern

```typescript
export function renderReport(report: Report): string {
  return [
    '# Title',
    '',
    `Generated: ${report.generatedAt}`,
    '',
    ...renderFindings(report.findings),
    '',
    ...report.suggestedValidation.map(c => `- ${c}`)
  ].join('\n');
}
```
