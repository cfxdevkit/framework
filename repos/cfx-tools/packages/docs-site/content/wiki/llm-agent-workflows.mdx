# LLM Agent Workflows

This module implements the autonomous agent layer for the cfx-tools monorepo. Agents run validation pipelines, generate structured reports, and drive LLM-assisted workflows including commit message generation, OpenSpec planning, and model smoke-testing. Everything writes output to `artifacts/llm/`.

---

## Agents

### `runReviewAgent` (`agents/review.ts`)

A lightweight, LLM-free scan of the working tree. It runs without an LLM (`useLlm: false`) and produces a structured report covering:

- **Generated/artifact paths** that appear in `git diff` or untracked files
- **Security-sensitive surfaces** (keystore, wallet, workflow, audit paths)
- **Code hotspot violations** from `check-hotspots.ts` — hard violations (files over the line limit) become `error` severity; soft warnings stay `warning`
- **Suggested validation commands** derived deterministically from which file types changed

Output: `artifacts/llm/reports/review.json` and `reports/review.md`.

`runAll` (`agents/all.ts`) runs `runReviewAgent` in silent mode and wraps the result in a combined `agent-run.json`/`agent-run.md` report.

---

### `runAgentCheck` (`agents/check.ts`)

The repo-check planning agent. Given `--quick`, `--dry-run`, `--create-branch`, and `--draft-pr` flags parsed by `parseAgentCheckFlags`, it executes this pipeline:

<Mermaid chart={`flowchart TD
    A[runRepoCheckCommand] --> B[readValidationArtifact]
    B --> C{actionable steps?}
    C -- yes --> D[buildAgentCheckPlan]
    C -- no --> E[skip plan]
    D --> F[materializeOpenSpecChanges]
    F --> G[maybeCreateBranchAndDraftPr]
    G --> H["writeJsonReport / writeMarkdownReport"]
`} />

**Planning phase** (`check-plan.ts`):

`buildAgentCheckPlan` formats the raw `pnpm cdk repo check` output and the `validation.json` artifact into a structured prompt via `summarizeValidationForPrompt`, which extracts actionable steps (status `warning` or `error`) and their hotspot/kebab-group details. The LLM returns a JSON plan matching this schema:

```ts
{
  summary: string;
  changes: Array<{ name: string; title: string; rationale: string; issues: string[] }>;
  branch: { name: string; title: string; bodyLines: string[] };
  handoff: { cloudPromptLines: string[]; notes: string[] };
}
```

If the response is unparseable, `normalizeAgentCheckPlan` (`check-render.ts`) attempts JSON repair: raw newline escaping, trailing comma stripping, and brace extraction before throwing.

**Artifact materialization phase** (`check-artifacts.ts`):

For each planned change, `materializeOpenSpecChanges`:
1. Calls `ensureOpenSpecChange` — runs `openspec new change <name>` if `.openspec.yaml` doesn't exist
2. Fetches live OpenSpec instructions for all four artifact types (`proposal`, `design`, `specs`, `tasks`) via parallel `openspec instructions` CLI calls
3. Calls `generateOpenSpecArtifactMarkdown` for each artifact — a `completeStructuredAgent` call with retry logic that strips markdown fences from responses
4. Writes the resulting markdown files under `openspec/changes/<name>/`

The `quick` flag halves `maxTokens` budgets throughout (2 200 vs 4 200 for artifacts, 3 600 vs 7 200 for the plan).

**Follow-up phase**:

`maybeCreateBranchAndDraftPr` creates or switches to the planned branch via `git switch -c` and optionally opens a draft PR via `gh pr create --draft`. All outcomes are captured in the `AgentCheckFollowUp` shape and surfaced in the report.

### `runAgentSmoke` (`agents/smoke.ts`)

Probes all configured models to verify they are loaded, warm, and capable of producing valid structured output. For each model in the config's `actions` map, it runs four sequential probes:

| Probe | Thinking | Purpose |
|---|---|---|
| `cold` | off | Measures load time, confirms the model responds |
| `hot` | off | Confirms warmth after load |
| `task` (tier-matched) | on | Validates JSON shape for the model's assigned tier |
| `taskNoThink` | off | Same prompt, no thinking tokens |

**Tier assignment** (`smoke-render.ts: assignTierProbe`): maps action names to one of three `TierProbe` entries. Tier 1 (validation) asks for shell commands; Tier 2 (docs) asks for a JSDoc sentence; Tier 3 (reasoning/review) asks about merge risk. A model with no assigned actions gets `'default'` tier.

**Viability criteria**: `cold.ok && hot.ok && (task.jsonShapeOk || taskNoThink.jsonShapeOk)`. A model that passes cold/hot but fails both task probes needs prompt or thinking-mode tuning — the smoke output says so explicitly.

Token budgets are computed from the model's discovered context window: `min(floor(ctx * fraction), cap)`. The `--quick` flag uses `max(quickBudget, floor(full / 4))`.

Results are printed as a formatted table via `renderSmokeTable`.

## Commit Workflow

`runCommitWorkflow` (`commit/commit.ts`) is the primary commit orchestrator. It runs as eight named steps with a terminal UI (`createWorkflowTerminalUi`) keeping the developer informed.

<Mermaid chart={`flowchart TD
    A[Quality gates] --> B{passed?}
    B -- no, force --> C[Repository policy gates]
    B -- yes --> C
    C --> D[Preflight: gitnexus + git signals]
    D --> E[Detect changed scopes]
    E --> F[Changeset plan]
    F --> G[Generate commit message]
    G --> H{Approval mode}
    H -- dry-run / defer --> I[Return preview]
    H -- prompt / yes --> J[Write changeset]
    J --> K[Post-checks]
    K --> L[Stage + commit]
`} />

**Gate system** (`commit/gates.ts`):

Two gate groups run in sequence:

- `runQualityGates` — runs `QUALITY_GATE_SPECS` (format, lint, typecheck, test, build) via `runRepoCommand`. Individual gates can be skipped with `--skip-tests`, `--skip-build`, or `--skip-checks`.
- `runRepositoryPolicyGates` — runs hotspot check (`--fail-on-hard`), kebab-group check, and full repo check. Required gates cause blocking; non-required gates produce warnings.

Both groups use `GateRunHooks` to stream real-time start/finish events to the terminal UI.

When gates fail, `analyzeGateFailures` (`commit/failure-analysis.ts`) sends the structured `GateResult` output to the LLM (`action: 'validation'`, tier 1 model) for a human-readable diagnosis. If the LLM is unavailable, `buildDeterministicHints` in `gate-output.ts` provides static hints keyed to gate IDs.

**Changeset plan** (`commit/changeset.ts`):

`generateChangesetPlan` detects which publishable packages are touched, skips if changesets already exist or if `--no-changeset` is set, and otherwise calls the LLM to produce per-package bump/summary pairs. `writeChangesetFile` renders the `.changeset/<slug>.md` format and handles filename collision with `nextAvailablePath`.

**Commit message generation** (`commit/message.ts`):

`generateCommitMessage` assembles a context string from the preflight block, changeset guidance, and `CONTRIBUTING.md`, then requests a JSON commit object:

```ts
{ subject: string; bodyLines: string[]; filesToStage?: string[]; risks?: string[] }
```

`validateCommitJson` parses and validates the response. On failure, a retry prompt includes the invalid excerpt. `fallbackCommitMessage` provides a safe default if both attempts fail.

**Approval modes**:

| `approvalMode` | `--yes` | Behaviour |
|---|---|---|
| `defer` | false | Returns `approval-required` without committing — caller (e.g. pi-agent) handles prompting |
| `prompt` | false | Pauses and calls `confirmPrompt` interactively |
| any | true | Proceeds without prompting |

**Pre-commit workflow** (`commit/precommit.ts`) runs a subset of the same gates without message generation and is wired to the git pre-commit hook via `runPrecommit`.

## Runtime Utilities (`agents/runtime/`)

The `runtime/` barrel (`index.ts`) exports utilities consumed by all agents:

| File | Key exports |
|---|---|
| `reports.ts` | `writeJsonReport`, `writeMarkdownReport`, `writeArtifact`, `renderReview`, `renderAgentRun`, `printSummary` |
| `checks.ts` | `gitChangedFiles`, `suggestValidationCommands`, `checkMoonRegistration`, `checkPackageExports` |
| `corpus.ts` | `collectCorpusFiles`, `chunkFile`, `extractDocIndex` |
| `docs.ts` | `findBrokenPathRefs`, `looksLikeLocalPath`, `resolveDocRefCandidates` |
| `models.ts` | `codeHotspotReport`, `extractModelInventory` |
| `paths.ts` | `isGeneratedPath`, `isSecuritySensitive`, `tierForPath`, `languageForPath`, `toRel` |
| `constants.ts` | `root`, `artifactsRoot`, `generatedDirs`, `textExtensions`, `sourceExtensions` |

All artifact writes go through `writeArtifact`, which resolves paths relative to `artifactsRoot` (`<cwd>/artifacts/llm/`) and creates parent directories automatically.

`suggestValidationCommands` returns a deterministic set of `pnpm` commands based on which file categories changed — TypeScript files trigger lint/typecheck/test, `.md`/`.yml` files trigger `check:docs`, changeset files trigger `changeset status`, security-sensitive files trigger `security:check`.

## Entry Points and CLI Wiring

| Function | Called from |
|---|---|
| `runCommit(args)` | `tooling-cli` `runRepoCli`, `runExploratoryCli`; `pi-agent` `executePiCommitWorkflow` |
| `runPrecommit(args)` | `tooling-cli` `runRepoCli`, `runDeterministicCli`; git pre-commit hook |
| `runAgentCheck(args)` | `pi-agent` `executePiAgentCheck` |
| `runAgentSmoke(args)` | `tooling-cli` `runAgentCli` |
| `runAll()` | `tooling-cli` batch agent run |

`runCommit` and `runPrecommit` set `process.exitCode = 1` on `blocked` or `aborted` results so the shell sees a non-zero exit without throwing.

type GateResult = { kind; id; label; command; required; status; elapsedMs; summary; output; signalLines; hints };

// commit/types.ts
type CommitWorkflowResult = { command; status; phase; executionContext; scopes; qualityGates; repositoryPolicies; changesetPlan?; commitPreview?; approval; failureAnalysis; ... };
