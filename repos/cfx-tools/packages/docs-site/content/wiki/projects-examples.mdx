# projects — examples

# Showcase Local (`apps/showcase-local`)

## Purpose & Overview
The `showcase-local` application is a single-page Next.js dashboard designed to interact with a local Conflux runtime environment (devnode-server). It provides a unified interface for managing keystores, deriving accounts, controlling local blockchain nodes, compiling and deploying smart contracts, and issuing/verifying session keys across eSpace and Core.

The module follows a thin-client architecture where Next.js API routes act as proxies to the backend runtime, and the frontend UI is composed of highly composable panels and modal dialogs that consume these routes via dedicated client-side adapters.

## Architecture & Request Flow

The application uses the Next.js App Router. Client-side components dispatch requests to server-side API routes, which forward them to the backend runtime via `requestRuntime`. Responses are parsed and returned to the client.

## API Layer: Proxy Adapters

All backend interactions are mediated through `app/api/` routes. These routes are intentionally thin, handling HTTP parsing, path extraction, and JSON serialization while delegating business logic to `lib/local-runtime.ts`.

### Core Proxy Pattern
Most routes (`accounts`, `keystore`, `network`, `node`, `bootstrap`, `deploy`, `compile`, `session-key`) implement a forwarding strategy:

1. **Path Extraction**: Dynamic catch-all routes (`[...path]/route.ts`) extract the trailing path segments using `request.url.pathname.split('/api/<route>/')[1]`.
2. **Method Routing**: Handlers delegate to an internal `forward(request, method)` function or route directly based on HTTP verb.
3. **Body Forwarding**: For non-GET requests, the raw body is captured via `request.text()` and forwarded with the appropriate `content-type` header.
4. **Runtime Invocation**: `requestRuntime(targetPath, options)` handles the actual HTTP call to the backend.
5. **Response Translation**: `readRuntimeJson<T>(response)` parses the backend response and returns it directly to the client via `NextResponse.json()`.

### Specialized Route Handling
Some routes deviate from the standard proxy pattern to implement validation or aggregation:

- **`devnode/mine/route.ts`**: Validates the `count` parameter (1-100) before invoking `runShowcaseDevnodeAction('/node/mine')`.
- **`devnode/accounts/route.ts`**: Aggregates account data and fetches the faucet balance separately, merging them into a `DevnodeAccountsResponse`.
- **`keystore/accounts/route.ts`**: Handles wallet creation with fallback logic (`generateMnemonic(128)`) and explicit JSON serialization.
- **`keystore/active/route.ts`**: Wraps `readRuntimeActiveWallet()` in a try/catch to normalize errors into a 403 response with a `wallet: null` fallback.
- **`health/route.ts`**: Returns a static `200 OK` payload for health checks.

All API routes enforce cache-busting by attaching `noStoreHeaders()` to their responses.

## Client-Side Adapters

TypeScript functions in `app/*-client.ts` abstract the API routes and `@cfxdevkit` runtime SDK into reusable async functions for the UI layer.

- **`devnode-client.ts`**: Wraps `showcaseRuntimeClient.node.*` for status, profiles, and lifecycle mutations. Implements `runNodeMutation()` to catch errors, attach a fallback payload via `fetchDevnodeStatus()`, and rethrow as `ErrorWithPayload`.
- **`keystore/client.ts``:` Manages the two-step `revealSecret()` flow (`/api/keystore/reveal/request` → `/api/keystore/reveal/consume`) and account fetching.
- **`contracts/contracts-client.ts`**: Constructs query strings for `GET /api/contracts` and enforces `cache: 'no-store'` on fetch. Throws on non-OK responses.

## UI Composition & Panels

The main interface is rendered by `ShowcaseWorkspacePanels`, which arranges a series of self-contained sections. Each panel consumes `ShowcaseWorkspacePanelsProps` for shared context (network, active wallet, callbacks) but manages its own local UI state.

| Panel | Responsibility |
|-------|----------------|
| `EnvironmentSetupPanel` | Network profile selection and chain configuration. |
| `KeystorePanel` | Wraps `@cfxdevkit/react/keystore/KeystoreShell` to manage passphrase setup, unlocking, wallet CRUD, and identity stripping. |
| `AccountsPanel` | Displays derived accounts, activation status, and local faucet funding controls. |
| `DevnodePanel` | Node lifecycle controls (start/restart/stop/wipe), profile selection, block mining, and genesis account listing. Local-only. |
| `ComposePanels` | Hosts the Compiler and Session Key state machines, artifact inspection, and code example toggles. |
| `DeployPanel` | Deployment readiness checks, contract registry browsing, and deploy sheet triggering. |
| `ContractContextPanel` | Displays selected contract metadata, ABI functions, and quick actions. |
| `CustomOperationPanel` | Read-only view for custom RPC calls (e.g., block number) with collapsible code examples. |
| `RevealPanel` | Secure secret/key revelation controls. |

Styling is centralized in `devnode-ui.tsx` using CSS variables (`var(--cfx-space-*)`, `var(--cfx-color-*)`) and predefined layout constants (`stackStyle`, `rowStyle`, `statsGridStyle`).

## Dialog System

Modals are managed by `ShowcaseWorkspaceDialogs`, which renders `ShowcaseWorkspaceDialogContent`. The dialog system:

1. Listens for the `Escape` key to dismiss the modal.
2. Switches rendered content based on `activeDialog` (`WorkspaceDialogId`).
3. Delegates form layout to specialized body components:
   - `CompilerDialogBody`
   - `DeployDialogBody`
   - `SessionKeyDialogBody`
   - `CustomOperationDialogBody`
   - `WorkspaceFamilyControl`

Each body component binds to `drafts` for input state and `compose` for async action dispatch.

## State Management & Drafts

State is divided into two primary hooks:

- **`useShowcaseWorkspaceDrafts`**: Persists and manages transient form inputs (contract name, solc version, source code, session key constraints, faucet address/amount). Calls `readStoredDraft` to hydrate from local storage.
- **`ShowcaseWorkspaceCompose`**: Orchestrates async operations (`runCompile`, `runDeploy`, `runIssueSession`, `runReadBlockNumber`). Exposes busy states, error strings, and success payloads to the UI. Triggers API calls through the client adapters.

## Contributing Guidelines & Patterns

- **Adding API Routes**: Prefer the `[...path]/route.ts` proxy pattern for simple CRUD or RPC endpoints. Extract the path dynamically, preserve `content-type`, and use `requestRuntime` + `readRuntimeJson`. For validation-heavy endpoints, implement explicit checks before calling the runtime.
- **UI Layout**: Use the pre-defined style constants from `devnode-ui.tsx`. Avoid inline styles unless necessary for dynamic values. Panels should be self-contained and receive shared state via `ShowcaseWorkspacePanelsProps`.
- **Error Handling**: Client-side mutations should follow the `runNodeMutation` pattern: attempt the operation, catch the error, attach a fallback status payload, and rethrow. API routes should always return JSON with `noStoreHeaders()`.
- **Type Safety**: Import response/request types from `lib/*-types.ts`. Use `catch(() => {} as T)` for resilient JSON parsing in routes that accept external payloads.
- **Theming**: Rely on `@cfxdevkit/theme/css` and `@cfxdevkit/theme/dark` variables. Do not hardcode colors or spacing.
