# repos — cfx-keys

# `@cfxdevkit/services` — Keystore, Wallet, and Crypto Module

`@cfxdevkit/services` implements the **Tier 0b audit-grade trust boundary** for the Conflux development framework. It provides a pluggable keystore abstraction, cryptographic primitives, authentication helpers, and embedded wallet management.

The core contract is `KeystoreProvider`. Private key material never crosses this boundary; instead, `getSigner` returns a `Signer` object that holds the secret internally.

## Keystore Provider

The `KeystoreProvider` interface defines the contract for all backends. Backends advertise `capabilities` (`write`, `list`, `rotate`). Optional methods (`put`, `remove`, `rotate`) must be omitted on read-only backends rather than throwing.

```typescript
interface KeystoreProvider {
  readonly id: string;
  readonly capabilities: KeystoreCapabilities;
  list(opts?: KeystoreListOptions): Promise<StoredSecret[]>;
  has(ref: SecretRef, opts?: KeystoreCallOptions): Promise<boolean>;
  getSigner(ref: SecretRef, capability?: Capability, opts?: KeystoreCallOptions): Promise<Signer>;
  put?(input: KeystorePutInput, opts?: KeystoreCallOptions): Promise<void>;
  updateMeta?(ref: SecretRef, meta: Record<string, string>, opts?: KeystoreCallOptions): Promise<void>;
  remove?(ref: SecretRef, opts?: KeystoreCallOptions): Promise<void>;
  rotate?(ref: SecretRef, opts?: KeystoreCallOptions): Promise<{ ref: SecretRef }>;
}
```

### Capability Enforcement

Backends can return a `Signer` wrapped by `applyCapability`. The wrapper intercepts `signTransaction` and rejects calls that violate the configured policy. `signMessage` and `signTypedData` are delegated unchanged.

**`Capability` Constraints:**
*   `chains`: Allowed `ChainId[]`.
*   `contracts`: Allowed target `Address[]`.
*   `selectors`: Allowed 4-byte function selectors (`Hex[]`).
*   `maxValuePerTx`: Maximum `bigint` value per transaction.
*   `notAfter`: Expiry timestamp.

### Audit Logging

Providers accept an optional `AuditLogger`. All mutations and `getSigner` calls record `AuditEntry` events. The default is `noopAuditLogger`.

```typescript
interface AuditLogger {
  record(entry: AuditEntry): void;
}
// AuditEntry includes: at, provider, action, ref, ok, meta
```

## Backends

### Memory Keystore

`createMemoryKeystore(options)` initializes an in-memory store. Supports seeding with initial private keys. Useful for testing and development environments where persistence is not required.

*   **Features:** Full `put`/`remove` support.
*   **Security:** Keys are held in memory; not encrypted at rest.

### File Keystore

`createFileKeystore(opts)` wraps an encrypted JSON file. The file uses an `Envelope` format:

```typescript
interface Envelope {
  version: 'cfx-v1';
  kdf: { name: 'argon2id'; salt: string; memKiB: number; iterations: number; parallelism: number };
  secrets: Record<string, EncryptedSecret>;
}
```

**Workflow:**
1.  **Initialization:** `initFileKeystore` creates the envelope with a random salt and writes it. Refuses to overwrite existing files.
2.  **Unlocking:** `createFileKeystore` uses an `unlock` callback to retrieve the passphrase.
3.  **Key Derivation:** `ensureKek` derives the Key Encryption Key (KEK) via `deriveKekFromEnvelope` (Argon2id). It probes decryption against a known secret to validate the passphrase.
4.  **Secrets:** Secrets are encrypted via `encryptAesGcm` with AAD containing `cfx-v1:{service}:{account}`.

**Passphrase Rotation:** `changeFilePassphrase` decrypts all secrets with the old passphrase, re-encrypts with a new KEK, and atomically writes the new envelope.

### Ledger Keystore

`createLedgerKeystore(opts)` exposes configured derivation paths as `opaque` secrets. It returns `Signer` objects that delegate to hardware.

*   **eSpace:** Uses `LedgerEthAppLike` (typically `@ledgerhq/hw-app-eth`).
*   **Core Space:** Uses raw APDU exchange via `coreTransport`.
*   **Account Resolution:** `resolveAccount` maps `LedgerAccountConfig` to `ResolvedLedgerAccount`, handling default paths and chain IDs.

## Authentication

### Session Tokens

`@cfxdevkit/services/auth/token` provides HMAC-SHA256 based session tokens.

*   **`signSessionToken`**: Encodes `address`, `issuedAt`, `expiresAt`, and optional `claims`. Outputs `body.mac`.
*   **`verifySessionToken`**: Verifies MAC using `timingSafeEqual` and checks expiration. Returns `null` on failure.
*   **`readBearerToken`**: Extracts the token from an `Authorization: Bearer <token>` header.

### Nonce Store

`@cfxdevkit/services/auth/nonces` provides replay protection.

*   **`MemoryNonceStore`**: Stores `NonceEntry` objects with TTL.
*   **`issue(address)`**: Generates a random nonce and stores it. Sweeps expired entries first.
*   **`consume(nonce, address)`**: Marks a nonce as used and validates the address matches.
*   **`sweepExpired`**: Cleans up expired entries.

## Embedded Wallet

`createEmbeddedWalletManager(options)` manages wallets backed by a `KeystoreProvider`. It generates `viem` private keys and stores them using the provider.

**`KeystoreEmbeddedWalletManager` Operations:**
*   **`createWallet(userId)`**: Generates a private key, derives eSpace address and Core address. Stores as `private-key` kind with metadata (`userId`, `espaceAddress`, `coreAddress`, `createdAt`).
*   **`getWallet(userId)`**: Retrieves metadata without exposing key material.
*   **`signerFor(userId)`**: Delegates to `provider.getSigner`.

## Ledger Core APDU Implementation

The Ledger backend supports Conflux Core Space via raw APDU. This is handled in `keystore/ledger/core/`.

### Transaction Serialization

`core/transaction.ts` handles RLP serialization for Core transactions:
*   **`serializeCoreTransaction`**: Dispatches based on `coreType` (`cip1559`, `cip2930`, `legacy`).
*   **`finaliseCoreTransaction`**: Attaches the ledger signature to the serialized transaction.

### APDU Framing

`core/framing.ts` manages chunked APDU exchange:
*   **`exchangeChunks`**: Splits large payloads into 255-byte chunks. Uses `P2_MORE`/`P2_LAST` flags.
*   **`exchangeLegacyChunks`**: Fallback for older app versions.
*   **`exchange`**: Wraps `transport.send` or `transport.exchange`. Handles status words (e.g., `0x9000` success). Maps Ledger status codes to user-friendly error messages via `STATUS_HINTS`.

### Core Signer Flow

`signerFromLedgerCore` creates a `Signer` for Core:
1.  **App Info:** Calls `getCoreLedgerInfo` to determine app version.
2.  **Address:** Calls `getCoreLedgerAddress` to retrieve `coreAddress` and public key.
3.  **Signing:**
    *   `signTransaction`: Calls `signCoreLedgerTransaction` → `exchangeChunks` with `INS_SIGN_TX`.
    *   `signMessage`: Checks app version. If ≥ 2.3.0, calls `signCoreLedgerMessage` → `exchangeChunks` with `INS_SIGN_PERSONAL`. Older versions throw `core-message-unsupported`.

## Crypto Primitives

The `crypto` module exposes low-level primitives used by backends and auth.

*   **`AesGcmKey`**: A branded `Uint8Array` (32 bytes). Constructed via `brandKey` or `generateAesGcmKey`.
*   **`encryptAesGcm` / `decryptAesGcm`**: Wrappers around `@noble/ciphers/aes`. Returns separate `iv` and `tag` for clarity. AAD is supported.
*   **`deriveKeyArgon2id`**: Derives a 32-byte key from a passphrase using Argon2id (RFC 9106 defaults).
*   **`deriveKeyHkdf`**: HKDF-SHA256 for deterministic key expansion.
*   **Encoding:** `toHex`, `fromHex`, `toBase64Url`, `fromBase64Url`. URL-safe base64 without padding.

## Error Handling

All errors extend from `@cfxdevkit/cdk`.
*   **`KeystoreError`**: Thrown by keystore backends and signers. Codes include `services/keystore/not-found`, `services/keystore/bad-passphrase`, `services/keystore/ledger/core-apdu-error`, etc.
*   **`CryptoError`**: Thrown by the crypto module. Codes include `services/crypto/bad-key`, `services/crypto/decrypt-failed`.
