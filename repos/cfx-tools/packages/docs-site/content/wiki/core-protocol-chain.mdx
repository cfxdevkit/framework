# Core Protocol & Chain Module

## Overview

The **Core Protocol & Chain** module constitutes the foundational layer of the `cfx-core` package. It provides the essential abstractions for interacting with the Conflux network, managing cryptographic operations, and handling network-specific configurations. This module encapsulates the logic for RPC communication, wallet management, address derivation, and type safety across both Conflux Core Space and eSpace (EVM-compatible) environments.

## Architecture

The module is organized into distinct functional layers: **Client**, **Wallet**, **Network**, and **Infrastructure**. The `client` layer handles external communication, while the `wallet` layer manages local cryptographic state. Shared utilities reside in `types`, `errors`, and `units`.

## Component Breakdown

### Client Layer (`src/client/`)

Responsible for establishing connections and executing RPC calls against Conflux nodes.

*   **`core.ts`**: Implements the base client logic for the Conflux Core Space. Handles standard RPC methods and state queries specific to the native Conflux protocol.
*   **`espace.ts`**: Implements the client logic for the eSpace (EVM-compatible) environment. Bridges Core Space functionality with Ethereum-compatible RPC standards.
*   **`transport.ts`**: Manages the underlying communication channel (HTTP, WebSocket, etc.). Abstracts network latency, retries, and connection management.
*   **`index.ts`**: Exports the primary client classes and factory functions for initializing connections.

### Wallet Layer (`src/wallet/`)

Handles private key management, transaction signing, and hierarchical deterministic (HD) wallet derivation.

*   **`signing.ts`**: Contains cryptographic primitives for signing transactions and messages. Ensures secure handling of private keys during the signing process.
*   **`derivation.ts`**: Implements BIP32/BIP44 derivation paths specific to Conflux. Converts seed phrases or master keys into usable account addresses.
*   **`index.ts`**: Exports wallet classes and helper functions for account generation and management.

### Network & Address (`src/address/`, `src/chains/`)

Defines network parameters and address formatting rules.

*   **`chains/index.ts`**: Configures chain IDs, network names, and RPC endpoints for supported Conflux networks (Mainnet, Testnet, Devnet).
*   **`address/index.ts`**: Provides utilities for address validation, checksumming, and conversion between different address formats (e.g., hex vs. cfx address).

### Infrastructure (`src/types/`, `src/errors/`, `src/units/`)

Shared definitions and utilities used across the module.

*   **`types/index.ts`**: Defines TypeScript interfaces for RPC responses, transaction objects, and configuration options. Ensures type safety across the SDK.
*   **`errors/index.ts`**: Centralized error handling. Defines custom error classes for network failures, validation errors, and cryptographic failures.
*   **`units/index.ts`**: Handles unit conversions for native tokens (e.g., Drip to CFX). Provides constants for gas limits and pricing calculations.

## Usage Patterns

### Initializing a Client

Clients are instantiated via the `client` module exports. Configuration typically involves specifying the network type (Core or eSpace) and the transport endpoint.

```typescript
// Conceptual usage pattern
import { Client } from '@cfx/core/client';

const client = new Client({
  url: 'https://mainnet.confluxrpc.com',
  network: 'core'
});
```

### Wallet Operations

Wallet operations require importing from the `wallet` module. Signing is decoupled from broadcasting to allow for offline signing workflows.

```typescript
// Conceptual usage pattern
import { Wallet } from '@cfx/core/wallet';

const wallet = new Wallet(privateKey);
const signedTx = await wallet.signTransaction(txData);
```

## Integration Guidelines

1.  **Type Safety**: Always import types from `src/types/index.ts` when constructing transaction objects or handling RPC responses to ensure compatibility with the current protocol version.
2.  **Error Handling**: Catch errors using the classes defined in `src/errors/index.ts`. Do not rely on generic `Error` instances for network-specific failures.
3.  **Network Configuration**: Use `src/chains/index.ts` to retrieve valid chain IDs and RPC endpoints rather than hardcoding values.
4.  **Unit Conversion**: Utilize `src/units/index.ts` for all monetary calculations to prevent precision errors associated with floating-point arithmetic.

## Development Notes

*   **Extensibility**: To add support for a new network, extend the configuration in `src/chains/index.ts` and ensure the `client` layer supports the specific RPC methods required.
*   **Security**: The `wallet` module handles sensitive key material. Ensure that any extensions to `signing.ts` or `derivation.ts` adhere to secure coding practices regarding memory management and key exposure.
*   **Testing**: Unit tests should cover the `transport` layer for network resilience and the `wallet` layer for cryptographic correctness.
