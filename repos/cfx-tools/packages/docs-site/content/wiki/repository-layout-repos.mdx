# Repository Layout — repos

# repos/ Module

## Overview

The `repos/` directory implements a **tier-aligned repository split strategy** defined in [ADR-0003](../docs/adr/0003-multi-repo-split.md). It organizes the codebase into logical slices that are structured to function as standalone git repositories while currently residing within the root monorepo.

This architecture allows for day-to-day development within a single workspace while preparing specific domains for eventual extraction into independent repositories.

## Architecture

### Directory Structure
Each subdirectory within `repos/` is structured as if it were already a standalone repository. Key files present in each slice include:
- `package.json`
- `pnpm-workspace.template.yaml`
- `README.md`

### Workspace Configuration
To maintain monorepo functionality without prematurely isolating slices, the workspace configuration uses a template mechanism:

1.  **Root Workspace:** The top-level `pnpm-workspace.yaml` in `root/` includes a glob pattern `repos/*/packages/*`. This ensures all packages within the slices are recognized by the root pnpm workspace.
2.  **Slice Templates:** Individual slices use `pnpm-workspace.template.yaml` instead of `pnpm-workspace.yaml`.
    -   **Purpose:** Prevents pnpm from treating each slice as a separate workspace during development.
    -   **Benefit:** Maintains cross-slice dependency resolution against `tools/*` and other root-level packages.
    -   **Activation:** At carve-out time, the file is renamed to `pnpm-workspace.yaml`.

## Repository Carve-out Procedure

When a slice is ready to be extracted into its own independent repository, follow this procedure:

1.  **Filter History:**
    Run `git filter-repo --subdirectory-filter repos/<name>` in a fresh clone of the root repository.
2.  **Activate Workspace:**
    Rename the workspace template:
    ```bash
    mv pnpm-workspace.template.yaml pnpm-workspace.yaml
    ```
3.  **Publish:**
    Push the filtered repository to its new remote (e.g., `cfx-keys`).
4.  **Update Root:**
    Remove the slice's entries from the root `pnpm-workspace.yaml`.
5.  **Update Consumers:**
    Update any remaining consumers in the root monorepo to depend on published npm versions of the extracted slice.

## Development Guidelines

-   **Dependency Management:** Do not modify the root `pnpm-workspace.yaml` to include slice-specific configurations. Rely on the root glob for package discovery.
-   **Cross-Slice Dependencies:** Dependencies between slices (e.g., `cfx-ui` depending on `cfx-core`) are resolved via the root workspace.
-   **Tooling:** Tools located in `cfx-tools/` are accessible to all slices during the monorepo phase. Ensure tooling compatibility is maintained before any carve-out occurs.
