# Keystore & Wallet Services

# Keystore & Wallet Services

`@cfxdevkit/services` provides the cryptographic infrastructure, key storage, and wallet management layer for the Conflux developer toolkit. It is organized into four areas: a crypto primitives library, a pluggable keystore system with multiple backends, an embedded wallet manager, and session authentication utilities.

Private key material **never crosses the `KeystoreProvider` boundary** â€” backends accept secrets on `put` and return only `Signer` objects from `getSigner`.

## Crypto Primitives (`src/crypto`)

Self-contained, side-effect-free building blocks used by the keystores.

### Key types and generation

`AesGcmKey` is a branded `Uint8Array` (32 bytes) that the type system prevents from being confused with arbitrary byte arrays. The only ways to produce one are:

```ts
generateAesGcmKey()                    // random, from CSPRNG
deriveKeyArgon2id(input)               // from passphrase
deriveKeyHkdf(input)                   // from existing key material
brandKey(bytes)                        // assert already-correct length
```

`brandKey` throws `CryptoError('services/crypto/bad-key')` if the length is wrong. Do not call it from application code; use one of the three constructors above.

### AEAD encryption

`encryptAesGcm` / `decryptAesGcm` wrap `@noble/ciphers` AES-256-GCM. The `@noble/ciphers` library appends the 16-byte authentication tag to the ciphertext; these functions split it out so callers can store `{ ciphertext, iv, tag }` separately.

```ts
const { ciphertext, iv, tag } = await encryptAesGcm({ key, plaintext, aad });
const plaintext = await decryptAesGcm({ key, ciphertext, iv, tag, aad });
```

The `aad` (Additional Authenticated Data) field binds ciphertext to its identity. The file backend uses `cfx-v1:{service}:{account}` as AAD so a record cannot be transplanted to a different slot without the tag failing.

`decryptAesGcm` throws `CryptoError('services/crypto/decrypt-failed')` on any tag mismatch or corrupt input.

### Key derivation

| Function | Algorithm | Use case |
|---|---|---|
| `deriveKeyArgon2id` | Argon2id (RFC 9106) | Passphrase â†’ file keystore KEK |
| `deriveKeyHkdf` | HKDF-SHA256 | Deterministic sub-key derivation |

Default Argon2id parameters: 64 MiB memory, 3 iterations, parallelism 1. Tests should pass `memKiB: 8` to avoid slow CI runs.

### Encoding helpers

`toHex` / `fromHex` operate on `0x`-prefixed hex. `toBase64Url` / `fromBase64Url` produce unpadded RFC 4648 Â§5 output. Both sets throw `CryptoError` on malformed input rather than returning garbage.

`randomBytes(n)` delegates to `@noble/hashes` and validates that `n` is a non-negative integer before calling through.

## Keystore Provider Interface (`src/keystore/index.ts`)

All backends implement `KeystoreProvider`:

```ts
interface KeystoreProvider {
  readonly id: string;
  readonly capabilities: KeystoreCapabilities;   // { write, list, rotate }

  list(opts?): Promise<StoredSecret[]>;
  has(ref, opts?): Promise<boolean>;
  getSigner(ref, capability?, opts?): Promise<Signer>;

  put?(input, opts?): Promise<void>;             // absent on read-only backends
  updateMeta?(ref, meta, opts?): Promise<void>;
  remove?(ref, opts?): Promise<void>;
  rotate?(ref, opts?): Promise<{ ref }>;
}
```

**`SecretRef`** is the stable address of a secret: `{ service: string; account: string }`. The file and memory backends join these with a null byte to form a map key.

**`Capability`** optionally scopes what a returned `Signer` is allowed to sign. When passed to `getSigner`, the backend wraps the base signer with `applyCapability` (see below). Constraints include `chains`, `contracts`, `selectors`, `maxValuePerTx`, and `notAfter`.

**`StoredSecret`** carries only public metadata (`ref`, `kind`, `createdAt`, `meta`). Key material is never included.

### Capability enforcement

`applyCapability(base, capability)` wraps `signTransaction` with policy checks. `signMessage` and `signTypedData` are delegated unchanged. Violations throw `KeystoreError('services/keystore/unsupported')`. This wrapper is shared across the memory, file, and Ledger backends.

### Audit logging

Every mutating provider method calls `audit.record({ at, provider, action, ref, ok })`. The default is `noopAuditLogger`. For production use, `createAppendOnlyAuditLogger({ path })` writes a newline-delimited JSON log where each entry includes a `sha256` hash of the previous entry, forming a tamper-evident chain. The logger is fire-and-forget (errors are reported via the optional `onError` callback, not thrown).

## Backend Implementations

### Memory (`src/keystore/memory`)

In-memory store for tests only. The module guard throws at load time when `NODE_ENV === 'production'`.

```ts
const provider = createMemoryKeystore({
  seed: [{ ref: { service: 'test', account: 'alice' }, privateKey: '0x...' }],
});
```

Supports `put`, `remove`, `updateMeta`. Accepts `kind: 'private-key'` (64-char hex) and `kind: 'mnemonic'` (BIP-39). Mnemonic derivation uses `m/44'/60'/0'/0/0` unless `derivationPath` is supplied on `getSigner`.

### File (`src/keystore/file`)

Encrypted JSON file, format version `cfx-v1`. The file envelope (`Envelope`) contains KDF parameters and a `secrets` map:

```
{
  version: "cfx-v1",
  kdf: { name: "argon2id", salt, memKiB, iterations, parallelism },
  secrets: {
    "service\0account": { kind, createdAt, meta?, iv, ct, tag }
  }
}
```

**Initialization:**

```ts
await initFileKeystore({ path: './keystore.json', passphrase: 'secret', kdf: { memKiB: 8 } });
const provider = createFileKeystore({ path: './keystore.json', unlock: async () => ({ passphrase }) });
```

`initFileKeystore` fails if the file already exists. `createFileKeystore` derives the Key Encryption Key (KEK) lazily on first use and caches it in memory for the lifetime of the provider instance. It validates the passphrase by attempting to decrypt the first available secret (if any) before caching.

**Passphrase rotation:**

```ts
await changeFilePassphrase({ path, oldPassphrase, newPassphrase });
```

Decrypts every secret under the old KEK, generates a fresh salt, derives a new KEK, re-encrypts all secrets, and atomically renames the file. The old file is not retained.

**Reading a mnemonic directly:**

```ts
const mnemonic = await readFileKeystoreMnemonic({ path, passphrase, ref });
```

Use this for one-shot mnemonic access without instantiating a full provider (e.g., to derive a control-plane key). The plaintext buffer is zeroed after decoding.

**Atomic writes:** `writeEnvelope` writes to a PID/timestamp-suffixed `.tmp` file then calls `fs.rename`, which is atomic on POSIX. File permissions are set to `0o600`.

### Ledger (`src/keystore/ledger`)

Read-only provider (`capabilities.write = false`) that exposes pre-configured derivation paths as `opaque` secrets. Each path produces a `Signer` on demand by communicating with the device.

```ts
const provider = createLedgerKeystore({
  eth,              // LedgerEthAppLike from @ledgerhq/hw-app-eth
  coreTransport,    // LedgerTransportLike for Conflux Core APDU app
  accounts: [
    { ref: { service: 'ledger', account: 'deployer' }, family: 'espace', path: "m/44'/60'/0'/0/0" },
    { ref: { service: 'ledger', account: 'core-0' }, family: 'core', path: "m/503'/0'/0'/0/0" },
  ],
});
```

`signerFromLedger` resolves to a `Signer` whose sign methods communicate with the hardware. For eSpace accounts it uses the Ethereum Ledger app (EIP-1559 transactions only). For Core Space it uses a custom APDU implementation.

**Transport setup:**

```ts
const transport = await createNodeHidLedgerTransport();  // requires @ledgerhq/hw-transport-node-hid
const eth = await createLedgerEthApp(transport);         // requires @ledgerhq/hw-app-eth
```

Both are dynamically imported to avoid bundling peer dependencies. If the packages are absent, a `KeystoreError('services/keystore/ledger/transport-unavailable')` or `app-unavailable` is thrown.

**Core Space APDU layer:** The `core-apdu.ts` module implements `INS_GET_PUBLIC_KEY (0x02)`, `INS_SIGN_TX (0x03)`, `INS_SIGN_PERSONAL (0x04)`, and `INS_GET_APP_INFO (0x01)`. Large payloads are chunked via `exchangeChunks`. Personal message signing requires app version â‰Ą 2.3.0; the signer checks this via `getCoreLedgerInfo` and throws with a clear message if the version is too old.

**APDU status codes:** `core-framing.ts` maintains a table of human-readable error messages for the most common Conflux Core app status words (locked device, wrong app, rejected operation, etc.).

## Embedded Wallet Manager (`src/embedded-wallet`)

Manages per-user custodial wallets stored in any `KeystoreProvider`.

```ts
const manager = createEmbeddedWalletManager({ provider, coreNetworkId: 1029 });

const wallet = await manager.createWallet('user-123');
// wallet: { userId, ref, espaceAddress, coreAddress, createdAt }

const signer = await manager.signerFor('user-123');
await manager.deleteWallet('user-123');
```

`createWallet` is idempotent â€” if a wallet already exists for the user ID, it returns the existing one. Under the hood it calls `generatePrivateKey` (viem), derives both the eSpace address (`privateKeyToAccount`) and the Core address (`coreAddressFromPrivateKey`), then stores the private key via `provider.put` with metadata including both addresses and `createdAt`.

`getWallet` does not decrypt â€” it reads metadata from `provider.list` and reconstructs the `EmbeddedWallet` record. If `espaceAddress` or `coreAddress` is missing from stored metadata, it throws `KeystoreError('services/keystore/unsupported')`.

The default service namespace is `cfxdevkit-embedded-wallet`. Override it with `options.service`.

## Auth Utilities (`src/auth`)

### Nonce store

`MemoryNonceStore` issues single-use challenge nonces for wallet-signature authentication flows.

```ts
const store = createMemoryNonceStore({ ttlMs: 5 * 60 * 1000 });

const nonce = store.issue('0xABC...');          // bind nonce to address
const ok = store.consume(nonce, '0xABC...');    // true once; false if expired/used/wrong address
store.sweepExpired();                            // called automatically on issue
```

`consume` checks: entry exists, not already used, not expired, address matches (case-insensitive). All four must pass for `true` to be returned. The store is in-memory only â€” not suitable for multi-process deployments without a shared backing store.

### Session tokens

Lightweight HMAC-SHA256 tokens in a two-segment `{payload}.{mac}` format (base64url, no header).

```ts
const token = signSessionToken(address, { secret: process.env.SESSION_SECRET, ttlMs: 86400000 });
const payload = verifySessionToken(token, { secret: process.env.SESSION_SECRET });
// payload: { address, issuedAt, expiresAt, claims? } or null

const token = readBearerToken(req.headers.authorization);  // strips "Bearer " prefix
```

`verifySessionToken` uses `timingSafeEqual` for MAC comparison. It returns `null` (never throws) for any malformed, tampered, or expired token. The payload is arbitrary JSON with a fixed `SessionToken` shape; additional claims can be embedded via `options.claims`.

## Error Handling

All errors are instances of `CryptoError` (for crypto primitives) or `KeystoreError` (for all other operations), both extending `CfxError` from `@cfxdevkit/cdk`. Both carry a `code` string, `message`, optional `cause`, and optional `meta` object. Common codes:

| Code | Meaning |
|---|---|
| `services/crypto/bad-key` | Wrong key length, bad hex, invalid random `n` |
| `services/crypto/decrypt-failed` | AES-GCM tag mismatch or corrupt input |
| `services/keystore/not-found` | `SecretRef` not present in backend |
| `services/keystore/bad-passphrase` | Decryption failed with supplied passphrase |
| `services/keystore/backend-unavailable` | File missing or unreadable |
| `services/keystore/unsupported` | Operation not supported by this backend |
| `services/keystore/ledger/device-error` | Ledger transport or app returned an error |
