# projects

# Projects Module

The `projects` module encapsulates the full development stack for Conflux automation, comprising the core non-custodial automation application and a local development showcase environment. It enables the creation of on-chain strategies, management of local runtime environments, and secure key derivation workflows.

## Sub-modules

- **[cas](cas.md)**: Non-custodial automation system for Conflux eSpace. Implements on-chain strategy execution (DCA, TWAP, limit orders) via a Next.js frontend, Express backend with SQLite persistence, and a keeper worker bridging to smart contracts via the Automation Manager.
- **[examples](examples.md)**: Local showcase dashboard (`showcase-local`) for interacting with a local Conflux runtime. Provides a unified interface for keystore management, account derivation, local node control, contract compilation/deployment, and session key verification across Core and eSpace.

### Key Integration Points

- **Runtime Foundation**: `examples` provisions the local blockchain environment (`devnode-server`) and deploys necessary contracts, including the Automation Manager. `cas` connects to this runtime to monitor state and dispatch jobs via the keeper worker.
- **Keystore and Security**: `examples` exposes comprehensive keystore management (memory, OneKey) and session key workflows. These utilities support the non-custodial design of `cas`, allowing developers and users to securely derive and manage the accounts required for on-chain strategy execution.
- **Cross-Chain Normalization**: Both modules operate across Conflux Core and eSpace. `examples` provides shared primitives for chain ID normalization and wallet state synchronization, ensuring consistent cross-chain behavior for strategies defined in `cas`.
- **Thin-Client vs. Persistent State**: `examples` follows a thin-client architecture where Next.js API routes proxy requests to the runtime. In contrast, `cas` maintains persistent state in SQLite and uses an SSE-based event system to notify the frontend of on-chain updates, highlighting its role as the stateful execution layer.

## Common Workflows

- **Local Strategy Development**: Developers use `examples` to compile contracts, deploy the Automation Manager, and derive test accounts. They then configure the `cas` backend against this local runtime to test strategy logic and keeper orchestration without mainnet risk.
- **Environment Provisioning**: `examples` automates the setup of local nodes and account configurations, reducing the friction of initializing the environment required for `cas` deployment.
- **Monitoring and Control**: `examples` provides dashboard panels for real-time node status and account balances, complementing the `cas` frontend which focuses on job creation, slippage configuration, and execution tracking.
