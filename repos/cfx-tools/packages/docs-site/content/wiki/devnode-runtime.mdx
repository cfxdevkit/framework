# @cfxdevkit/devnode & Runtime

This module provides a local development environment for Conflux, wrapping the native `@xcfx/node` binary to spawn dual-space (Core + eSpace) nodes with pre-funded accounts. It includes the `@cfxdevkit/executor` utility library for managing asynchronous tasks, retries, and polling within the runtime ecosystem.

## Overview

The **DevNode** component is designed for development and testing only. It boots a local Conflux node on deterministic ports matching `@cfxdevkit/cdk/chains` defaults (`core-local` / `espace-local`), allowing immediate interaction via standard HTTP/WebSocket transports.

The **Executor** component provides generic asynchronous execution primitives (retry, concurrency, locking, polling) used by the broader runtime infrastructure (e.g., `automation`, `devnode-server`) to manage long-running tasks and state transitions safely.

## CLI Usage

The `cfxdevkit-devnode` binary is the primary entry point for interactive development. It parses arguments, initializes a `DevNode`, and handles process signals for graceful shutdown.

```bash
# Start with defaults (ports 12537/8545, 10 accounts)
cfxdevkit-devnode

# Customize configuration
cfxdevkit-devnode --accounts 4 --balance 1000 --logging

# Disable auto-mining
cfxdevkit-devnode --no-mining
```

### CLI Options

| Flag | Description | Default |
| :--- | :--- | :--- |
| `--core-port` | Core JSON-RPC HTTP port | `12537` |
| `--espace-port` | eSpace JSON-RPC HTTP port | `8545` |
| `--core-ws-port` | Core WebSocket port | `12536` |
| `--espace-ws-port` | eSpace WebSocket port | `8546` |
| `--chain-id` | Core chain ID | `2029` |
| `--evm-chain-id` | eSpace chain ID | `2030` |
| `--accounts` | Pre-funded genesis accounts | `10` |
| `--balance` | Initial balance per account (CFX) | `1000000` |
| `--mining-interval` | Auto-miner tick (ms) | `2000` |
| `--no-mining` | Disable auto-miner | `false` |
| `--mnemonic` | Override BIP-39 mnemonic | Random |
| `--data-dir` | Override data directory | `~/.cfxdevkit/devnode/` |
| `--logging` | Forward node logs to stdout | `false` |

## Programmatic API

### DevNode Lifecycle

Use `createDevNode` to instantiate a node handle. The lifecycle is managed via `start()` and `stop()`.

```typescript
import { createDevNode } from '@cfxdevkit/devnode';

const node = createDevNode({ 
  accounts: 5, 
  miningIntervalMs: 1000 
});

await node.start();
console.log(node.urls.core); // http://127.0.0.1:12537

// Manual mining
await node.packMine(); 

await node.stop();
```

#### Key Methods

*   `createDevNode(config?: DevNodeConfig)`: Constructs a new instance.
*   `start()`: Spawns the `@xcfx/node` server. Throws `devnode/already-running` if called while `running` or `starting`.
*   `stop()`: Halts the auto-miner and shuts down the server.
*   `restart()`: Convenience wrapper for `stop()` followed by `start()`.
*   `getStatus()`: Returns current lifecycle state (`stopped`, `starting`, `running`, `stopping`, `error`).
*   `getMiningStatus()`: Returns current auto-miner state (`enabled`, `intervalMs`, `ticks`).

#### Mining Control

*   `mine(blocks = 1)`: Mines empty Core blocks.
*   `packMine()`: Packs pending Core and eSpace transactions into the next block.
*   `startMining(intervalMs?)`: Enables the auto-miner loop. Prevents concurrent ticks via `autoMineInFlight` flag.
*   `stopMining()`: Disables the auto-miner.

### Executor Utilities

The `@cfxdevkit/executor` module provides primitives for robust task management. These are often used in conjunction with DevNode operations in higher-level services (e.g., `devnode-server`, `automation`).

#### Task Execution

```typescript
import { execute, withLock, createPoller } from '@cfxdevkit/executor';

// Execute with retry logic
const result = await execute(async (ctx) => {
  // ctx.attempt, ctx.signal, ctx.metadata
  return await performAsyncTask();
}, {
  attempts: 3,
  timeoutMs: 5000,
  shouldRetry: (err, attempt) => attempt < 3
});

if (result.ok) {
  console.log(result.value);
}
```

#### Concurrency & Locking

*   `executeBatch(tasks, options)`: Runs multiple tasks with controlled concurrency.
*   `withLock(key, task)`: Ensures mutual exclusion for tasks sharing the same `key`. Uses an in-memory `keyedLocks` map.
*   `createPoller(task, intervalMs)`: Creates a repeating task loop that respects an `AbortSignal`.

## Architecture & Flow

The DevNode lifecycle relies on dynamic imports of `@xcfx/node` and `cive` to avoid bundling native binaries. The Executor module handles the orchestration of these async operations.

### Execution Flow Details

1.  **Initialization**: `main()` in `cli.ts` parses arguments and calls `createDevNode()`.
2.  **Configuration**: `resolveConfig()` in `internals.ts` merges user input with `DEFAULTS`, generates a mnemonic if missing, and validates it via `@cfxdevkit/cdk`.
3.  **Startup**: `DevNode.start()` creates the data directory, instantiates `XcfxServer` (via `server.ts`), and starts the native process.
4.  **Mining**: If `miningIntervalMs > 0`, `startMining()` launches a `setInterval` loop. It checks `autoMineInFlight` to prevent `cive` RPC lockups from concurrent `mine({ numTxs: 1 })` calls.
5.  **Shutdown**: `shutdown()` in `cli.ts` listens for `SIGINT`/`SIGTERM`, calls `node.stop()`, and waits for cleanup before exiting.
6.  **External Integration**: External controllers (e.g., `devnode-server/src/controller.ts`) interact with `DevNode` methods directly. They often wrap these calls in `execute()` or `withLock()` to handle retries or ensure exclusive access during state changes.

## Configuration

The `DevNodeConfig` interface defines all tunable parameters.

```typescript
interface DevNodeConfig {
  chainId?: number;
  evmChainId?: number;
  coreRpcPort?: number;
  evmRpcPort?: number;
  dataDir?: string;
  mnemonic?: string;
  accounts?: number;
  balanceCfx?: string;
  miningIntervalMs?: number; // 0 disables auto-miner
  logging?: boolean;
}
```

**Note on Funding**: `balanceCfx` is primarily for display. The actual genesis funding is hardcoded in `@xcfx/node` based on the number of `genesisSecrets` provided.

## Error Handling

The module throws `DevNodeError` for lifecycle and configuration issues.

| Code | Description |
| :--- | :--- |
| `devnode/already-running` | `start()` called on a running node. |
| `devnode/not-running` | Operation (e.g., `mine`) called on a stopped node. |
| `devnode/start-failed` | Underlying `@xcfx/node` process failed to launch. |
| `devnode/stop-failed` | Server shutdown encountered an error. |
| `devnode/mining-already-running` | `startMining()` called while active. |
| `devnode/mining-not-running` | `stopMining()` called while inactive. |
| `devnode/invalid-config` | Mnemonic validation failed or interval < 100ms. |

```typescript
import { DevNodeError } from '@cfxdevkit/devnode';

try {
  await node.start();
} catch (e) {
  if (e instanceof DevNodeError) {
    console.error(e.code); // 'devnode/start-failed'
    console.error(e.meta); // Additional context
  }
}
```

## Integration Notes

*   **Cross-Module Calls**: The `executor` module is frequently invoked by `automation/src/keeper.ts` (via `createPoller`) and `devnode-server` (via `execute`/`withLock`) to manage DevNode interactions safely.
*   **Transaction Packing**: Functions like `abiCallWrite` (in `src/helpers/abi.ts`) often invoke `node.packMine()` to ensure transaction inclusion before returning results. This flow is protected by `assertRunning()` checks which throw `DevNodeError` if the node is down.
*   **Port Conflicts**: Ensure ports (12537, 8545, etc.) are free before starting. The module does not automatically retry on port binding errors.
