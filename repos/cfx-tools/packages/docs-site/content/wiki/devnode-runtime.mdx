# DevNode & Runtime

# `@cfxdevkit/devnode` — Local Conflux Node & `@cfxdevkit/executor` Runtime Utilities

## Overview

`@cfxdevkit/devnode` wraps the native `@xcfx/node` binary to spin up a dual-space (Core + eSpace) Conflux chain locally. A single `createDevNode().start()` call gives you deterministic ports, pre-funded genesis accounts, and an auto-miner — matching the default chain configs in `@cfxdevkit/cdk/chains` so clients work with zero extra configuration.

`@cfxdevkit/executor` is a companion runtime utility package providing retried task execution, batch concurrency, polling, and keyed mutual exclusion. It has no dependency on `devnode` but is used throughout the wider ecosystem wherever durable async work is needed.

Both packages are **Node.js only**. Never import them from a browser or production bundle.

### `DevNodeConfig` and Defaults

All fields are optional. The resolved defaults match the `coreSpaceLocal` / `espaceLocal` chain definitions in `@cfxdevkit/cdk/chains`:

| Field | Default | Notes |
|---|---|---|
| `chainId` | `2029` | Core Space |
| `evmChainId` | `2030` | eSpace |
| `coreRpcPort` | `12537` | HTTP |
| `evmRpcPort` | `8545` | HTTP |
| `coreWsPort` | `12536` | WebSocket |
| `evmWsPort` | `8546` | WebSocket |
| `accounts` | `10` | Genesis accounts |
| `balanceCfx` | `'10000'` | Display only — xcfx hardcodes 10,000 CFX per genesis secret |
| `miningIntervalMs` | `500` | `0` disables auto-miner |
| `logging` | `false` | Forward xcfx logs to stdout |
| `mnemonic` | random BIP-39 | Deterministic if provided |
| `dataDir` | `~/.cfxdevkit/devnode/<random>` | |

`resolveConfig` in `internals.ts` validates the mnemonic via `validateMnemonic` from `@cfxdevkit/cdk` and throws `DevNodeError({ code: 'devnode/invalid-config' })` on failure.

### Account Derivation

`createAccounts` derives two sets of accounts from the BIP-39 mnemonic using `deriveDualAccount` from `@cfxdevkit/cdk`:

- **`accounts[]`** — `n` standard accounts (index 0…n-1), each with an eSpace address, Core address, public key, private key, and derivation paths.
- **`faucet`** — A dedicated mining account at index 0 with `accountType: 'mining'`. This is set as `miningAuthor` in the xcfx server config and receives block rewards.

Both sets are passed as `genesisSecrets` and `genesisEvmSecrets` to `@xcfx/node`, funding them at startup.

### `DevNode` Class

Construct via `createDevNode(config?)` — do not use `new DevNode()` directly in most cases.

#### Lifecycle

```
stopped → starting → running → stopping → stopped
                   ↘ error ↗
```

`start()` creates the data directory, calls `createXcfxServer`, awaits `server.start()`, then calls `startMining` if `miningIntervalMs > 0`. On failure it cleans up the partially-started server and transitions to `'error'`.

`stop()` stops the auto-miner (best-effort), calls `server.stop()`, nulls the test client, and transitions to `'stopped'`. Calling `stop()` on an already-stopped node is a no-op.

`restart()` is a convenience wrapper: `stop()` then `start()`.

#### RPC URLs

`node.urls` returns the four endpoints after construction (before `start()` is called — the ports are deterministic from config):

```ts
{
  core:     'http://127.0.0.1:12537',
  espace:   'http://127.0.0.1:8545',
  coreWs:   'ws://127.0.0.1:12536',
  espaceWs: 'ws://127.0.0.1:8546',
}
```

#### Mining

The auto-miner calls `cive.createTestClient().mine({ numTxs: 1 })` on the Core RPC at the configured interval. Two guard flags prevent concurrent calls:

- `autoMineInFlight` — set while the timer callback's mine call is pending.
- `packing` — set while a manual `packMine()` call is executing.

The `setInterval` callback skips its tick if either flag is true.

**Manual mining methods:**

| Method | Behaviour |
|---|---|
| `mine(blocks?)` | Mine `n` empty Core blocks, no tx packing |
| `packMine()` | `mine({ numTxs: 1 })` — packs pending txs, increments tick counter |
| `startMining(intervalMs?)` | Start the auto-miner; throws `devnode/mining-already-running` if already running at a different interval |
| `stopMining()` | Clear the interval; throws `devnode/mining-not-running` if not running |
| `getMiningStatus()` | Returns a `MiningStatus` snapshot (enabled, intervalMs, ticks, startedAt) |

The test client (`CiveTestClient`) is created lazily on first call to `requireTestClient()` and cached.

### `DevNodeError`

Extends `CfxError` from `@cfxdevkit/cdk`. Every error thrown by this package uses this type with one of the following codes:

| Code | When |
|---|---|
| `devnode/already-running` | `start()` called while running or starting |
| `devnode/not-running` | Mining or mine methods called before `start()` |
| `devnode/start-failed` | `@xcfx/node` throws during startup |
| `devnode/stop-failed` | `@xcfx/node` throws during shutdown |
| `devnode/mining-already-running` | `startMining` called with a different interval |
| `devnode/mining-not-running` | `stopMining` called when not running |
| `devnode/invalid-config` | Invalid mnemonic or `miningIntervalMs < 100` |

### CLI (`cfxdevkit-devnode`)

The binary parses `process.argv` via `parseArgs`, constructs a `DevNode`, starts it, prints endpoints and account addresses, then waits. `SIGINT` / `SIGTERM` trigger a graceful `node.stop()`.

```
cfxdevkit-devnode
cfxdevkit-devnode --accounts 4 --balance 1000 --logging
cfxdevkit-devnode --no-mining
cfxdevkit-devnode --core-port 13000 --espace-port 9000
```

`--no-mining` is a shortcut that sets `miningIntervalMs: 0` in the resolved config.

### Integration Points

- **`testing/createDevNodeFixture`** — calls `start()` / `stop()` around test suites.
- **`devnode-server/controller`** — proxies all `DevNode` methods over HTTP for the dev server API.
- **`helpers/contract/commands.deployContractCommand`** and **`src/helpers/abi.abiCallWrite`** — call `packMine()` after submitting transactions to force inclusion.
- **Integration tests** in `compiler` and `protocol` packages — call `createDevNode()` directly.

## `@cfxdevkit/executor`

A collection of async execution primitives with no external dependencies beyond the standard runtime.

### `execute<T>(task, options?)`

Runs a single task with optional retry, timeout, and abort signal. Returns `ExecutionResult<T>` — a discriminated union that never throws:

```ts
type ExecutionResult<T> =
  | { ok: true;  value: T;      attempts: number; durationMs: number }
  | { ok: false; error: unknown; attempts: number; durationMs: number }
```

Key options:

| Option | Description |
|---|---|
| `attempts` | Total tries (default 1 — no retry) |
| `delayMs` | Fixed ms or `(attempt, error) => ms` between retries |
| `shouldRetry` | Predicate; defaults to always retry |
| `timeoutMs` | Per-attempt timeout; throws after N ms |
| `signal` | `AbortSignal` to cancel mid-execution |
| `metadata` | Passed through to `ExecutionContext` |

`runWithTimeout` races the task against a `setTimeout`-driven abort. The outer `AbortSignal` is also raced, so cancellation is immediate.

### `executeBatch<T>(tasks, options?)`

Runs an array of tasks with bounded concurrency (`options.concurrency`, default 1). Internally maintains a shared index and a pool of `worker()` coroutines. Each worker calls `execute()` so per-task retry and timeout apply. Set `stopOnError: true` to halt remaining tasks after the first failure.

### `createTaskQueue(options?)`

A thin wrapper around `executeBatch`. Tasks are accumulated with `.add(task)` and drained with `.run()`, which calls `executeBatch` on the queued tasks and fires `onResult` callbacks.

### `createPoller(task, intervalMs)`

Creates a `Poller` that calls `task` in a loop with `intervalMs` delay between iterations. Uses `AbortController` internally; `stop()` aborts the controller and awaits the loop promise.

```ts
interface Poller {
  start(): void;
  stop(): Promise<void>;
  isRunning(): boolean;
}
```

Errors from `task` propagate out of the loop promise only if the signal is not yet aborted — transient errors during a graceful stop are suppressed.

### `withLock<T>(key, task)`

Serialises async tasks sharing the same string `key` using a promise chain. Tasks with different keys run concurrently. The lock entry is removed from the internal `Map` once the chain is no longer needed, preventing unbounded growth.

```ts
// These two calls will execute sequentially:
await Promise.all([
  withLock('my-resource', () => doWork()),
  withLock('my-resource', () => doMoreWork()),
]);
```

`automation/src/keeper.ts` uses both `withLock` and `executeBatch` (via `runAllTicks`) to serialise keeper ticks while processing them in bounded-concurrency batches.
