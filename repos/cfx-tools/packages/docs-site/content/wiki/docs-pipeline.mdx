# Docs Pipeline

# `@cfxdevkit/docs-pipeline`

A CLI tool and programmatic API for keeping the `@cfxdevkit` documentation site in sync with source code. It discovers packages across the monorepo, generates MDX content skeletons, syncs wiki pages, and validates the resulting content.

## Overview

The pipeline has three main concerns:

1. **Package pages** — scan all public packages in the monorepo, generate or update `.mdx` skeletons in `docs-site/content/packages/`, and maintain a Nextra `_meta.js` navigation file.
2. **Wiki sync** — copy `.md` files generated by gitnexus from `.gitnexus/wiki/` into `docs-site/content/wiki/`, converting them to MDX-safe content and transforming raw mermaid fences into JSX `<Mermaid>` components.
3. **Supplementary pages** — regenerate `architecture.mdx` (package tier map) and `coverage.mdx` (Vitest coverage summary) from live filesystem data.

## CLI

The binary entrypoint (`bin.ts`) calls `runCli(process.argv.slice(2))`.

```
pnpm --filter @cfxdevkit/docs-pipeline run docs -- <command> [target]
```

### Commands

| Command | Targets | Effect |
|---|---|---|
| `sync` | `all` `packages` `wiki` `architecture` `coverage` | Regenerate docs content |
| `validate` | `content` `wiki` `wiki-fix` | Check MDX validity / mermaid syntax |

`sync` with no target defaults to `all`. `validate` with no target defaults to `content`.

`sync:all` runs `sync:wiki` → `sync:packages` → `sync:architecture` → `sync:coverage` in sequence.

### Routing

`runCli` (in `run.ts`) calls `findDocsCommand` then `resolveDocsInvocation`, which maps the `(command, target)` pair to a `DocsCommandName` string (e.g. `"sync:packages"`). `runCommand` in `scripts.ts` then lazily imports and executes the relevant module.

## Package Discovery

`discoverDocsPackages(repoRoot)` walks six repo slices across two parent directories each (`packages/` and `infra/`):

- `repos/cfx-core`, `repos/cfx-keys`, `repos/cfx-ui`, `repos/cfx-solidity`, `repos/cfx-tools`, `repos/cfx-domain`

For each directory it reads `package.json`, skips private packages and a hard-coded exclusion list (config-only packages like `@cfxdevkit/biome-config`), and collects `name`, `description`, `exports`, `README.md`, and `API.md`.

`discoverDocsPagePackages` wraps this and adds `slug` (derived from `toSlug`) and `skeletonHash` for use by the sync and LLM pipeline layers.

## Package Page Sync

`syncPackages()` (in `package-pages.ts`) is the main entry point for package documentation.

### Skeleton generation

`generateMdxSkeleton` in `package-content.ts` builds an MDX file with:

- YAML frontmatter (`title`, `description`)
- An install block with pnpm/npm/yarn tabs
- Content sourced from `README.md` (with the h1, lead blockquote, and `## Install` section stripped) and `API.md`, or a generated placeholder if neither exists
- A sub-paths table when the package exports multiple entry points

Raw Markdown is sanitized via `toMdxSafe` before embedding: HTML comments are removed, ``...`` references are converted to code spans, and HexAddress template literals are replaced with `HexAddress`.

### Change detection

Each generated file embeds a SHA-256 hash as an MDX comment at the end:

```
{/* readme-hash: <hex> */}
```

The hash covers the sync version constant, package name, description, sorted export sub-paths, README content, and API content. On subsequent runs `syncPackagePage` reads this hash and skips writing if it matches and the file has no stale content patterns (leftover HTML comments, `{@link}`) and compiles as valid MDX.

### Navigation metadata

After processing all packages, `updateMetaFile` writes `_meta.js` (sorted alphabetically) and `syncIndexCards` fixes any `Cards.Card` href slugs in `index.mdx` to match current slugs.

## Wiki Sync

`syncWiki()` reads `.md` files from `.gitnexus/wiki/`, skipping `other.md`, `meta.json`, `module_tree.json`, `index.html`, and any `other-*.md` files (low-value generated metadata pages).

Each file is processed by `toMdxSafeContent`, which:

1. Strips HTML comments and sanitizes `{@link}` and template literal types (same as package pages).
2. Replaces ```` ```mermaid ``` ```` fences with `<Mermaid chart={...} />` JSX after two transformations:
   - `simplifyMermaidDiagram` — if a diagram exceeds 30 content lines, prunes it to top-level edges only, appending a comment noting the original size.
   - `fixMermaidLabels` — quotes node labels that contain characters Mermaid's parser rejects (`@ / & < > ( ) { } #`), and normalises pipe characters inside labels.

The output is wrapped in YAML frontmatter and, if `<Mermaid>` components are present, a `import { Mermaid }` statement. Stale `.mdx` files whose source `.md` no longer exists are deleted.

---

## Wiki Validation

`validateWikiMermaid(options)` scans all `.mdx` files in `content/wiki/` and extracts both `<Mermaid chart={...} />` JSX blocks and raw ```` ```mermaid ``` ```` fences.

Each block is checked by:

- **`heuristicCheck`** — validates diagram type keyword, direction argument for `flowchart`/`graph`, catches unquoted `@` characters in node labels, checks `subgraph`/`end` balance, and checks bracket balance.
- **`mermaidParse`** — calls `mermaid.parse()` if available (requires a DOM; silently falls back to heuristic-only in headless environments).

With `{ fix: true }` (the `validate:wiki-fix` command), broken blocks are replaced in-place with a plaintext fallback:

```
> ⚠️ *Diagram syntax error — rendering skipped. Raw source:*

​```text
<original diagram>
​```
```

---

## Content Validation

`validateGeneratedContent()` recursively collects all `.mdx` files under `docs-site/content/` and compiles each with `@mdx-js/mdx` (with `remark-gfm` and `remark-frontmatter`). Files that fail to compile are reported; the process exits with code 1 if any errors exist.

---

## Architecture and Coverage Pages

Both are fully regenerated on each run — do not edit the output files manually.

**`syncArchitecturePage`** discovers all public packages, groups them into their tier (0 = framework, 1 = developer platform, 2 = domain logic), and writes `content/architecture.mdx` with an ASCII tier diagram and a sortable package map table.

**`syncCoveragePage`** looks for `coverage/coverage-summary.json` (or `coverage-summary.json`) next to each package and writes `content/coverage.mdx` with a statement/branch/function/line percentage table. Packages without coverage data are listed separately.

## Programmatic API

The package exports everything needed to build higher-level tooling on top of the pipeline.

### Core discovery

```ts
import { discoverDocsPackages, discoverDocsPagePackages } from '@cfxdevkit/docs-pipeline';

const packages = await discoverDocsPackages(repoRoot);
const pageTargets = await discoverDocsPagePackages({ packageName: '@cfxdevkit/cdk' });
```

### Content utilities

```ts
import {
  computeSkeletonHash,
  generateMdxSkeleton,   // not exported — internal to package-pages
  embedHash,
  readEmbeddedHash,
  stripEmbeddedHash,
  toSlug,
  extractDescription,
  isValidGeneratedMdx,
} from '@cfxdevkit/docs-pipeline';
```

### Pipeline layer (`pipeline/`)

The `pipeline/` sub-modules expose stable, higher-level wrappers used by the LLM agent workers:

| Export | Source | Purpose |
|---|---|---|
| `syncPackagePageSkeletons` | `pipeline/package-pages.ts` | Delegates to `syncPackages` |
| `discoverPackagePageTargets` | `pipeline/package-pages.ts` | Delegates to `discoverDocsPagePackages` |
| `getPackagePagePath` | `pipeline/package-pages.ts` | Resolves `.mdx` path for a target |
| `discoverApiTargets` | `pipeline/api.ts` | Lists packages with an `API.md` |
| `refreshApiSkeletons` | `pipeline/api.ts` | Runs `pnpm run gen:api` in workspace root |
| `refreshDocsAlignmentArtifacts` | `pipeline/api.ts` | Runs `pnpm run check:docs`, captures output |
| `syncWikiContent` | `pipeline/wiki.ts` | Wraps `syncWiki` |
| `validateWikiContent` | `pipeline/wiki.ts` | Wraps `validateWikiMermaid` |
| `getDocsPipelineReviewContext` | `pipeline/context.ts` | Returns file paths for LLM review context |

### Namespace registration

```ts
import { docsToolingNamespace } from '@cfxdevkit/docs-pipeline';
// { name: 'docs', description, commands, run: runCli }
```

This object is consumed by the tools registry to register the `docs` command namespace alongside other tooling namespaces.

## Workspace Utilities

`findRepoRoot()` delegates to `@cfxdevkit/workspace-utils`, which walks up from `process.cwd()` until it finds a directory containing both `pnpm-workspace.yaml` and `package.json`. `getDocsSitePaths(repoRoot?)` derives all content directory paths from that root and is the single source of truth for path resolution across the pipeline.
