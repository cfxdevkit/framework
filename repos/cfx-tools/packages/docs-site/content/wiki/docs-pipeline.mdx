# Docs Pipeline

# Docs Pipeline Module

The **Docs Pipeline** module (`@cfxdevkit/docs-pipeline`) provides the command-line interface and orchestration logic for managing documentation workflows within the `cfx-tools` monorepo. It handles synchronization of generated content (packages, wiki, architecture, coverage), validation of documentation artifacts, and interaction with the documentation site (`docs-site`).

This module acts as the central coordinator for documentation CI/CD tasks, ensuring consistency between source code, generated API docs, and the published documentation site.

## CLI Usage

The pipeline is executed via the `docs` command. It is typically invoked through `pnpm` filters targeting the package.

```bash
pnpm --filter @cfxdevkit/docs-pipeline run docs -- <command> [options]
```

### Commands

| Command | Usage | Description |
| :--- | :--- | :--- |
| **sync** | `sync [all\|packages\|wiki\|architecture\|coverage]` | Sync generated docs content to the docs site. |
| **validate** | `validate [content\|wiki\|wiki-fix]` | Validate generated docs content for syntax or structure. |

### Help

```bash
pnpm --filter @cfxdevkit/docs-pipeline run docs -- help
```

## Architecture

The CLI execution flow follows a strict resolution path from entry point to specific task implementation.

*   **Entry:** `src/bin.ts` invokes `runCli`.
*   **Resolution:** `src/commands.ts` (`resolveDocsInvocation`) parses arguments into internal `DocsCommandName` types.
*   **Execution:** `src/scripts.ts` (`runCommand`) dynamically imports and executes the specific pipeline logic.
*   **Workspace:** `src/workspace.ts` manages path resolution relative to the monorepo root.

## Command Reference

### Sync Commands

Synchronization commands update the documentation site content based on source artifacts.

*   **`sync:all`**: Executes all sync commands sequentially (`wiki`, `packages`, `architecture`, `coverage`).
*   **`sync:packages`**: Updates package documentation pages. Implemented via `syncPackages` in `src/package/pages.js`.
*   **`sync:wiki`**: Syncs wiki content. Implemented via `syncWiki` in `src/wiki/sync.js`.
*   **`sync:architecture`**: Syncs architecture documentation. Implemented via `syncArchitecturePage` in `src/sync/architecture.js`.
*   **`sync:coverage`**: Syncs coverage documentation. Implemented via `syncCoveragePage` in `src/sync/coverage.js`.

### Validate Commands

Validation commands ensure generated content meets quality standards.

*   **`validate:content`**: Compiles all `.mdx` files in the docs site to check for syntax errors. Implemented via `validateGeneratedContent` in `src/validate-content.ts`.
*   **`validate:wiki`**: Validates Mermaid diagrams in wiki content. Implemented via `validateWikiMermaid` in `src/wiki/validate.js`.
*   **`validate:wiki-fix`**: Attempts to automatically fix Mermaid diagram issues in wiki content. Implemented via `validateWikiMermaid` with `{ fix: true }`.

## Programmatic API

The module exports utilities for use in other scripts or CI workflows.

### Command Execution

```typescript
import { runCommand, type DocsCommandName } from './scripts.js';

// Execute a specific command programmatically
await runCommand('sync:packages');
await runCommand('validate:content');
```

**Type:** `DocsCommandName`
```typescript
| 'sync:wiki' | 'sync:all' | 'sync:packages' | 'sync:architecture' | 'sync:coverage'
| 'validate:content' | 'validate:wiki' | 'validate:wiki-fix'
```

### Pipeline Orchestration

High-level pipeline functions reside in `src/pipeline/`.

*   **`refreshApiSkeletons(options?)`** (`src/pipeline/api.ts`): Triggers `pnpm run gen:api` in the workspace to regenerate API skeletons.
*   **`refreshDocsAlignmentArtifacts(options?)`** (`src/pipeline/api.ts`): Triggers `pnpm run check:docs` and returns the output string.
*   **`discoverApiTargets(options?)`** (`src/pipeline/api.ts`): Returns a list of `DocsApiTarget` objects for packages containing `API.md`.
*   **`syncWikiContent()`** (`src/pipeline/wiki.ts`): Wrapper for `syncWiki`.
*   **`validateWikiContent(options?)`** (`src/pipeline/wiki.ts`): Wrapper for `validateWikiMermaid`.
*   **`syncPackagePageSkeletons()`** (`src/pipeline/package-pages.ts`): Wrapper for `syncPackages`.
*   **`discoverPackagePageTargets(options?)`** (`src/pipeline/package-pages.ts`): Discovers package page targets.
*   **`getPackagePagePath(target, repoRoot?)`** (`src/pipeline/package-pages.ts`): Resolves the file path for a specific package page MDX file.

### Content Utilities

*   **`computeSkeletonHash`, `embedHash`, `readEmbeddedHash`, `stripEmbeddedHash`** (`src/package/content.ts`): Utilities for managing content hashes embedded within documentation files to track changes.
*   **`toSlug`** (`src/package/content.ts`): Converts strings to URL-safe slugs.
*   **`extractDescription`** (`src/package/content.ts`): Extracts description metadata from content.

### Workspace & Paths

*   **`findRepoRoot(startDir?)`** (`src/workspace.ts`): Locates the monorepo root by searching for `pnpm-workspace.yaml` and `package.json`.
*   **`getDocsSitePaths(repoRoot?)`** (`src/workspace.ts`): Returns an object containing absolute paths for `contentDir`, `packagesContentDir`, `wikiContentDir`, `scriptsDir`, etc.
*   **`getDocsPipelineReviewContext()`** (`src/pipeline/context.ts`): Returns a list of file paths relevant for pipeline review contexts.

### Process Execution

*   **`runWorkspacePnpm(args, options?)`** (`src/pipeline/process.ts`): Spawns a `pnpm` process at the workspace root. Inherit stdio. Rejects on non-zero exit.
*   **`captureWorkspacePnpm(args, options?)`** (`src/pipeline/process.ts`): Spawns a `pnpm` process and captures stdout/stderr as a string.

## Workspace Integration

The module relies heavily on the monorepo structure defined by `@cfxdevkit/workspace-utils`.

### Path Resolution

All file operations assume a standard layout relative to the workspace root:
*   **Docs Site:** `repos/cfx-tools/packages/docs-site`
*   **Content:** `repos/cfx-tools/packages/docs-site/content`
*   **Packages Content:** `content/packages`
*   **Wiki Content:** `content/wiki`

Use `getDocsSitePaths()` to resolve these locations dynamically rather than hardcoding paths.

### Pnpm Interactions

Pipeline tasks often invoke workspace scripts defined in `package.json` files (e.g., `gen:api`, `check:docs`). Use `runWorkspacePnpm` or `captureWorkspacePnpm` to execute these commands safely from within the module.

```typescript
import { runWorkspacePnpm } from './pipeline/process.js';

// Runs `pnpm run gen:api` at the workspace root
await runWorkspacePnpm(['run', 'gen:api']);
```

## Validation Logic

The `validateGeneratedContent` function ensures documentation files are valid MDX.

1.  **Discovery:** Recursively scans the `contentDir` for `.mdx` files using `collectMdx`.
2.  **Compilation:** Attempts to compile each file using `@mdx-js/mdx` with `remarkGfm` and `remarkFrontmatter`.
3.  **Reporting:** Logs success/failure per file and returns a `ValidationResult` containing error counts and details.

```typescript
import { validateGeneratedContent } from './validate-content.js';

const result = await validateGeneratedContent();
if (result.errors.length > 0) {
  process.exitCode = 1;
}
```

## Contribution Guidelines

*   **Adding Commands:** Define the command in `src/commands.ts` (`docsCommands` array) and implement the logic in `src/scripts.ts` (`runCommand`).
*   **Path Changes:** If the docs site structure changes, update `getDocsSitePaths` in `src/workspace.ts`.
*   **New Sync Targets:** Create a new sync function (e.g., `src/sync/new-target.ts`) and add a corresponding `sync:new-target` case in `scripts.ts`.
*   **Validation:** Add new validation rules in `src/validate-content.ts` or create new validators in `src/wiki/validate.ts`.
