# CLI & Developer Tools

# CLI & Developer Tools

This module provides the command-line interface and project scaffolding utilities for the Conflux Developer Kit ecosystem. It consists of two primary packages:

- **`@cfxdevkit/cli`**: A minimal CLI for interacting with Conflux chains, deriving accounts, and managing contract artifacts.
- **`@cfxdevkit/create`** (`scaffold-cli`): A project scaffolding tool for generating new dApp templates.

Both packages rely on `@cfxdevkit/cdk` for core blockchain functionality but maintain zero external CLI dependencies (e.g., no `commander` or `yargs`) to keep the install footprint low.

## Architecture

The CLI module is structured around a minimal argument parser and a command router. The scaffold module focuses on template rendering and project initialization.

### CLI Execution Flow

The CLI entry point (`bin.ts`) invokes the `run` function, which parses arguments and dispatches to specific command handlers.

## Package: `@cfxdevkit/cli`

The CLI package exposes the `cfx` command. It is designed for developer tooling such as chain health checks, account management, and build automation.

### Entry Point

The CLI is bootstrapped via `src/bin.ts`, which calls `run` from `src/run.ts`.

```typescript
// src/bin.ts
import { run } from './run.js';
const argv = process.argv.slice(2);
run(argv).then((code) => process.exit(code));
```

### Argument Parsing

`src/args.ts` implements a lightweight parser to avoid heavy dependencies. It supports boolean flags, key-value pairs, and positional arguments.

**Key Functions:**
- `parseArgs(argv: readonly string[])`: Parses `process.argv` into `ParsedArgs`.
- `getString(flags, key)`: Retrieves string flag values.
- `getNumber(flags, key)`: Retrieves numeric flag values.
- `getBool(flags, key)`: Retrieves boolean flag values.

```typescript
// Example: src/args.ts
export function parseArgs(argv: readonly string[]): ParsedArgs {
  // ... implementation
}
```

### Commands

The `run` function in `src/run.ts` routes commands to specific handlers.

#### 1. `status`
Pings Conflux chains to verify connectivity and retrieve block/epoch numbers.
- **Handler:** `statusFromFlags`
- **Core Logic:** `runStatus` → `pingChain`
- **Dependencies:** `@cfxdevkit/cdk` (`createClient`, `http`, `getChain`)
- **Usage:**
  ```bash
  cfx status --chain core-testnet
  cfx status --json
  ```

#### 2. `derive`
Derives dual-space accounts (EVM + Core) from a mnemonic.
- **Handler:** `deriveFromFlags`
- **Core Logic:** `runDerive`
- **Dependencies:** `@cfxdevkit/cdk` (`deriveDualAccounts`, `generateMnemonic`)
- **Usage:**
  ```bash
  cfx derive --generate --count 3 --core-network-id 1
  cfx derive --mnemonic "..." --show-private-keys
  ```

#### 3. `generate`
Generates a fresh BIP-39 mnemonic.
- **Handler:** `generateFromFlags`
- **Core Logic:** `runGenerate`
- **Dependencies:** `@cfxdevkit/cdk` (`generateMnemonic`)
- **Usage:**
  ```bash
  cfx generate --strength 256
  ```

#### 4. `contracts extract`
Extracts ABI and bytecode from Hardhat artifacts into generated TypeScript modules.
- **Handler:** `contractsExtractFromFlags`
- **Core Logic:** `runContractsExtract`
- **Dependencies:** `@cfxdevkit/codegen-contracts` (`writeContractModules`)
- **Usage:**
  ```bash
  cfx contracts extract --artifacts ./artifacts --out ./src/generated
  ```

## Package: `@cfxdevkit/create` (Scaffold)

This package provides the `create` command for bootstrapping new projects using predefined templates.

### Scaffolding Logic

**Key Functions:**

### Template System

Templates are defined in `src/templates.ts` and specific template files (e.g., `minimal-dapp.ts`).

**Template Structure:**
- **`TemplateDefinition`**: Defines `name`, `description`, `files`, and optional `targets`.
- **`TemplateTarget`**: Supports `default`, `devcontainer`, and `docker` configurations.
- **`renderFile`**: Replaces `{{variable}}` placeholders in file content.

**Available Templates:**
1.  **`minimal-dapp`**: Vite + React + `@cfxdevkit/react`.
2.  **`wallet-probe`**: Wallet detection and signing demo.
3.  **`project-example`**: Full-stack (Vite frontend + Hono backend + Solidity).

**Adding a Template:**
1.  Create a file in `src/templates/` exporting a `TemplateDefinition`.
2.  Register it in the `templates` array in `src/templates.ts`.
3.  Ensure `getTemplate` can find it by name.

```typescript
// src/templates.ts
export function getTemplate(name: string): TemplateDefinition | undefined {
  const template = templates.find((item) => item.name === name);
  return template ? { ...template, files: [...template.files] } : undefined;
}
```

### Argument Parsing

The scaffold CLI uses a custom parser in `src/args.ts` distinct from the main CLI package.

- **`parseArgs(args: string[])`**: Returns `ParsedArgs` containing `template`, `target`, `force`, `skipInstall`, and `positional` arguments.
- **Flags:**
  - `-t` / `--template`: Select template.
  - `--target`: Select deployment target (e.g., `devcontainer`).
  - `--force`: Overwrite existing directory.
  - `--skip-install`: Skip `npm install`.

## Contributing

### Adding a CLI Command
1.  Create a new file in `packages/cli/src/commands/`.
2.  Implement `run<CommandName>` (core logic) and `<CommandName>FromFlags` (CLI interface).
3.  Register the command in the `switch` statement within `packages/cli/src/run.ts`.
4.  Update `HELP` text in `run.ts`.

### Adding a Scaffold Template
1.  Define the template in `packages/scaffold-cli/src/templates/`.
2.  Import and add the definition to the `templates` array in `packages/scaffold-cli/src/templates.ts`.
3.  Ensure `package.json` dependencies match the target environment (e.g., Vite, React, Wagmi).

### Testing
- **CLI:** Tests exist in `index.test.ts`, `args.test.ts`. Focus on `parseArgs` and command output.
- **Scaffold:** Tests exist in `scaffold.test.ts`, `templates.test.ts`. Focus on `scaffoldProject` and `renderFile`.
