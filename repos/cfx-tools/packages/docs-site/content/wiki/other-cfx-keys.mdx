# Other — cfx-keys

# cfx-keys Module Documentation

## Overview

The `cfx-keys` module provides cryptographic key management, keystore implementations, and wallet abstractions for Conflux blockchain operations. It consists of two packages:

- **`@cfxdevkit/services`** — Pluggable backends for keystore, crypto, and audit logging
- **`@cfxdevkit/wallet`** — Session keys, signers, hardware wallet adapters, and capability policies

This module is designated as **Tier 0b — audit-grade trust boundary** per ADR-0003, meaning it handles sensitive key material and requires rigorous security review.

## Package: @cfxdevkit/services

### Purpose

Provides pluggable backend implementations for cryptographic operations, keystore management, and audit logging. Designed to be consumed by the wallet package and other consumers requiring secure key storage.

### Exports

```typescript
// Package root
export * from './auth/index.js';
export * from './embedded-wallet/index.js';
export * from './keystore/audit.js';
export * from './keystore/index.js';

// Sub-path exports
export * from './crypto/index.js';
export * from './keystore/memory/index.js';
export * from './keystore/file/index.js';
export * from './keystore/ledger/index.js';
```

### Crypto Utilities

Located in `src/crypto/`, provides cryptographic primitives:

| Function | Purpose |
|----------|---------|
| `encryptAesGcm` / `decryptAesGcm` | AES-256-GCM encryption with AAD support |
| `deriveKeyArgon2id` | Password-based key derivation (Argon2id) |
| `deriveKeyHkdf` | Key derivation using HKDF |
| `generateAesGcmKey` | Generate random AES-256 key |
| `randomBytes` | Cryptographically secure random bytes |
| `toHex` / `fromHex` | Hex encoding/decoding with `0x` prefix |
| `toBase64Url` / `fromBase64Url` | URL-safe base64 encoding |

**Constants:**
```typescript
export const AES_KEY_LEN = 32;      // 256 bits
export const GCM_IV_LEN = 12;       // 96 bits
export const GCM_TAG_LEN = 16;      // 128 bits
```

**Error Handling:**
All crypto operations throw `CryptoError` on failure. Tests verify:
- Roundtrip encryption/decryption
- AAD tampering detection
- Ciphertext tampering detection
- Salt length validation for KDFs

### Keystore Implementations

The keystore abstraction provides a unified interface for storing and retrieving secrets:

```typescript
interface Keystore {
  has(ref: KeyRef): Promise<boolean>;
  list(): Promise<KeystoreItem[]>;
  getSigner(ref: KeyRef): Promise<Signer>;
  put?(item: KeystoreItem): Promise<void>;
  remove?(ref: KeyRef): Promise<void>;
}
```

#### Memory Keystore

In-memory storage for testing and ephemeral use:

```typescript
import { createMemoryKeystore } from '@cfxdevkit/services/keystore-memory';

const ks = createMemoryKeystore({
  seed: [{
    ref: { service: 'cfxdevkit', account: 'deployer' },
    privateKey: '0x...',
    meta: { label: 'test' }
  }],
  audit: { record: (e) => console.log(e) }
});

const signer = await ks.getSigner({ service: 'cfxdevkit', account: 'deployer' });
```

**Features:**
- Seeded with initial secrets
- Supports capability constraints via `getSigner(ref, capability)`
- Audit logging integration
- Secrets never exposed through `list()`

#### File Keystore

Encrypted file-based storage with passphrase protection:

```typescript
import {
  initFileKeystore,
  createFileKeystore,
  changeFilePassphrase
} from '@cfxdevkit/services/keystore-file';

// Initialize (creates file, refuses to overwrite)
await initFileKeystore({
  path: './keystore.json',
  passphrase: 'strong-passphrase',
  kdf: { memKiB: 1024, iterations: 10 }
});

// Open and use
const ks = createFileKeystore({
  path: './keystore.json',
  unlock: async () => ({ passphrase: 'strong-passphrase' })
});

// Change passphrase (re-encrypts all secrets)
await changeFilePassphrase({
  path: './keystore.json',
  oldPassphrase: 'old',
  newPassphrase: 'new'
});
```

**Security Features:**
- Argon2id key derivation (configurable memory/iterations)
- AES-256-GCM encryption with AAD
- Passphrase validation on unlock
- Refuses to overwrite existing keystore files
- Secrets never exposed through `list()`

#### Ledger Keystore

Hardware wallet integration for Conflux eSpace and Core Space:

```typescript
import { createLedgerKeystore, signerFromLedger } from '@cfxdevkit/services/keystore-ledger';

// eSpace (uses Ledger Ethereum app)
const ethProvider = createLedgerKeystore({
  eth: await createLedgerEthApp(),
  accounts: [{
    ref: { service: 'deploy', account: 'ledger-0' },
    path: "m/44'/60'/0'/0/0"
  }]
});

// Core Space (uses Conflux Ledger app)
const coreSigner = await signerFromLedger({
  family: 'core',
  coreTransport: await createNodeHidLedgerTransport(),
  chainId: 1029
});
```

**Supported Operations:**
- Address derivation and display on device
- Personal message signing
- Transaction signing (EIP-1559 for eSpace, legacy for Core)
- EIP-712 typed data signing (eSpace)

**APDU Error Handling:**
The Ledger adapter provides human-readable error messages for common APDU status codes:

| Status Code | Error Message |
|-------------|---------------|
| `0x650e` | "Open the Conflux Core app" |
| `0x6a15` | "Wrong Ledger app" |
| `0x6e03` | "rejected the APDU length" |
| `0x6a86` | "does not expose INS=0x04 personal-signing" |
| `0x5515` | "Ledger is locked or not ready" |

### Audit Logger

Append-only, hash-chained audit logging for keystore operations:

```typescript
import { createAppendOnlyAuditLogger } from '@cfxdevkit/services/keystore-audit';

const logger = createAppendOnlyAuditLogger({ path: './audit.log' });

logger.record({
  at: 1,
  provider: 'file',
  action: 'put',
  ok: true
});
```

**Log Format:**
```json
{
  "sequence": 1,
  "previousHash": "0x0000...",
  "entryHash": "0xabc123...",
  "timestamp": 1234567890,
  "provider": "file",
  "action": "put",
  "ok": true
}
```

**Security Properties:**
- Hash-chained entries (each entry includes hash of previous)
- Append-only (no modification of existing entries)
- Sequence numbers for ordering
- Suitable for compliance/audit requirements

### Session Keys

Create ephemeral signing keys with capability constraints:

```typescript
import { createSessionKey } from '@cfxdevkit/wallet/session-key';

const session = await createSessionKey({
  parent: parentSigner,
  capability: {
    chains: [1030],
    maxValuePerTx: 1000n,
    notAfter: Date.now() + 60_000,
    contracts: ['0x...'],
    selectors: ['0xa9059cbb']
  },
  privateKey: sessionPrivateKey
});

// Use session signer (enforces capability)
await session.signer.signTransaction(tx);

// Revoke session
session.revoke();
```

**Attestation Format:**
```json
{
  "v": 1,
  "type": "cfxdevkit.session-key.v1",
  "parent": "0x...",
  "session": "0x...",
  "capability": { ... },
  "signature": "0x..."
}
```

**Security Properties:**
- Parent signs attestation (verifiable on-chain)
- Capability constraints enforced at signer level
- Revocation flag prevents further signing
- Session address differs from parent

### Policy Engine

Capability-based access control for signers:

```typescript
import { withCapability, checkCapability } from '@cfxdevkit/wallet/policies';

const cappedSigner = withCapability(baseSigner, {
  chains: [1030],
  contracts: ['0x...'],
  selectors: ['0xa9059cbb'],
  maxValuePerTx: 1000n,
  notAfter: Date.now() + 60_000
});

// Check without executing
const error = checkCapability(capability, tx);
if (error) {
  console.log(`Transaction denied: ${error.message}`);
}
```

**Capability Fields:**

| Field | Purpose |
|-------|---------|
| `chains` | Allowed chain IDs |
| `contracts` | Allowed target addresses |
| `selectors` | Allowed function selectors |
| `maxValuePerTx` | Maximum transaction value |
| `notAfter` | Expiration timestamp |

**Error Codes:**
- `wallet/policies/chain-denied` — Chain not permitted
- `wallet/policies/contract-denied` — Target not permitted
- `wallet/policies/selector-denied` — Function not permitted
- `wallet/policies/value-exceeded` — Value exceeds limit
- `wallet/policies/expired` — Capability expired

## Security Considerations

### Key Material Handling

- **Never log** private keys, mnemonics, or passphrases
- **Memory keystore** is for testing only — use file or hardware for production
- **File keystore** encrypts secrets at rest with AES-256-GCM
- **Hardware wallets** never expose private keys to host

### Capability Enforcement

- Capabilities are enforced at the **signer level**, not just the keystore
- Session keys provide **ephemeral, revocable** access
- Empty capabilities (`{}`) are rejected — always specify constraints

### Audit Requirements

- All keystore operations should be **audit-logged**
- Audit logs are **hash-chained** for tamper detection
- Logs should be stored **separately** from keystore data

### Hardware Wallet Verification

- Always verify **device address** matches expected address
- Handle **APDU errors** with user-friendly messages
- Support **fallback paths** (e.g., legacy APDU when new fails)
