# Repository Layout — repos

## Overview

The `repos/` directory implements a **tier-aligned repository split strategy** defined in [ADR-0003](../docs/adr/0003-multi-repo-split.md). It organizes the codebase into logical slices that function as standalone git repositories while currently residing within the root monorepo. This architecture enables day-to-day development within a single workspace while preparing specific domains for eventual extraction into independent repositories.

Each subdirectory within `repos/` is structured as if it were already a standalone repository, containing key files such as `package.json`, `pnpm-workspace.template.yaml`, and `README.md`.

## Architecture

The ecosystem is divided into specific tiers based on stability, security boundaries, and release cadence. Dependencies flow strictly from lower tiers (stable/core) to higher tiers (verticals/tools), ensuring that core primitives remain decoupled from application logic.

## Module Interactions

### Core Runtime (Tier 0)
The foundation of the ecosystem consists of stable, low-level primitives that decouple release cadences from higher-level logic.
*   **[cfx-core](cfx-core.md):** Provides chain primitives, types, and protocol bindings. It serves as the primary dependency for UI and Domain layers.
*   **[cfx-keys](cfx-keys.md):** Defines the audit-grade trust boundary. All cryptographic operations and signing logic are isolated here to reduce audit scope. Consumers use `@cfxdevkit/services` for primitives and `@cfxdevkit/wallet` for orchestration.
*   **[cfx-solidity](cfx-solidity.md):** Manages the Solidity compilation pipeline, standard ABIs, and contract bindings. It is usable independently without pulling the rest of the DevKit.
*   **[cfx-ui](cfx-ui.md):** Provides React-based UI surfaces. It couples to `cfx-core` and `cfx-keys` through stable npm ranges, ensuring chain code is not blocked by UI iteration.

### Vertical Logic (Tier 2)
*   **[cfx-domain](cfx-domain.md):** Houses vertical-specific logic (e.g., `game-engine`, `automation`, `hardware-bridge`). It adheres to strict dependency rules: it may depend on internal SDKs (`@cfxdevkit/cdk`, `@cfxdevkit/wallet`) but is **forbidden** from depending on `cfx-ui` or `cfx-tools`. Packages split into independent repositories when a second distinct consumer arrives.

### Developer Experience (Tier 1)
*   **[cfx-tools](cfx-tools.md):** Contains CLI, scaffolding, and IDE integrations. It operates on a fast release cadence (weekly) and is **never** a runtime dependency of a deployed application. Core packages must not depend on `cfx-tools`.
*   **[cfx-llm](cfx-llm.md):** Centralizes LLM tooling and utilities, encapsulating model interaction and prompt management logic within `packages/llm-tools`.

### Governance
*   **[cfx-meta](cfx-meta.md):** Acts as the authoritative source for project-wide governance. It hosts Architecture Decision Records (ADRs), security policies (`SECURITY.md`), and release orchestration metadata. It does not contain shippable application code.

## Extraction Strategy

The layout is designed to support the transition from monorepo to multi-repo.
1.  **Current State:** All slices reside in `repos/` within the monorepo for shared development velocity.
2.  **Preparation:** Each slice maintains standalone configuration (`pnpm-workspace.template.yaml`) to facilitate extraction.
3.  **Future State:** Specific domains (e.g., `cfx-domain` packages) will be extracted into independent repositories once consumer count justifies the split, while `cfx-core` and `cfx-keys` remain stable foundations.
