# Conflux DevKit VS Code Extension

## Overview

The Conflux DevKit extension (`cfxdevkit`) provides an integrated development environment for building and deploying smart contracts on the Conflux blockchain directly from VS Code. It manages a local Conflux dev node, an encrypted file-based keystore, contract deployments, and direct ABI interaction — all surfaced through a sidebar tree view and command palette.

The extension supports all three Conflux environments (local devnet, testnet, mainnet) and both Conflux execution spaces: eSpace (EVM-compatible, `0x` addresses) and Core Space (Conflux-native base32 addresses).

## Architecture

<Mermaid chart={`graph TD
    A["activate / deactivate"] --> B[ExtensionRuntime]
    B --> C["helpers/*"]
    B --> D[StaticTreeProvider]
    C --> E[createSharedNodeRuntime]
    C --> F[KeystoreProvider]
    C --> G["deployContract / sendWrite / readContract"]
    D --> H["views/*"]
    H --> I[ViewSnapshot]
    C --> I

`ExtensionRuntime` is the single stateful object for the lifetime of the extension. All logic is implemented as standalone helper functions that accept `this: ExtensionRuntime` and are called via `.call(this)` from the class, keeping the class itself thin while maintaining access to all shared runtime state.

---

## Entry Points

### `activate(context)`

Called by VS Code when the extension activates. Creates the `ExtensionRuntime`, registers it for disposal, then calls `runtime.activate()` which:

1. Registers all VS Code commands via `registerCommands()`
2. Calls `refreshAll()` to build and populate the sidebar tree
3. Auto-starts the local node if `cfxdevkit.autoStartNode` is `true` and `local` is the selected network

### `deactivate()`

Gracefully awaits `stopNodeForDeactivation()` before disposing the runtime. This ensures the local node releases its ports before VS Code restarts the extension host (e.g., on reload). The stop is explicitly awaited here rather than in `dispose()` because `dispose()` is synchronous.

## ExtensionRuntime State

The class holds all mutable runtime state:

| Field | Type | Purpose |
|---|---|---|
| `node` | `SharedNodeRuntime \| null` | Handle to the running local Conflux node |
| `unlockedPassphrase` | `string \| null` | Cached keystore passphrase; cleared on lock |
| `fileProvider` | `KeystoreProvider \| null` | Open keystore provider instance |
| `cachedSigner` | `CachedSigner \| null` | Last-resolved signer, invalidated on wallet/network changes |
| `localNodeMnemonic` | `string \| null` | Mnemonic used to seed the running local node |
| `auditLogger` | — | Append-only audit log for keystore operations |
| `output` | `OutputChannel` | "Conflux DevKit" VS Code output channel |
| `mainProvider` | `StaticTreeProvider` | Data provider for the `cfxdevkit.mainView` tree |

Workspace-scoped selections (active network, space, wallet ref, account index) are persisted in `context.workspaceState` using the following keys:

```
cfxdevkit.selectedNetwork       — 'local' | 'testnet' | 'mainnet'
cfxdevkit.selectedSpace         — 'espace' | 'core'
cfxdevkit.activeMnemonicRootRef — SecretRef
cfxdevkit.activeMnemonicAccountIndex — number
```

## Helper Modules

All logic is decomposed into helper files under `src/helpers/`. Each helper is a standalone function with `this: ExtensionRuntime` and is re-exported through `src/helpers/index.ts`.

### `state.ts`
Reads and writes all persisted and derived configuration state:
- Path helpers: `keystorePath`, `deploymentsPath`, `nodeDataDir`, `auditLogPath`
- Selection accessors: `selectedNetwork`, `selectedSpace`, `selectedFileRef`, `selectedAccountIndex`
- Keystore helpers: `keystoreExists`, `fileKeystore`, `ensureFileKeystoreUnlocked`, `listFileWallets`
- `derivationPath(index)` — returns a BIP-44 path at `m/44'/60'/0'/0/{index}`
- `setSelectedNetwork` — also stops the local node when switching away from `'local'`

### `chain.ts`
Manages chain configuration and RPC clients:
- `chainsFor(network)` — returns `{ espace, core }` `ChainConfig` objects for the given network
- `currentChain(target)` — resolves the active chain config for the selected or specified target
- `createClientFor(target)` — builds a typed RPC client; for `local`, uses the running node's URL; for other networks, uses the chain's public RPC
- `walletSignerFor(target)` — resolves the active signer, checks that the local node was started from the same wallet, and attaches a base32 Core address when `target === 'core'`
- `ensureUnlockedWallet()` — unlocks the keystore if needed, selects the appropriate wallet ref, and returns a `{ provider, signer, ref, path }` result

### `node.ts`
Controls the embedded local Conflux dev node:
- `startNode()` — retrieves or creates a node mnemonic, creates `createSharedNodeRuntime(...)`, starts it, and refreshes the tree
- `stopNode()`, `restartNode()`, `wipeNode()`, `wipeNodeAndRestart()` — lifecycle management
- `mineBlocks()`, `startMining(intervalMs)`, `stopMining()` — block production control
- `getOrCreateNodeMnemonic()` — returns `localNodeMnemonic` if valid, otherwise attempts to unlock the wallet to populate it; prompts to add a wallet if none exists

The node runtime is created via `createSharedNodeRuntime` in `shared.ts`, which wraps the `@cfxdevkit/client` API. When `CFXDEVKIT_DEVNODE_SERVER_URL` is set, the client connects to an external devnode server; otherwise it embeds a local `@cfxdevkit/devnode-server` instance.

### `abi.ts`
Handles ABI function calls:
- `abiCallRead(fn, contract)` — prompts for arguments, calls `readContract`, and prints the result to the output channel
- `abiCallWrite(fn, contract)` — prompts for arguments and optional payable value, submits the transaction via `sendWrite`, then confirms: on local networks it uses `packMine()` + polling; on testnet/mainnet it polls via `waitForReceipt`
- `promptAbiArgs(fn)` — shows a single JSON-array input box with type hints from the ABI inputs
- `promptPayableValue()` — prompts for a wei/drip value for payable functions
- `parseArgValue(value, solidityType)` — converts a raw string to the appropriate JS type (`BigInt` for integers, parsed JSON for arrays, boolean coercion for `bool`)

### `artifact.ts`
Contract source selection and compilation:
- `pickTemplateArtifact()` — presents built-in templates from `listTemplates()`, compiles the selected one with `@cfxdevkit/compiler`
- `pickWorkspaceArtifact()` — finds all `.sol` files in the workspace (excluding `node_modules`, `dist`, `coverage`), prompts for a file and solc version, compiles, and lets the user pick among multiple artifacts
- `promptConstructorArgs(artifact, defaults?)` — iterates over constructor ABI inputs and prompts for each argument individually

### `contract/commands.ts`
High-level contract deployment and browsing:
- `deployContractCommand()` — orchestrates the full deploy flow: source selection → artifact compilation → chain target selection → constructor args → `deployContract` → receipt confirmation → write to `deployments.json` → tree refresh
- `showAccountsQuickPick()` / `showContractsQuickPick()` — quick-pick wrappers that copy addresses to the clipboard
- `copyAddress(address)` — writes to clipboard and shows a notification

### `contract/import.ts`
- `importContractCommand()` — collects address (manual or from environment variable), chain target, network, name, optional ABI, and optional tx hash, then appends a `DeploymentRecord` to `deployments.json`

### `wallet/index.ts`
- `initializeWallet()` — entry point that branches to `addWallet`, `selectWallet`, or `unlockKeystore` based on keystore state
- `addWallet()` — creates the keystore if absent (prompting for a new passphrase), then stores a new or imported mnemonic under a user-chosen ref with configurable account count
- `selectWallet(target?)` — updates `STATE_ACTIVE_FILE_REF` and `STATE_ACTIVE_ACCOUNT_INDEX`; if no target is provided, shows a quick-pick of all derived accounts
- `removeWallet(target?)` — removes a mnemonic entry from the keystore with confirmation

### `wallet/lock.ts`
- `lockKeystore()` — clears `unlockedPassphrase`, `fileProvider`, `cachedSigner`, and `localNodeMnemonic`
- `rotateKeystorePassphrase()` — prompts for old and new passphrases, delegates to `rotateLocalPassphrase`, and updates in-memory state
- `unlockKeystore(target?)` — calls `ensureUnlockedWallet` and shows the resolved address

### `selection.ts`
- `selectNetwork(network?)` — presents a quick-pick of the three networks if no argument is given, then calls `setSelectedNetwork`

### `view.ts`
- `refreshAll()` — builds a `ViewSnapshot`, renders it immediately (without balances), then fetches balances in the background and re-renders
- `buildSnapshot()` — aggregates all current state (deployments, wallet roots, accounts, node status) into a `ViewSnapshot`
- `populateAccountBalances(accounts)` — fires parallel `getBalance` requests against both eSpace and Core clients, mutates the account records in place

### `commands.ts`
- `registerCommands()` — registers every `cfxdevkit.*` VS Code command. All commands that can trigger async work are wrapped by the `guard(id, fn)` helper, which drops re-entrant calls and surfaces errors via `showErrorMessage`.

## Tree View

The sidebar view (`cfxdevkit.mainView`) is powered by `StaticTreeProvider<vscode.TreeItem>`, a simple `TreeDataProvider` implementation that holds a flat list of top-level items with optional `children` arrays for nested sections. Items are replaced atomically by `setItems()`, which fires `onDidChangeTreeData`.

`makeMainItems(snapshot)` in `src/views/index.ts` composes three sections:

1. **Network row** (`makeNetworkNodeRow`) — a single item showing the active network and node status with context-value variants (`cfxLocalNodeRunning`, `cfxLocalNodeStopped`, `cfxLocalNodeLocked`, `cfxNonLocalNetwork`) that control which inline toolbar buttons appear in `package.json`
2. **Wallets + Accounts** (`makeAccountItems`) — wallet roots with lock/ready state, then derived accounts with collapsible eSpace/Core address children showing balances
3. **Contracts** (`makeContractItems`) — deploy and import action items, then contracts grouped by network → space (eSpace / Core Space) → contract → address + READ/WRITE ABI function groups

Contract ABI items carry `arguments: [fn, contract]` on their `command` so that `cfxdevkit.abiCallRead` / `abiCallWrite` receive the full `AbiFunctionTreeRecord` and `ContractTreeRecord` context.

## Deployment Records

Deployments are persisted as a JSON array in `.cfxdevkit/deployments.json` (configurable via `cfxdevkit.deploymentsPath`). Each `DeploymentRecord` contains:

```ts
interface DeploymentRecord {
  id: string;          // timestamp-name composite
  name: string;        // contract name
  address: string;     // deployed address (0x or base32)
  target: 'espace' | 'core';
  network: 'local' | 'testnet' | 'mainnet';
  chainId: number;
  txHash: string;
  deployedAt: string;  // ISO timestamp
  abi?: unknown[];     // stored for in-tree ABI interaction
}
```

New deployments are prepended (most recent first). The file is created lazily on first deploy.

## Local Node

The embedded node is wrapped by `createSharedNodeRuntime` in `shared.ts`. This factory creates a `@cfxdevkit/devnode-server` app (or uses a remote server via `CFXDEVKIT_DEVNODE_SERVER_URL`) and wraps it with a `@cfxdevkit/client` HTTP client. The returned object exposes:

- `start(config)`, `stop()`, `restart()`, `wipe(restart?)` — lifecycle
- `mine(blocks)`, `packMine()` — manual block production
- `startMining(intervalMs)`, `stopMining()` — automatic mining
- `isRunning()`, `.accounts`, `.urls` — runtime state from the last API response snapshot

The `packMine()` + polling pattern used in `deployContractCommand` and `abiCallWrite` is required because Conflux deferred execution means a transaction must be both packed into a block *and* have several epochs advance before the receipt becomes available.

## Configuration

All settings live under the `cfxdevkit` namespace:

| Setting | Default | Purpose |
|---|---|---|
| `keystorePath` | `.cfxdevkit/keystore.json` | Encrypted keystore location (absolute or workspace-relative) |
| `deploymentsPath` | `.cfxdevkit/deployments.json` | Deployment registry |
| `nodeDataDir` | `.cfxdevkit/devnode` | Local node data directory |
| `nodeAccounts` | `10` | Number of pre-funded accounts to generate |
| `nodeLogging` | `false` | Enable verbose node logs |
| `autoStartNode` | `false` | Start local node on extension activation |

## Contributing

All helper functions follow the same pattern: they are plain exported functions with `this: ExtensionRuntime` as a typed `this` parameter, imported from `helpers/index.ts` and delegated from the class:

```ts
// In helpers/myfeature.ts
export async function myFeature(this: ExtensionRuntime): Promise<void> {
  this.log('doing something');
  const client = await this.createClientFor('espace');
  // ...
}

// In extension.ts
private async myFeature(): Promise<void> {
  return myFeature.call(this);
}
```

This pattern keeps the class declaration as a routing surface while allowing helpers to be developed and tested in isolation. When adding a new command, also add a `guard(id, fn)` wrapper in `registerCommands()` unless the command is intentionally re-entrant.
