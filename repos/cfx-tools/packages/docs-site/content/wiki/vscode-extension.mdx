# VS Code Extension

# VS Code Extension Module

## Overview

The Conflux DevKit VS Code Extension provides an integrated development environment for Conflux blockchain development directly within Visual Studio Code. It enables developers to manage wallets, interact with local and remote networks, deploy and import contracts, and execute contract functions through a tree-based UI.

The extension runs as a self-contained runtime that manages a local Conflux dev node, encrypted keystore for mnemonic wallets, and workspace state for network/chain selection.

## Extension Lifecycle

### Activation

The `activate()` function initializes the extension:

```typescript
export async function activate(context: vscode.ExtensionContext): Promise<void> {
  runtime = new ExtensionRuntime(context);
  context.subscriptions.push(runtime);
  await runtime.activate();
  
  // Set up status bar for signer selection
  // Set up file watcher for signer.json changes
}
```

**Key initialization steps:**
1. Creates `ExtensionRuntime` instance
2. Registers all commands via `registerCommands()`
3. Refreshes the tree view via `refreshAll()`
4. Auto-starts local node if configured and network is 'local'
5. Sets up status bar item showing active signer
6. Watches `.cfxdevkit/signer.json` for changes

### Deactivation

The `deactivate()` function cleans up resources:

```typescript
export async function deactivate(): Promise<void> {
  await runtime?.stopNodeForDeactivation();
  await stopDevnodeServer();
  runtime?.dispose();
  runtime = null;
}
```

## Core Components

### ExtensionRuntime Class

The main class that encapsulates all extension functionality:

```typescript

  private readonly auditLogger = createAppendOnlyAuditLogger(...);

  private readonly mainProvider = new StaticTreeProvider<vscode.TreeItem>();
  private readonly disposables: vscode.Disposable[] = [];
}
```

**Key responsibilities:**
- Manages local node lifecycle
- Handles keystore unlock/lock state
- Provides access to workspace configuration and state
- Wraps all helper functions as methods

### Helper Function Pattern

All helper functions are defined in separate modules and called using `.call(this)` to access the runtime context:

```typescript
// In extension.ts
private async startNode(): Promise<void> { return startNode.call(this); }

// In node.ts
export async function startNode(this: ExtensionRuntime): Promise<void> {
  // Accesses this.node, this.config(), this.log(), etc.
}
```

This pattern allows:
- Clean separation of concerns across files
- Runtime context access without circular dependencies
- Easy testing of individual functions

## Command System

### Registration

Commands are registered in `helpers/commands.ts` with a guard pattern to prevent re-entrant execution:

```typescript
const inFlight = new Set<string>();

function guard(id: string, fn: () => Promise<void> | void): void {
  if (inFlight.has(id)) return;
  inFlight.add(id);
  // ... execute fn, remove from set on completion
}

export function registerCommands(this: ExtensionRuntime): void {
  this.context.subscriptions.push(
    vscode.commands.registerCommand('cfxdevkit.selectNetwork', (network?) =>
      guard('selectNetwork', () => this.selectNetwork(network)),
    ),
    // ... more commands
  );
}
```

## Tree View Architecture

The extension uses a `StaticTreeProvider` to render a tree view in the VS Code sidebar:

```typescript
class StaticTreeProvider<T extends vscode.TreeItem> implements vscode.TreeDataProvider<T> {
  private readonly emitter = new vscode.EventEmitter<T | undefined | null>();
  readonly onDidChangeTreeData = this.emitter.event;
  
  setItems(items: readonly T[]): void {
    this.items = items;
    this.emitter.fire(undefined);
  }
}
```

The `buildSnapshot()` function creates a complete view state:

This snapshot is then rendered by view-specific functions:

- `makeNetworkNodeRow()` - Network selection and node status

### Node Lifecycle

```typescript
export async function startNode(this: ExtensionRuntime): Promise<void> {
  const mnemonic = await this.getOrCreateNodeMnemonic();
  await this.ensureUnlockedWallet();
  await ensureDevnodeServerRunning(keystorePath, nodeProfileDataRoot);
  const node = createSharedNodeRuntime({ mnemonic, dataDir, accounts, logging });
  await node.start();
  this.node = node;
}
```

**Node operations:**
- `start()` - Start the node from mnemonic
- `stop()` - Stop the node
- `restart()` - Restart the node
- `wipe()` - Delete node data
- `mine(n)` - Mine n blocks
- `packMine()` - Pack pending transactions and mine
- `startMining(intervalMs)` - Start continuous mining
- `stopMining()` - Stop continuous mining

## Contract Interaction

### Deployment Flow

```typescript
export async function deployContractCommand(this: ExtensionRuntime): Promise<void> {
  // 1. Select source (template or workspace file)
  const artifact = await this.pickTemplateArtifact() || await this.pickWorkspaceArtifact();
  
  // 2. Select chain target (eSpace or Core)
  const target = await this.pickChainTarget(`Deploy ${artifact.contractName}`);
  
  // 3. Prompt constructor arguments
  const args = await this.promptConstructorArgs(artifact);
  
  // 4. Deploy transaction
  const submitted = await deployContract({ client, signer, abi, bytecode, args });
  
  // 5. Wait for receipt (pack-mine on local, poll on remote)
  const receipt = await waitForReceipt(client, submitted.hash);
  
  // 6. Save to deployments.json
  deployments.unshift({ id, name, address, target, network, chainId, txHash, deployedAt, abi });
  await this.writeDeployments(deployments);
}
```

### ABI Function Execution

**Read functions** call `abiCallRead()`:
```typescript
export async function abiCallRead(
  this: ExtensionRuntime,
  fn: AbiFunctionTreeRecord,
  contract: ContractTreeRecord,
): Promise<void> {
  const args = await this.promptAbiArgs(fn);
  const client = await this.createClientFor(contract.target);
  const result = await readContract({ client, address: contract.address, abi, functionName: fn.name, args });
}
```

**Write functions** call `abiCallWrite()`:
```typescript
export async function abiCallWrite(
  this: ExtensionRuntime,
  fn: AbiFunctionTreeRecord,
  contract: ContractTreeRecord,
): Promise<void> {
  const args = await this.promptAbiArgs(fn);
  const value = fn.stateMutability === 'payable' ? await this.promptPayableValue() : undefined;
  const signer = await this.walletSignerFor(contract.target);
  const submitted = await sendWrite({ client, signer, address, abi, functionName: fn.name, args, value });
  // Wait for receipt with pack-mine on local
}
```

### Signer Selection

The extension supports multiple signer backends configured via `.cfxdevkit/signer.json`:

```typescript
export function selectedBackend(this: ExtensionRuntime): KeystoreBackend {
  try {
    const config = JSON.parse(readFileSync(signerJsonPath, 'utf8'));
    const entry = config?.signers?.[config?.defaultSigner];
    if (entry?.kind === 'onekey') return 'onekey';
    if (entry?.kind === 'file-keystore') return 'file';
    return 'file';
  } catch {
    return 'file';
  }
}
```

### Chain Spaces

Each network supports two chain spaces:
- **eSpace**: EVM-compatible with 0x addresses
- **Core Space**: Conflux-native with base32 addresses

```typescript
export function currentChain(
  this: ExtensionRuntime,
  target: ChainTarget = this.selectedSpace(),

  const chains = this.currentChains();
  return target === 'core' ? chains.core : chains.espace;
```

## Development Guidelines

### Adding Features

1. **Keep helpers focused** - Each helper module should have a single responsibility
2. **Use the call pattern** - All helpers should accept `this: ExtensionRuntime`
3. **Prefer async/await** - Most operations are asynchronous
4. **Handle errors gracefully** - Use `guard()` for commands
5. **Update the tree** - Call `refreshAll()` after state changes

### Testing Considerations

- Mock `vscode.window` and `vscode.workspace` for UI tests
- Test helper functions in isolation with mock runtime context
- Verify state persistence in workspaceState and files
- Test node lifecycle operations (start/stop/restart)

### Performance Notes

- Balance fetching is done in background to avoid blocking tree rendering
- Commands prevent re-entrant execution via `guard()`
- Node operations use `withProgress()` for user feedback
- View updates fire `onDidChangeTreeData` only when necessary
