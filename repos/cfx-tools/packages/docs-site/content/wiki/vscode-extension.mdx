# VS Code Extension Module

## Overview

The VS Code Extension provides a development environment for Conflux blockchain within Visual Studio Code. It integrates wallet management, local node control, contract deployment, and blockchain interaction into a unified sidebar interface.

## Extension Lifecycle

### Activation

```typescript
export async function activate(context: vscode.ExtensionContext): Promise<void>
```

On extension activation:

1. Creates `ExtensionRuntime` instance with the VS Code context
2. Registers all commands via `registerCommands()`
3. Refreshes the UI tree via `refreshAll()`
4. Auto-starts local node if `autoStartNode` config is enabled and network is 'local'

### Deactivation

```typescript
export async function deactivate(): Promise<void>
```

On extension deactivation:

1. Awaits `stopNodeForDeactivation()` to release ports cleanly
2. Calls `dispose()` on the runtime to clean up disposables
3. Nullifies the runtime reference

**Note**: `dispose()` is intentionally synchronous; node stopping is handled separately to ensure ports are released before VS Code restarts the extension host.

## Core Components

### ExtensionRuntime Class

The central state manager for the extension. Maintains:

| Property | Purpose |
|----------|---------|
| `node` | Local dev node runtime instance |
| `unlockedPassphrase` | Cached keystore passphrase |
| `fileProvider` | Active keystore provider |
| `cachedSigner` | Cached wallet signer |
| `localNodeMnemonic` | Mnemonic used for local node |
| `auditLogger` | Append-only audit log writer |
| `output` | VS Code output channel |
| `mainProvider` | Tree data provider for sidebar |

### Command Registration

Commands are registered in `registerCommands()` with re-entrancy protection:

```typescript
const inFlight = new Set<string>();

function guard(id: string, fn: () => Promise<void> | void): void {
  if (inFlight.has(id)) return;
  // ... executes fn, handles errors, deletes from set
}
```

Key commands include:
- `cfxdevkit.selectNetwork` - Switch between local/testnet/mainnet
- `cfxdevkit.deployContract` - Deploy from template or workspace file
- `cfxdevkit.importContract` - Register existing deployed contracts
- `cfxdevkit.nodeStart/Stop/Restart/Wipe` - Node lifecycle management
- `cfxdevkit.addWallet/selectWallet/removeWallet` - Wallet operations
- `cfxdevkit.abiCallRead/abiCallWrite` - Contract function calls

## Key Features

### Network Selection

Three networks are supported:

| Network | Core ChainId | eSpace ChainId |
|---------|--------------|----------------|
| Local | 2029 | 2030 |
| Testnet | 1 | 71 |
| Mainnet | 1029 | 1030 |

Selection is persisted via `workspaceState` using `STATE_NETWORK` key.

### Chain Space (eSpace vs Core)

Conflux has two address spaces:
- **eSpace**: EVM-compatible, `0x` addresses
- **Core Space**: Conflux-native, base32 addresses

Switching is handled via `pickChainTarget()` and persisted with `STATE_SPACE`.

### Local Node Management

The extension embeds a Conflux dev node runtime through `createSharedNodeRuntime()`:

```typescript
const node = createSharedNodeRuntime({
  mnemonic,
  dataDir: this.nodeDataDir(),
  accounts: this.config().get<number>('nodeAccounts', 10),
  logging: this.config().get<boolean>('nodeLogging', false),
});
```

Node operations:
- `startNode()` - Starts node with mnemonic from unlocked wallet
- `stopNode()` - Gracefully stops the node
- `restartNode()` - Restarts without data wipe
- `wipeNode()` - Clears node data directory
- `mineBlocks()` - Manually mine specified block count
- `startMining()` / `stopMining()` - Continuous mining

### Wallet Management

File-based keystore stores encrypted mnemonic wallets:

```typescript
const provider = createFileKeystore({
  path: this.keystorePath(),
  unlock: async () => ({ passphrase }),
  audit: this.auditLogger,
});
```

Key operations:
- `addWallet()` - Generate or import mnemonic wallet
- `selectWallet()` - Choose active mnemonic and derived account
- `removeWallet()` - Delete mnemonic from keystore
- `unlockKeystore()` - Provide passphrase to access wallets
- `lockKeystore()` - Clear cached passphrase and signer
- `rotateKeystorePassphrase()` - Change keystore password

### Contract Deployment

Deployment flow:

```typescript
async function deployContractCommand(): Promise<void> {
  // 1. Select source (template or workspace file)
  const artifact = await this.pickTemplateArtifact() 
                || await this.pickWorkspaceArtifact();
  
  // 2. Select target chain
  const target = await this.pickChainTarget(`Deploy ${artifact.contractName}`);
  
  // 3. Prompt constructor arguments
  const args = await this.promptConstructorArgs(artifact);
  
  // 4. Get signer and deploy
  const signer = await this.walletSignerFor(target);
  const submitted = await deployContract({ client, signer, abi, bytecode, args });
  
  // 5. Wait for receipt (local: packMine + mine; remote: poll)
  const receipt = await waitForReceipt(client, submitted.hash);
  
  // 6. Save deployment record
  await this.writeDeployments([...deployments, newRecord]);
  await this.refreshAll();
}
```

### Contract Interaction

Read/write calls are triggered from the contracts tree view:

**Read Calls** (`abiCallRead`):
- Prompts for function arguments via JSON input box
- Calls `readContract()` without signer
- Displays result in output channel

**Write Calls** (`abiCallWrite`):
- Prompts for function arguments
- Prompts for payable value if `stateMutability === 'payable'`
- Submits transaction via `sendWrite()`
- Confirms receipt (local: packMine + poll; remote: `waitForReceipt()`)

## Data Storage

### Configuration Files

| File | Path | Purpose |
|------|------|---------|
| Keystore | `.cfxdevkit/keystore.json` | Encrypted wallet storage |
| Deployments | `.cfxdevkit/deployments.json` | Contract deployment records |
| Node Data | `.cfxdevkit/devnode/` | Local node data directory |
| Audit Log | `.cfxdevkit/audit.log` | Append-only audit events |

Paths are configurable via `cfxdevkit.keystorePath`, `cfxdevkit.deploymentsPath`, and `cfxdevkit.nodeDataDir` settings.

### Workspace State

Persistent settings stored per-workspace:

```typescript
STATE_NETWORK = 'cfxdevkit.selectedNetwork'
STATE_SPACE = 'cfxdevkit.selectedSpace'
STATE_ACTIVE_FILE_REF = 'cfxdevkit.activeMnemonicRootRef'
STATE_ACTIVE_ACCOUNT_INDEX = 'cfxdevkit.activeMnemonicAccountIndex'
```

The sidebar uses a `StaticTreeProvider` to render a hierarchical view:

```

│   └── Add wallet
├── Accounts
│   ├── Account #0 (active)
│   │   ├── eSpace: 0x... (balance)
│   │   └── Core: cfx:... (balance)
│   └── Unlock wallet
└── Contracts
    ├── Deploy contract
    ├── Import contract
    └── [Network]

        └── Core Space
            └── ...
```

## Integration Points

### External Dependencies

- `@cfxdevkit/client` - Conflux RPC client
- `@cfxdevkit/compiler` - Solidity compilation
- `@cfxdevkit/contracts` - Deploy/read/write operations
- `@cfxdevkit/services` - Keystore and audit services
- `@cfxdevkit/wallet` - Derivation and signing

### VS Code APIs

- `vscode.window.createTreeView` - Sidebar tree
- `vscode.commands.registerCommand` - Command registration
- `vscode.window.showQuickPick` - Dropdown selections
- `vscode.window.showInputBox` - Text input
- `vscode.window.withProgress` - Progress notifications
- `vscode.workspace.workspaceFolders` - Workspace detection

## Error Handling

Commands use the `guard()` wrapper to prevent re-entrant execution and display errors:

```typescript
guard('deployContract', () => this.deployContractCommand())
```

Errors are shown via `vscode.window.showErrorMessage()`. Node operations catch and log errors internally to prevent deactivation failures.

## Contributing Guidelines

1. **Helper Functions**: Keep helper modules focused on single responsibilities. Import shared utilities from `shared.ts`.

2. **State Management**: Use `workspaceState` for persistence, not `globalState`. Clear `cachedSigner` when network or wallet changes.

3. **UI Updates**: Always call `refreshAll()` after state changes. Balance fetching is deferred to avoid blocking tree rendering.

4. **Node Operations**: Ensure node is stopped before deactivation. Use `wipeNode()` before `startNode()` when data corruption is suspected.

5. **Command Guards**: Wrap async commands with `guard()` to prevent concurrent execution of the same command.
