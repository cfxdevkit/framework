# repos — cfx-domain

# `cfx-domain` Module

## Overview

The `cfx-domain` module houses vertical implementation packages for the Conflux ecosystem, targeting game-engine logic and on-chain automation. The primary active package is `@cfxdevkit/automation`, which implements a resilient, strategy-driven job execution system for on-chain trading operations (limit orders, DCA, TWAP, and direct swaps). `@cfxdevkit/game-engine` is currently a structural stub awaiting future implementation.

The automation system manages the full lifecycle of execution jobs: creation, state tracking, condition evaluation, on-chain transaction submission, retry handling, and audit logging. It is designed to run as a long-lived worker process that polls for active jobs, enforces safety constraints, and interfaces directly with the `AutomationManager` smart contract and Swappi liquidity router.

## Package Structure

| Package | Description |
|---------|-------------|
| `@cfxdevkit/automation` | Core automation runtime. Contains job evaluators, price sources, safety guards, retry queues, persistence layers, and on-chain clients. |
| `@cfxdevkit/game-engine` | Stub package for future game-state logic. Exports `__packageName` only. |
| `scripts/` | Monorepo tooling: `validate-workspace.mjs` ensures directory alignment with `pnpm-workspace.template.yaml`, and `check-deps.mjs` validates cross-package workspace dependencies. |

## Architecture & Execution Flow

The `Keeper` class is the central orchestrator. It runs on a configurable poll interval, merges active jobs from the repository with jobs due from the `RetryQueue`, and processes them concurrently using the `@cfxdevkit/executor` package. Each tick follows a deterministic pipeline:

    A[Keeper.runAllTicks] --> B[list active jobs]

    E --> G[evaluate job]
    G --> H[PriceChecker]
    G --> I[SafetyGuard]
    H & I --> J{shouldExecute?}
    J -->|true| K[keeperClient.execute]
    J -->|false| L["skip / update state"]
    K --> M[recordExecution]
    K --> N[handleExecutionError]
    N --> O[retryQueue.enqueue]
    M --> P[update job status]
    L & P & O --> Q[heartbeat callback]
    Q --> A
`} />

Jobs transition through states (`pending` → `active` → `executed` | `failed` | `cancelled` | `expired`). The `Keeper` delegates strategy-specific logic to `StrategyEvaluator` implementations, which rely on `PriceChecker` for market data and `SafetyGuard` for risk enforcement.

## Core Components

### Job Evaluation & Strategy Pattern
Strategy execution follows a uniform contract defined by `StrategyEvaluator<TJob>`. Each job type has a dedicated evaluator that:
1. Checks execution eligibility (time windows, tranches/swaps completed, price thresholds).
2. Fetches current pricing via `PriceChecker`.
3. Validates constraints via `SafetyGuard`.
4. Returns `EvalResult` with `shouldExecute`, optional `reason`, and `estimatedSwapUsd`.

| Evaluator | Job Type | Key Conditions |
|-----------|----------|----------------|
| `LimitOrderEvaluator` | `LimitOrderJob` | `direction === 'gte'` or `'lte'` vs `targetPrice` |
| `DCAEvaluator` | `DCAJob` | `isDCADue()` checks `nextExecution` + buffer, `swapsCompleted < totalSwaps` |
| `TWAPEvaluator` | `TWAPJob` | `tranchesCompleted < trancheCount`, `nextExecution` window |
| `SwapEvaluator` | `SwapJob` | One-shot execution, safety checks only |

### Price Checking & Sources
`PriceChecker` abstracts price retrieval and provides helpers like `estimateUsdValue`. It wraps any `PriceSource` implementation:
- **`SwappiPriceSource`**: Queries a Swappi router via `getAmountsOut(unitAmount, [tokenIn, tokenOut])`. Returns normalized 18-decimal scaled prices.
- **`GeckoTerminalPriceSource`**: Fetches USD prices from the GeckoTerminal API, computes the ratio, and scales to 18 decimals.

All prices are represented as `bigint` scaled by `PRICE_SCALE = 10n ** 18n`.

### Safety Guard & Constraints
`SafetyGuard` enforces risk limits before execution. It evaluates jobs against `SafetyConfig` (defaults: `maxSwapUsd: 10_000`, `maxSlippageBps: 500`, `maxRetries: 5`, `minExecutionIntervalSeconds: 30`). Violations include:
- `global_pause`: Runtime pause override
- `status`: Job not in `active` state
- `max_retries`: Retry cap exceeded
- `expiry`: Job past `expiresAt`
- `slippage`: Job slippage exceeds configured limit
- `swap_size`: Estimated USD value exceeds cap
- `min_interval` / `dca_interval`: DCA timing violations

The guard is mutable at runtime (`updateConfig`, `pauseAll`, `resumeAll`) and tracks violation history via `getViolations()`.

### Retry Queue & Resilience
`RetryQueue` handles transient execution failures. It:
- Replaces existing entries for the same `jobId` to prevent duplicates.
- Applies exponential backoff: `baseDelayMs * 2^attempt`, capped at `maxDelayMs`.
- Applies configurable jitter (`0.0–0.5` range) to prevent thundering herd effects.
- Returns cloned `Job` objects via `drainDue()` to prevent external mutation of queue state.

When `Keeper.#handleExecutionError` detects a failure, it increments the retry counter, appends the error message, enqueues the job, and returns a `skipped` tick result.

### Persistence & Data Access
The automation layer abstracts storage behind `JobRepository` and `ExecutionRepository` interfaces.

**SQLite Implementation (`DrizzleJobRepository` / `DrizzleExecutionRepository`)**
- Schema defined in `src/db/schema.ts` (`jobs`, `executions`, `worker_heartbeat`, `settings`).
- `bigint` values are serialized to strings before DB insertion via `serializeParams()` and deserialized back via `rowToJob()` and `rowToRecord()`.
- `DrizzleJobRepository` supports filtered listing (`JobListFilter`), atomic state transitions (`markActive`, `markExecuted`, `markFailed`, `incrementRetry`, `cancel`), and `getUpdatedSince()` for worker synchronization.
- Owner validation is enforced on `cancel()` (case-insensitive comparison).

**Memory Implementation (`MemoryJobRepository` / `MemoryExecutionRepository`)**
- In-memory `Map` and array storage for testing.
- Deep-clones job params on read to prevent test cross-contamination.

### On-Chain Execution (`KeeperClient`)
`KeeperClient` defines the contract for submitting jobs to the `AutomationManager`. `KeeperClientImpl` provides the concrete implementation:
- **Pre-flight checks**: Verifies contract is not `paused()` and gas price is within `maxGasPriceGwei`.
- **Transaction construction**: Builds `SignableTx` using `@cfxdevkit/cdk`'s `EspaceClient` and `Signer`. Calculates `maxFeePerGas` dynamically based on `baseFeePerGas`.
- **Receipt polling**: Delegates to `@cfxdevkit/protocol`'s `waitForTransactionReceipt`.
- **Result decoding**: Extracts `amountOut` from `Transfer` events targeting the job owner.

`AutomationManagerClient` wraps the same contract but delegates actual execution logic to injected `AutomationManagerActions`, enabling mock or custom backend implementations.

### Swap Execution (`SwapExecutor`)
Used directly for `SwapJob` types or as a fallback when off-chain automation isn't required. Interacts with the Swappi router via `getAmountsOut` for quotes and `swapExactTokensForTokens` for execution. Calldata is encoded using `buildSwapCalldata()` and the Swappi router ABI. Decodes output amounts by scanning transaction logs for `Transfer` events matching the recipient address.

## Integration & Dependencies

The module integrates with the broader Conflux toolkit:
- **`@cfxdevkit/cdk`**: Provides `EspaceClient`, `Signer`, and transaction types for on-chain interactions.
- **`@cfxdevkit/executor`**: Supplies `createPoller`, `executeBatch`, and `withLock` for concurrent job processing.
- **`@cfxdevkit/protocol`**: Provides `waitForTransactionReceipt` and transaction verification utilities.
- **`drizzle-orm` / `better-sqlite3`**: Power the persistence layer. Optional peer dependencies for `mysql2`/`postgres` are reserved for future multi-DB support.

External consumers (e.g., backend workers) typically instantiate `createSqliteDb()`, wire up `DrizzleJobRepository`/`DrizzleExecutionRepository`, configure `KeeperClientImpl`, and pass them to `Keeper`. The `Keeper` is then started via `.start()`, which initializes the internal `createPoller`.

## Testing & Tooling

- **Test Framework**: `vitest` (v4.x) with Node environment. Configured in `vitest.config.ts` with alias resolution for workspace packages.
- **Test Helpers**: `src/testHelpers.ts` provides factory functions (`limitOrderJob()`, `dcaJob()`, `twapJob()`, `swapJob()`) that generate fully typed `Job` objects with default owners, tokens, and parameters.
- **Coverage**: All core modules include `.test.ts` files covering evaluation logic, repository CRUD, retry backoff, safety violations, and client transaction encoding.
- **Build**: `vite` + `vite-plugin-dts` for ES module bundling and TypeScript declaration generation. Outputs separate `dist/index.js` and `dist/db/index.js` entry points.
