# repos — cfx-meta

# repos — cfx-meta Module

## Overview

The `cfx-meta` module acts as the central authority for monorepo architecture rules, tier definitions, and meta-tooling. It enforces structural constraints across the codebase by defining strict dependency boundaries, publishing typed access to these rules, and providing executable scripts to validate workspace dependencies and directory structures.

This module ensures that all packages adhere to a unidirectional dependency model, prevents cross-cutting tooling from leaking into runtime dependencies, and maintains consistency across `package.json` declarations and pnpm workspace configurations.

## Monorepo Architecture Tiers

The module defines a strict vertical tiering system via `arch-rules.yaml`. Packages are classified into tiers based on their filesystem paths, and cross-tier imports are strictly governed by the `allowedFromTiers` and `versioningPolicy` fields.

| Tier ID | Level | Versioning Policy | Allowed Import Sources | Description |
| :--- | :--- | :--- | :--- | :--- |
| `cross-cutting` | `-1` | N/A | N/A (dev-only) | Build/tooling packages (e.g., `cfx-meta`, `cfx-config`). Must never appear in `dependencies`. |
| `framework` | `0` | `semver` | `platform`, `domains`, `projects` | Reusable libraries published to npm. Must use explicit semver and named exports. |
| `platform` | `1` | `workspace` | `domains`, `projects` | Developer platform packages. May use `workspace:*` for framework deps during pre-release. |
| `domains` | `2` | `workspace` | `projects` | Reusable domain logic. Must not depend on `projects`. |
| `projects` | `3` | `internal` | N/A | Application code. Never published. Can import any lower tier. |

## Key Components

### Typed Architecture Rules (`packages/arch-rules`)

The `@cfxdevkit/arch-rules` package provides a type-safe API around the raw architecture rules definition. Built with Vite and Vite-plugin-dts, it exposes the parsed schema and utility functions for querying the tier structure.

**Core Exports (`src/index.ts`)**
*   `getTierFor(path: string)`: Resolves a filesystem path to its corresponding tier definition (`ResolvedTier` or `null`).
*   `getRulesFor(tierId: string)`: Returns an array of `ArchRule` objects applicable to a specific tier.
*   `isImportAllowed(fromTierId: string, toTierId: string)`: Validates whether a runtime import between two tiers is permitted based on the tier graph.
*   `getLifecycle()` / `getVersioningPolicy(tierId: string)`: Utility accessors for current schema metadata.

**Type Definitions (`src/types.ts`)**
*   `ArchRulesSchema`: The root interface matching the JSON schema.
*   `TierDef`: Defines tier constraints, paths, and versioning policies.
*   `ArchRule`: Represents individual architectural constraints (e.g., `no-upward-imports`, `no-default-exports`, `file-size-hard-limit`).

### Validation Scripts (`scripts/`)

Two Node.js ES modules provide runtime validation for the workspace:

*   `check-deps.mjs`: Traverses `packages/`, reads `package.json` files, and prints a cross-package runtime dependency graph. It specifically flags `workspace:*` dependencies in `dependencies` that reference packages outside the local monorepo, issuing warnings for cross-repo references.
*   `validate-workspace.mjs`: Parses `pnpm-workspace.template.yaml` to extract glob patterns (currently supports trailing `/*`), and verifies that the expected directories exist and contain a `package.json`. Fails if the pattern base directory is missing.

## How It Works

### Rule Parsing and Schema Validation

When `@cfxdevkit/arch-rules` is imported, it loads `arch-rules.yaml` as a raw string and parses it as JSON. The schema undergoes immediate validation via `validateArchRules`:

1.  **Schema Integrity**: Verifies `version === 1` and checks that `lifecycle` is either `'pre-release'` or `'release'`.
2.  **Cross-Cutting Constraints**: Ensures the `cross-cutting` tier exists, has `level: -1`, `crossCutting: true`, and strictly enforces `usageConstraint: 'dev-only'`.
3.  **Tier Graph Validation**: Checks for duplicate tier IDs, validates `versioningPolicy` against a whitelist (`semver`, `workspace`, `internal`), and verifies that all entries in `allowedFromTiers` reference valid, existing tier IDs.
4.  **Rule Deduplication**: Ensures all `ArchRule` IDs are unique.

If any validation step fails, a descriptive `Error` is thrown, preventing the module from initializing with an invalid architecture definition.

### Path Matching Logic

Tier resolution relies on a lightweight pattern matcher in `matchesPattern` and `matchesAnyPattern`. Paths are normalized via `normalizePath` (converting backslashes to forward slashes and stripping leading `./` or `//`).

*   `**` Trailing globs (e.g., `repos/cfx-core/packages/**`) match the prefix directory and all descendants.
*   `/*` Direct child globs match only the immediate directory level.
*   Exact matches fallback to strict string equality or direct prefix matching.

This allows the tier system to dynamically classify any file path into a specific tier based on the `paths` array defined in `arch-rules.yaml`.

### Dependency and Workspace Validation

The validation scripts operate independently of the TypeScript package, providing CI/CD and pre-release checks:

1.  **Dependency Graphing (`check-deps.mjs`)**: Collects all workspace package names. For each package, it extracts `dependencies` and `devDependencies`/`peerDependencies`. It filters for `workspace:*` prefixes and checks membership against the known package set. External `workspace:*` references in `dependencies` trigger warnings, as they indicate a reliance on packages outside the monorepo that should ideally use published semver ranges.
2.  **Workspace Layout (`validate-workspace.mjs`)**: Reads the template file line-by-line, extracting quoted glob patterns. It resolves the base directory for each pattern and checks the filesystem. Directories lacking a `package.json` trigger warnings (indicating they are non-package directories), while missing base directories cause a hard failure.

## Usage and Integration

### Running Validation Scripts

Developers and CI pipelines execute workspace checks via the npm scripts defined in `repos/cfx-meta/package.json`:

```bash
# Validates that pnpm-workspace.template.yaml matches the actual packages/ structure
pnpm validate:workspace

# Prints the dependency graph and checks workspace:* consistency
pnpm validate:deps
```

### Consuming Architecture Rules in Code

Packages can query the tier system to make runtime decisions or enforce local constraints:

```typescript
import { getTierFor, getRulesFor, isImportAllowed } from '@cfxdevkit/arch-rules';

// Determine which tier a file belongs to
const tier = getTierFor('./repos/cfx-core/packages/cdk/src/index.ts');
if (tier?.id === 'framework') {
  // Apply framework-specific checks
}

// Check if an import is architecturally valid
const allowed = isImportAllowed('projects', 'framework'); // true
const crossCuttingAllowed = isImportAllowed('framework', 'cross-cutting'); // false (dev-only)

// Retrieve applicable rules for a tier
const frameworkRules = getRulesFor('framework');
// e.g., [{ id: 'no-upward-imports', severity: 'error', ... }, ...]
```

## Contributing to `cfx-meta`

When modifying the architecture or adding new packages:

1.  **Updating Tiers**: Edit `repos/cfx-meta/arch-rules.yaml`. Update the `paths` array for the relevant tier and ensure `allowedFromTiers` accurately reflects the new dependency boundaries. Run `pnpm validate:deps` to verify consistency.
2.  **Adding Rules**: Add a new object to the `rules` array in `arch-rules.yaml`. Define the `id`, `enforce` policy (`always` or `on-release`), `severity` (`error` or `warning`), and `scope` (e.g., `['framework']` or `'all'`). The schema validation will catch missing required fields like `description` or invalid `enforce` values.
3.  **Type Updates**: The `src/types.ts` interfaces in `packages/arch-rules` should be updated to reflect new fields or constraints introduced in `arch-rules.yaml`. Rebuild the package using `pnpm build` (Vite) to regenerate the declaration files.
4.  **Validation Scripts**: If workspace layout rules change, update `scripts/validate-workspace.mjs` to handle new pattern structures. Note that the script currently only supports trailing `/*` globs.
