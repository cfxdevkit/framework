# Infrastructure & Security

# Infrastructure & Security Module

## Overview

The `infrastructure/` module manages cross-cutting operational concerns including deployment, CI/CD, and observability. It contains provisioning scripts, security policies, and secret management templates.

**Key Principles:**
- **Separation of Concerns:** Project code does **not** import from this module.
- **No Secrets in Repo:** This folder contains policies and templates only. Actual secrets are stored in a managed secret manager (KMS, Vault, or OIDC).
- **Immutable Infrastructure:** VPS provisioning is handled via Ansible.

## VPS Provisioning

The shared Hetzner VPS hosts Dockerized runtime services behind Caddy. The primary service is the docs site (`www.cfxdevkit.org`). Provisioning is automated using Ansible.

### Prerequisites
1. Install Ansible Galaxy collections:
   ```sh
   cd infrastructure/ansible
   ansible-galaxy collection install -r requirements.yml
   ```
2. Configure inventory and variables locally:
   ```sh
   cp inventory.ini inventory.local.ini
   cp vars/all.yml vars/local.yml
   ```
3. Execute the playbook:
   ```sh
   ansible-playbook playbook.yml -i inventory.local.ini -e @vars/local.yml
   ```

### Roles
The `playbook.yml` orchestrates the following roles:
- `base`: System initialization.
- `docker`: Container runtime setup.
- `caddy`: Reverse proxy and TLS management.
- `docs`: Documentation site deployment.
- `monitoring`: Observability stack.
- `backups`: Automated backup routines.

## Secrets Management

The `infrastructure/secrets/` directory defines how secrets are handled. **Never commit raw secrets**, including plaintext keys, tokens, or credentials.

### Storage & Access
- **Production:** Secrets must reside in a managed store (AWS KMS, GCP KMS, HashiCorp Vault, or GitHub Actions OIDC).
- **CI/CD:** Uses OIDC or short-lived tokens. Long-lived private keys are prohibited in repository secrets.
- **Development:** Local keystores are encrypted and treated as personal state.

### Environment Templates
Use `secrets/env.template` to define required environment variables. Do not populate real values.

```ini
CFXDEVKIT_ENV=development
CFXDEVKIT_AUDIT_SINK=file://.cfxdevkit/audit.log
CFXDEVKIT_KMS_PROVIDER=
CFXDEVKIT_KMS_KEY_ID=
CFXDEVKIT_VAULT_ADDR=
CFXDEVKIT_VAULT_MOUNT=
```

### GitHub Actions Integration
CI workflows (located in `.github/workflows/`) consume specific secrets and variables.

| Name | Type | Used By | Purpose |
| :--- | :--- | :--- | :--- |
| `VPS_HOST` | Secret | `deploy-docs.yml` | Hetzner VPS public IP or hostname |
| `VPS_SSH_KEY` | Secret | `deploy-docs.yml` | Private key for VPS `deploy` user |
| `GHCR_TOKEN` | Secret | `deploy-docs.yml` | PAT with `read:packages` (if private) |
| `GHCR_USERNAME` | Variable | `deploy-docs.yml` | GHCR username (default: `cfxdevkit`) |
| npm Trusted Publishing | Config | `release.yml` | OIDC publishing for public packages |

## Security Policies

All production operations must adhere to the policies defined in `secrets/policies/`.

### Access Control
- **Least Privilege:** Grant access by deployment, environment, and operation type.
- **Identity:** Production access requires named identities; shared accounts are prohibited.
- **Reviews:**
  - Production signing/deployment secrets: **Monthly**
  - CI publish credentials: **Monthly**
  - Non-production service tokens: **Quarterly**

### Secret Rotation
| Secret Type | Cadence | Trigger |
| :--- | :--- | :--- |
| Production signing roots | Annual review | Exposure, departure, custody change |
| Session-signing credentials | ≤ 24 hours | Scope change, suspicious activity |
| CI publish tokens | 90 days (if not OIDC) | Workflow compromise, departure |
| RPC/API tokens | 90 days | Provider incident, quota abuse |

**Rotation Procedure:**
1. Create replacement secret in managed backend.
2. Deploy consumers to read replacement (keep old value active).
3. Verify health checks and audit events.
4. Revoke old secret.
5. Record rotation details in deployment runbook.

### Audit & Backup
- **Retention:** Production audit events (keystore reads/writes) must be retained for **1 year**. Access review records for **2 years**.
- **Integrity:** Audit logs must be append-only and tamper-evident.
- **Backup:** KMS/HSM root keys use provider-native backup. Plaintext export of production private keys is prohibited.
- **Recovery:** Requires at least two authorized maintainers or provider-enforced quorum.

## Compliance Checklist

Before merging changes to `infrastructure/`:
- [ ] No plaintext secrets or keys are present.
- [ ] `secrets/env.template` is updated if new env vars are required.
- [ ] Ansible roles follow the `roles/` directory structure.
- [ ] CI secrets referenced in workflows match `secrets/README.md`.
- [ ] Rotation and access policies are reviewed if permissions change.
