# LLM Completion Engine

The **LLM Completion Engine** (`workers/completion`) provides the core infrastructure for interacting with Large Language Models within the Conflux DevKit monorepo. It abstracts provider connectivity, manages configuration layers, constructs execution contexts (prompts), and standardizes completion reporting.

This module is the primary dependency for agent workflows, commit automation, documentation generation, and code review tasks.

## Architecture Overview

The engine follows a layered architecture: Configuration → Provider Resolution → Context Assembly → Completion → Reporting.

## Core Entry Points

The module exports three primary functions for initiating completions:

### `complete`
The general-purpose completion function. It resolves configuration, selects a provider, builds the system/user message pair, and returns a `CompletionReport`.

```typescript
export async function complete({
  action,
  modelOverride,
  userPrompt,
  context,
  quick = false
}): Promise<CompletionReport>
```

*   **action**: Identifier for the task (e.g., `review`, `commit`).
*   **modelOverride**: Optional model ID to bypass config defaults.
*   **quick**: If true, reduces token budgets and context size (see `tokenBudget.quick`).
*   **Returns**: `CompletionReport` containing `content`, `model`, `baseUrl`, and `generatedAt`.

### `completeCommitAgent` & `completeStructuredAgent`
Specialized wrappers for specific agent workflows. They delegate to `completeDirect` but enforce specific prompt structures and token budgets suitable for structured output (JSON) or commit message generation.

```typescript
export async function completeCommitAgent(request: CompleteAgentRequest)
export async function completeStructuredAgent(request: CompleteAgentRequest)
```

## Configuration Management

Configuration is layered and normalized to support global defaults, environment overrides, and scoped unit configurations.

### `readConfig`
Loads and merges configuration from disk and environment variables.

*   **File Resolution**: Checks `CFXDEVKIT_LLM_CONFIG_PATH` or `CFXDEVKIT_LLM_AGENT_CONFIG_PATH`. Falls back to `.pi/providers.json`, `artifacts/llm/config/llm.json`, or `artifacts/llm/config/lemonade.json`.
*   **Normalization**: Calls `normalizeConfig` to ensure all fields match `LlmConfig` shape, applying `defaultConfig()` values for missing keys.
*   **Scoped Overlays**: If the config path indicates a unit overlay (`/artifacts/llm/config/units/`), `mergeConfigLayers` is used to blend base and scoped settings.

### `normalizeConfig`
Validates and sanitizes raw configuration objects.

*   **Provider Validation**: Ensures `provider` is one of `lemonade`, `litellm`, `openai-compat`, or `github-models`.
*   **Token Budget**: Merges `tokenBudget` settings (context fraction, caps, quick mode limits).
*   **Harness Settings**: Validates `harness` mode (`deterministic` vs `exploratory`) and provider strategy.

### `resolveRequestTimeoutMs`
Determines the request timeout in milliseconds. Priority order:
1.  Config `requestTimeoutMs`
2.  Env `LLM_REQUEST_TIMEOUT_MS`
3.  Env `LEMONADE_REQUEST_TIMEOUT_MS`
4.  Default `600000` (10 minutes)

## Provider Abstraction

The engine supports multiple LLM backends via a unified `LlmProvider` interface.

### `resolveProvider`
Discovers and instantiates the correct provider implementation based on configuration and environment.

**Resolution Priority:**
1.  **Configured Provider**: If `config.provider` is set (e.g., `github-models`), uses that.
2.  **Base URL**: If `config.baseUrl` is set, selects `LiteLLMProvider` or `OpenAICompatProvider` based on type, otherwise `LemonadeProvider`.
3.  **Environment Fallbacks**:
    *   `LITELLM_BASE_URL` → `LiteLLMProvider`
    *   `LEMONADE_URL` / `LEMONADE_BASE_URL` → `LemonadeProvider`
    *   `OPENAI_BASE_URL` + `OPENAI_API_KEY` → `OpenAICompatProvider`
    *   `GITHUB_TOKEN` → `GitHubModelsProvider`
4.  **Local Discovery**: Attempts to connect to local Lemonade instances (`localhost:13305`, `host.docker.internal:13305`, etc.) using `discoverModels()`.

### Supported Providers
*   `LemonadeProvider`: Local or internal inference server.
*   `LiteLLMProvider`: Proxy for various LLM backends.
*   `OpenAICompatProvider`: Standard OpenAI API compatibility.
*   `GitHubModelsProvider`: GitHub Models API integration.

## Context Assembly

Prompts are constructed by combining static documentation, repository state, and dynamic command outputs.

### `buildActionContext`
Assembles the user prompt context based on an action specification (`spec`).

*   **Static Files**: Reads files listed in `spec.context` (e.g., `README.md`, `ARCHITECTURE.md`).
*   **Git State**: Appends `changedFilesBlock()` and `gitDiffBlock()` if requested (`includeChangedFiles`, `includeGitDiff`).
*   **Preflight Checks**: Runs `commitPreflightBlock()` to capture `gitnexus`, `moon`, and `arch-check` outputs.
*   **Truncation**: Limits total length to `12000` chars (quick mode) or `60000` chars (standard).

### `commitPreflightBlock`
Executes a series of diagnostic commands to inform the LLM about repository health.

*   **Commands**: `gitnexus ensure`, `git status`, `git diff`, `arch-check` (hotspots, kebab groups), `moon changed-files`.
*   **Output**: Formatted as command blocks with exit codes and truncated output (max 30k chars per command).

## JSON Parsing & Repair

LLM outputs often require parsing for structured data (e.g., JSON plans). The `json.ts` module provides robust extraction and repair.

### `parseJsonObject`
Extracts and parses JSON from LLM response text.

1.  **Fenced Extraction**: Looks for ```json ... ``` blocks.
2.  **Candidate Extraction**: If no fence, finds balanced `{ ... }` blocks.
3.  **Repair Attempts**:
    *   Escapes raw newlines within strings.
    *   Strips trailing commas before `}` or `]`.
    *   Attempts parsing up to 3 variations.
4.  **Error**: Throws if no valid JSON object is found.

## Actions & Monorepo Units

The engine defines predefined tasks and scoping units to standardize agent behavior.

### `repoActions`
A map of predefined actions (e.g., `review`, `commit`, `docs-api`) defined in `shared/repo-actions.ts`. Each action includes:
*   **mode**: `deterministic` (strict) or `exploratory` (flexible).
*   **defaultPrompt**: System instruction template.
*   **context**: Required files to include.
*   **flags**: Options like `includeChangedFiles`.

### `monorepoUnitSpecs`
Defines logical scopes within the monorepo (`delivery`, `implementation`, `operations`).
*   **`findMonorepoUnitByConfigPath`**: Maps a config file path to a specific unit to determine active context and default modes.

## Reporting & Execution Context

### `CompletionReport`
The standardized return type for all completion functions.

```typescript
interface CompletionReport {
  generatedAt: string;
  action: string;
  baseUrl: string;
  model: string;
  content: string;
  attempts: readonly CompletionAttempt[];
}
```

### `resolveExecutionContext`
(Found in `shared/execution-context.ts`)
High-level summary generator for agents. It resolves the current `MonorepoUnit`, checks LLM availability, and logs the execution state via `logExecutionContext`.

## Error Handling

*   **Config Loading**: `readConfig` handles `ENOENT` gracefully, falling back to defaults if config files are missing.
*   **Provider Resolution**: Throws if no provider can be resolved from config or environment.
*   **Stream Parsing**: `applyChatCompletionChunk` ignores malformed chunks to prevent stream interruption.
*   **Progress Reporting**: `emitProgress` in `provider-stream-parse.ts` wraps `onProgress` callbacks in try/catch to ensure completion flow is not broken by logging errors.

## Integration

Other modules interact with this engine via `workers/completion/index.ts`:

*   **Agents**: `runAgentCheck`, `runReviewAgent` call `complete` or `completeStructuredAgent`.
*   **Commit Workflow**: `runCommitWorkflow` uses `completeCommitAgent` and `commitPreflightBlock`.
*   **Docs**: `runDocsApi`, `runWikiGenerate` use `buildActionContext` for documentation updates.
*   **Commands**: `listModels`, `configure` utilize `readConfig` and `resolveProvider`.
