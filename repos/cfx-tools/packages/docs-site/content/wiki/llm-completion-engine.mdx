# LLM Completion Engine

The LLM Completion Engine provides a unified interface for generating LLM completions across multiple provider backends. It is designed to support repository upkeep tasks in the Conflux DevKit monorepo, including code review, documentation generation, commit preparation, and validation workflows.

## Core Functions

### `complete()`

Primary entry point for standard completions. Constructs a system prompt for repository upkeep tasks and combines user prompt with repository context.

```typescript
export async function complete({
  action,
  modelOverride,
  userPrompt,
  context,
  quick = false
})
```

**Parameters:**
| Parameter | Type | Description |
|-----------|------|-------------|
| `action` | `string` | Action identifier for config lookup |
| `modelOverride` | `string \| undefined` | Override default model selection |
| `userPrompt` | `string` | User's task description |
| `context` | `string` | Repository context to include |
| `quick` | `boolean` | Use reduced token budget |

**Returns:** `CompletionReport` with generated content, model info, and attempt history.

### `completeCommitAgent()` / `completeStructuredAgent()`

Specialized wrappers for commit-related and structured agent requests. Both delegate to `completeDirect()` with different prompt patterns.

```typescript
export async function completeCommitAgent(request: CompleteAgentRequest)
export async function completeStructuredAgent(request: CompleteAgentRequest)
```

### `completeDirect()`

Internal function for agent-specific completions with custom system prompts and progress callbacks.

**Parameters:**
| Parameter | Type | Description |
|-----------|------|-------------|
| `action` | `string` | Action identifier |
| `flags` | `{ model?: string, noThinking?: boolean }` | Request flags |
| `systemPrompt` | `string` | Custom system prompt |
| `userPrompt` | `string` | User prompt |
| `maxTokens` | `number \| undefined` | Token limit override |
| `onProgress` | `(event) => void` | Progress callback |

## Configuration

### Reading Configuration

```typescript
export async function readConfig(configFile?: string): Promise<LlmConfig>
export async function writeConfig(config: LlmConfig, configFile?: string): Promise<void>
```

**Configuration Resolution Order:**
1. Environment variable `CFXDEVKIT_LLM_AGENT_CONFIG_PATH` (agent-pinned, not overridden by sessions)
2. Environment variable `CFXDEVKIT_LLM_CONFIG_PATH`
3. `.pi/providers.json` (default)
4. `artifacts/llm/config/llm.json` (legacy)
5. `artifacts/llm/config/lemonade.json` (legacy compat)

**Scoped Config Merging:**
Config files under `/artifacts/llm/config/units/` are treated as scoped overlays and merged with base config via `mergeConfigLayers()`.

### Default Configuration

```typescript
export function defaultConfig(): LlmConfig
```

**Key defaults:**
- `provider`: `'lemonade'`
- `defaultModel`: `'Qwen3-Coder-Next-GGUF'`
- `requestTimeoutMs`: `600000` (10 minutes)
- `tokenBudget`: `{ contextFraction: 0.75, cap: 32768, quick: 512, cloudFallback: 4096 }`
- `harness.defaultMode`: `'deterministic'`

## Provider System

### Supported Providers

| Provider Type | Class | Environment Variables |
|---------------|-------|----------------------|
| `lemonade` | `LemonadeProvider` | `LEMONADE_URL`, `LEMONADE_BASE_URL` |
| `litellm` | `LiteLLMProvider` | `LITELLM_BASE_URL`, `LITELLM_API_KEY`, `LITELLM_MODEL` |
| `openai-compat` | `OpenAICompatProvider` | `OPENAI_BASE_URL`, `OPENAI_API_KEY`, `OPENAI_MODEL` |
| `github-models` | `GitHubModelsProvider` | `GITHUB_TOKEN`, `GITHUB_MODEL` |

### Provider Resolution

```typescript
export async function resolveProvider(config?: LlmConfig): Promise<LlmProvider>
```

**Resolution Priority:**
1. Config `provider` field
2. Config `baseUrl` with provider type
3. Environment variables (LiteLLM, Lemonade, OpenAI, GitHub)
4. Local discovery (tries default Lemonade URLs)

### Model Discovery

Each provider implements `discoverModels()` to fetch available models from the backend. Model selection follows this priority:

1. Exact match on preferred model ID or checkpoint
2. First model marked as `suggested`
3. First available model

## Context Building

### `buildActionContext()`

Constructs the full context payload for a completion request by combining:

```typescript
export async function buildActionContext(spec, opts: { quick?: boolean } = {})
```

**Context Sources:**
| Source | Description |
|--------|-------------|
| `spec.context` | Explicit file paths from action spec |
| `includeChangedFiles` | Git diff --name-only, staged, untracked |
| `includeGitDiff` | Full diff with stat (excludes pnpm-lock.yaml) |
| `includeCommitPreflight` | Pre-commit validation checks |
| `buildBaseContext()` | Core repo files (README, ARCHITECTURE, etc.) |

**Token Limits:**
- Quick mode: 12,000 characters total
- Normal mode: 60,000 characters total

### `commitPreflightBlock()`

Runs pre-commit validation commands and formats output for LLM consumption:

- `gitnexus ensure`
- `git status --short --branch`
- `git diff --stat` / `--cached --stat`
- `arch-check` hotspots and kebab groups
- `repo:review`
- `gitnexus detect-changes`
- `moon query changed-files`

## Streaming & Progress

### Progress Events

```typescript
export interface CompletionProgressEvent {
  readonly phase: 'request' | 'headers' | 'reasoning' | 'content' | 'heartbeat' | 'complete';
  readonly url: string;
  readonly attempt: number;
  readonly elapsedMs: number;
  readonly status?: number;
  readonly contentChars?: number;
  readonly reasoningChars?: number;
  readonly finishReason?: string;
  readonly promptTokens?: number;
  readonly completionTokens?: number;
  readonly tps?: number;
  readonly pp?: number;
}
```

**Progress Phases:**
1. `request` - Request sent
2. `headers` - Response headers received
3. `reasoning` - Reasoning content streaming
4. `content` - Answer content streaming
5. `heartbeat` - Periodic keepalive (every 15s)
6. `complete` - Stream finished with token metrics

### Streaming Implementation

```typescript
export async function postChatCompletion(params): Promise<string | undefined>
```

**Retry Logic:**
- Up to 2 retries per chat path
- 5-second delay when model not loaded (200 + empty body)
- Tries multiple chat paths: `/api/v1/chat/completions`, `/v1/chat/completions`, `/chat/completions`

**Error Handling:**
- Detects Lemonade error envelopes in 200 responses
- Merges malformed JSON chunks
- Escapes raw newlines in JSON strings
- Strips trailing commas

## Token Budget Management

### `resolveMaxTokens()`

Computes `max_tokens` for requests based on context window and budget policy:

**Resolution Order:**
1. `opts.maxTokens` if explicitly set
2. `ctx * contextFraction` (capped by `tokenBudget.cap`)
3. `tokenBudget.cloudFallback` (4096 default) when context unknown

**Local Hardware Tuning:**
```json
{
  "tokenBudget": {
    "contextFraction": 1.0,
    "cap": null
  }
}
```

## Error Handling

### Provider Errors

- Missing provider: `Error('No LLM provider could be resolved...')`
- Missing credentials: `Error('OPENAI_BASE_URL and OPENAI_API_KEY are required...')`
- Model not loaded: Empty response triggers retry with 5s delay

### Network Errors

- Timeout via `AbortSignal.timeout()`
- Retry logic with attempt tracking
- Error messages include status codes and causes

### JSON Parsing Errors

- Malformed response: `Error('LLM did not return a JSON object: ...')`
- Repair failures: Propagates last JSON parse error
