# LLM Completion Engine

The completion engine is the core runtime layer for all LLM interactions in the Conflux DevKit monorepo. It handles provider selection and routing, configuration loading and normalization, context assembly, HTTP communication with LLM endpoints (including streaming), and structured output parsing. Everything exposed from `workers/completion/index.ts` feeds the higher-level agent commands (`executeAction`, `runCommitWorkflow`, etc.) without those callers needing to know which provider is active.

## Entry Points

### `complete(options)` — `complete.ts`

The general-purpose completion function used by `executeAction` and similar callers. It applies a fixed system prompt establishing the assistant's role as a repository upkeep helper for the Conflux DevKit monorepo, then appends the supplied `context` and `userPrompt` as the user turn.

```ts
const report = await complete({
  action: 'review',
  userPrompt: 'Find regressions in the changed files.',
  context: await buildActionContext(spec),
  quick: false,           // if true, truncates context and uses a smaller token budget
  modelOverride: null,    // overrides config.actions[action] and config.defaultModel
});
```

Model resolution order: `modelOverride` → `config.actions[action]` → `config.defaultModel`.

Temperature is fixed at `0.2` for `complete`.

### `completeDirect(request)` — `complete.ts`

Used internally by `completeCommitAgent` and `completeStructuredAgent`. The caller supplies a fully formed `systemPrompt` instead of relying on the built-in system message. Temperature is `0.1`. The `flags.noThinking` option disables chain-of-thought reasoning on providers that support it (Lemonade/Qwen3).

Both `completeCommitAgent` and `completeStructuredAgent` are thin wrappers around `completeDirect` and exist as named exports so callers can communicate intent.

All three entry points return a `CompletionReport`:

```ts
interface CompletionReport {
  generatedAt: string;   // ISO timestamp
  action: string;
  baseUrl: string;
  model: string;
  content: string;
  attempts: readonly CompletionAttempt[];
}
```

## Configuration

### Loading — `config.ts`

`readConfig(configFile?)` implements a two-layer merge strategy:

1. **Base config** — searched in order: `.pi/providers.json` → `artifacts/llm/config/llm.json` → `artifacts/llm/config/lemonade.json`. The first file found wins.
2. **Scoped config** — if the resolved config path contains `/artifacts/llm/config/units/`, it is treated as a *scoped overlay* and merged on top of the base via `mergeConfigLayers`. Otherwise it is loaded standalone with `normalizeConfig`.

The active config path is resolved from environment variables in priority order:

```
CFXDEVKIT_LLM_AGENT_CONFIG_PATH  (agent commands, not overridden by session temp configs)
CFXDEVKIT_LLM_CONFIG_PATH        (general override)
.pi/providers.json               (default)
```

`writeConfig` always normalizes before writing, so the file on disk is always in canonical form.

`resolveRequestTimeoutMs` reads from `config.requestTimeoutMs`, then `LLM_REQUEST_TIMEOUT_MS`, then `LEMONADE_REQUEST_TIMEOUT_MS`, defaulting to 600,000 ms (10 minutes).

### Normalization — `config-normalize.ts`

`normalizeConfig(value)` coerces arbitrary JSON into a typed `LlmConfig`, filling gaps from `defaultConfig()`. Key defaults:

| Field | Default |
|---|---|
| `provider` | `'lemonade'` |
| `baseUrl` | `http://host.containers.internal:13305/` |
| `defaultModel` | `Qwen3-Coder-Next-GGUF` |
| `requestTimeoutMs` | 600,000 ms |
| `tokenBudget.contextFraction` | 0.75 |
| `tokenBudget.cap` | 32,768 |
| `tokenBudget.quick` | 512 |
| `tokenBudget.cloudFallback` | 4,096 |

`mergeConfigLayers(base, scoped)` deep-merges `actions`, `providerProfiles`, `actionPolicies`, `tokenBudget`, and `harness` sub-objects so scoped unit configs can override individual fields without replacing the entire base.

### Token Budget

`resolveMaxTokens` in `provider-stream.ts` applies this resolution order at request time:

1. `opts.maxTokens` if explicitly provided.
2. `floor(contextWindow × contextFraction)`, capped by `tokenBudget.cap` (or uncapped if `cap === null`).
3. `tokenBudget.quick` (default 512) for quick-mode calls without a context window.
4. `tokenBudget.cloudFallback` (default 4096) when the model reports no context window.

For local hardware with large unified memory (e.g. Strix Halo), set `cap: null` and `contextFraction: 0.95` or similar in `providers.json`. Cloud/gateway providers default to the conservative 4096-token fallback.

## Provider System

### Resolution — `providers.ts`

`resolveProvider(config?)` selects an `LlmProvider` implementation using this waterfall:

1. `config.provider === 'github-models'` → `GitHubModelsProvider`
2. `config.baseUrl` present:
   - `litellm` → `LiteLLMProvider`
   - `openai-compat` → `OpenAICompatProvider`
   - otherwise → `LemonadeProvider`
3. `LITELLM_BASE_URL` env → `LiteLLMProvider`
4. `LEMONADE_URL` / `LEMONADE_BASE_URL` env → `LemonadeProvider`
5. Default Lemonade URLs probed — if any model is discovered, use it.
6. `OPENAI_BASE_URL` + `OPENAI_API_KEY` → `OpenAICompatProvider`
7. `GITHUB_TOKEN` → `GitHubModelsProvider`
8. Throws if nothing resolves.

### Provider Classes — `provider-classes.ts`

All four providers implement `LlmProvider`:

```ts
interface LlmProvider {
  readonly type: LlmProviderType;
  complete(messages: readonly ChatMessage[], opts?: CompletionOptions): Promise<string>;
  discoverModels(): Promise<readonly LlmModel[]>;
  chooseModel(models: readonly LlmModel[], preferred?: string | null): LlmModel | undefined;
}
```

**`LemonadeProvider`** — targets the Lemonade local inference server. Tries multiple chat paths (`/api/v1/chat/completions`, `/v1/chat/completions`, `/chat/completions`) and up to two retries per path. Detects the "model loading" condition (200 OK + empty body) and waits 5 seconds before retrying. Supports multiple `baseUrls` for container/host network discovery.

**`LiteLLMProvider`** / **`OpenAICompatProvider`** — standard OpenAI-compatible chat completions with bearer auth. Both discover models from `/models` and fall back to `LITELLM_*` / `OPENAI_*` environment variables.

**`GitHubModelsProvider`** — targets `https://models.inference.ai.azure.com` with `GITHUB_TOKEN` auth.

Model selection in all providers delegates to `chooseBestModel(models, preferred)`, which tries exact `id`/`checkpoint` match, then the first model flagged `suggested`, then `models[0]`.

### Model Resolution — `provider-meta.ts`

`resolveProviderModel(provider, preferred?)` calls `provider.discoverModels()` and returns the identifier of the best match. `modelIdentifier(model, fallback)` prefers `model.id` over `model.checkpoint` over the fallback string.

`parseModels(text, baseUrl)` parses both array-root and `{ data: [] }` response shapes, mapping the Lemonade-extended fields (`checkpoint`, `labels`, `recipe`, `suggested`, `max_context_window`) into `LlmModel`.

## HTTP Layer — `provider-stream.ts` / `provider-stream-parse.ts`

`postChatCompletion` sends a single chat completion request. It:

- Sets `stream: true` only when `opts.onProgress` is provided.
- Uses `AbortSignal.timeout` for the configured deadline.
- Emits `CompletionProgressEvent` at the `request` and `headers` phases unconditionally.
- For non-streaming responses, calls `extractAssistantText` to handle both standard OpenAI shapes and Lemonade error envelopes.
- For streaming responses, delegates to `readStreamedChatCompletion`.

`readStreamedChatCompletion` processes the SSE byte stream chunk by chunk:

- Splits on `\r\n\r\n` event boundaries.
- Calls `applyChatCompletionChunk` for each event block, which parses `data:` lines and accumulates `content` and `reasoning_content` deltas.
- Detects phase transitions (`reasoning` → `content`) and emits progress events.
- Emits a `heartbeat` event every 15 seconds for long-running completions.
- Emits a `complete` event with `finishReason` and server timing metrics (`tps`, `pp`, token counts) at the end.
- Detects Lemonade's "no model loaded" error envelope (200 + JSON body with `error` key) and returns an empty string to trigger a retry.

`emitProgress` wraps `opts.onProgress` in a try/catch so progress reporting never breaks a completion.

## Context Assembly — `context.ts`

`buildActionContext(spec, opts)` assembles the user-facing context string injected before the task prompt. It:

1. Reads files listed in `spec.context` via `readContextFile` (repo-relative paths).
2. Optionally appends `changedFilesBlock()` (unstaged + staged + untracked files).
3. Optionally appends `gitDiffBlock()` (stat + diff excerpt, excluding `pnpm-lock.yaml`).
4. Optionally appends `commitPreflightBlock()` — runs `gitnexus ensure`, `git status/diff`, hotspot scan, kebab group check, `repo:review`, `gitnexus detect-changes`, and `moon query changed-files` in parallel and concatenates their output.
5. Appends `buildBaseContext()` (README, ARCHITECTURE, LLM automation docs, corpus manifest).

Total context is hard-truncated to 60,000 characters (12,000 in quick mode). Individual files are truncated to 12,000 characters each.

`writeLlmReport(action, response)` writes the completion output as a Markdown report to `artifacts/llm/reports/llm-{action}.md`.

## JSON Parsing — `json.ts`

`parseJsonObject(content)` extracts a JSON object from LLM output that may include prose, markdown fences, or malformed JSON. It:

1. Extracts fenced content if present.
2. Falls back to finding the first `{` and locating its balanced closing `}` using a character-level parser that handles string escaping.
3. Attempts `JSON.parse` on the candidate, then with raw newlines escaped in strings, then with trailing commas stripped. Throws only if all three attempts fail.

## Monorepo Units — `units.ts`

`findMonorepoUnitByConfigPath(configFilePath)` maps a config file path back to one of the three defined monorepo units (`delivery`, `implementation`, `operations`). It resolves `configPath` to the unit config at `artifacts/llm/config/units/{name}.json`. This is used by `resolveExecutionContext` to surface which unit scope is active in CLI output.

---

## Execution Context — `shared/execution-context.ts`

`resolveExecutionContext(options)` is called by all agent entry points (`executeAction`, `runCommitWorkflow`, `runAgentCheck`, etc.) before doing any LLM work. It:

1. Identifies the active unit from the config path.
2. If `useLlm: true`, probes the provider and resolves the model, reporting `ready` or `unavailable`.
3. Returns an `ExecutionContextSummary` that callers use for logging and runtime payloads.

`logExecutionContext` / `renderExecutionContextLines` format this summary for console output:
```
Execution context:
  - unit: implementation (repos)
  - config: .pi/providers.json
  - llm: lemonade :: Qwen3-Coder-Next-GGUF @ http://localhost:13305/
```

## Key Types

| Type | Purpose |
|---|---|
| `LlmConfig` | Full normalized configuration object |
| `LlmProvider` | Interface implemented by all four provider classes |
| `CompletionOptions` | Per-request options: model, temperature, token budget, streaming callback |
| `CompletionReport` | Returned by all `complete*` functions |
| `CompletionProgressEvent` | Emitted during streaming: `request` → `headers` → `reasoning` → `content` → `heartbeat` → `complete` |
| `LlmTokenBudget` | Token budget policy embedded in config |
| `CompleteAgentRequest` | Input shape for `completeCommitAgent` and `completeStructuredAgent` |
| `MonorepoUnit` | Resolved unit metadata including config and root paths |

## Adding a New Provider

1. Implement `LlmProvider` in `provider-classes.ts`. Delegate `complete` to `postChatCompletion` and model selection to `chooseBestModel`.
2. Add the type string to `LlmProviderType` in `types.ts`.
3. Handle the new type in `normalizeProviderType` (`config-normalize.ts`) and `configuredProvider` / `resolveProvider` (`providers.ts`).
4. Export the class from `index.ts` if external callers need direct access.
