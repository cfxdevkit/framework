# repos — cfx-tools

# @cfxdevkit/llm-agents

## Overview

`@cfxdevkit/llm-agents` provides a suite of typed LLM workflow agents for automating Conflux DevKit repository operations. It orchestrates large language model interactions to handle code review, repository validation planning, commit message generation, changeset creation, and model viability testing.

The module acts as the bridge between the repository's CI/CD and validation tools (such as `cdk-repo-check` and `docs-pipeline`) and the LLM completion layer, executing complex multi-step workflows that produce OpenSpec artifacts, Git branches, and validated commits.

## Architecture

The module is built around a `workers/` architecture where distinct agents handle specific domains. The public API is exposed via `src/index.ts`.

## Core Agents

### `runAgentCheck` (Repository Check & Planning)

Located in `workers/agents/check.ts`, this agent automates the remediation of repository validation failures. It follows a "Plan and Handoff" pattern:

1.  **Validation:** Runs `pnpm repo check` and reads the validation artifact (`artifacts/llm/repo-check/checks/validation.json`).
2.  **Analysis:** Collects actionable steps (warnings/errors) and summarizes them for the LLM.
3.  **Planning:** Uses `buildAgentCheckPlan` to generate a structured OpenSpec change set. The LLM is instructed to group related failures into contained remediation changes.
4.  **Materialization:** Calls `materializeOpenSpecChanges` to generate proposal, design, spec, and task files based on the plan and live OpenSpec instructions.
5.  **Handoff:** Optionally creates a Git branch (e.g., `opsx/repo-check-handoff`) and a draft PR to hand off the plan to cloud-based implementation agents.

**Key Types:**
*   `AgentCheckFlags`: Controls `--quick`, `--dry-run`, `--create-branch`, and `--draft-pr`.
*   `AgentCheckPlan`: Defines the summary, changes, branch details, and handoff instructions.

### `runCommitWorkflow` (Commit Orchestration)

Located in `workers/commit/commit.ts`, this function implements a gated commit pipeline:

1.  **Quality Gates:** Runs validation sequences (format, lint, typecheck, tests, hotspots).
2.  **Failure Analysis:** If gates fail, `analyzeGateFailures` invokes the LLM to provide root cause analysis and minimal fixes. The commit is blocked unless `--force` is used.
3.  **Scope Detection:** Identifies changed packages and files.
4.  **Changeset Generation:** Determines if a changeset is needed. Uses `generateChangesetPlan` to request LLM-generated release notes and bump levels for publishable packages.
5.  **Approval:** Presents the commit message and changeset details. Supports `--yes`, `--dry-run`, and interactive approval.
6.  **Commit:** Stages files, runs post-generation checks (optional), and executes the commit.

**Key Types:**
*   `CommitWorkflowResult`: Returns status (`blocked`, `committed`, `dry-run`), scopes, and changeset plan.

### `runReviewAgent` (Code Review)

Located in `workers/agents/review.ts`, this agent scans uncommitted changes for high-risk patterns:

*   **Generated Files:** Detects changes to files in `artifacts/`, `dist/`, etc.
*   **Security Sensitivity:** Flags changes to keystore, wallet, or secret files.
*   **Code Hotspots:** Integrates with `codeHotspotReport` to flag files exceeding line limits (hard/soft).
*   **Validation Suggestions:** Recommends relevant CI commands based on changed file paths.

### `runAgentSmoke` (Model Viability)

Located in `workers/agents/smoke.ts`, this agent tests the viability of configured LLM models:

*   **Probes:** Executes cold, hot, and task-based probes (with and without thinking mode).
*   **Tiering:** Assigns models to tiers (Lightweight, Documentation, Reasoning) based on configured actions.
*   **Metrics:** Reports latency, JSON shape validity, and reasoning observation.
*   **Output:** Renders a tabular summary via `renderSmokeTable`.

## CLI & Configuration

### Configuration (`configure`)

The `configure` command in `workers/commands.ts` manages LLM provider settings stored in `.pi/providers.json`:

*   `set provider <type>`: Set provider type (`lemonade`, `litellm`, `openai-compat`, `github-models`).
*   `set base-url <url>`: Configure the provider endpoint.
*   `set default-model <id>`: Select the default model, optionally triggering an interactive selection.
*   `set action <name> <model>`: Bind a specific repository action to a model.

### Action Execution (`executeAction`)

The `executeAction` function provides a generic runner for repository actions defined in `repoActions`:

1.  **Parsing:** Extracts flags like `--model`, `--quick`, and `--generate`.
2.  **Context:** Resolves execution context and LLM configuration.
3.  **Completion:** Invokes the LLM via `complete` with the action's context and prompt.
4.  **Special Handling:**
    *   **Changeset:** The `--generate` flag for the `changeset` action bypasses the LLM review flow and directly generates `.changeset/*.md` files using `generateChangesetPlan`.

## Shared Runtime

Common utilities are centralized in `workers/agents/runtime/`:

*   **Reports:** `writeJsonReport` and `writeMarkdownReport` handle output to `artifacts/llm/`.
*   **Corpus:** `collectCorpusFiles` and `chunkFile` prepare documentation and source code for LLM context.
*   **Paths:** Utilities like `isSecuritySensitive`, `isGeneratedPath`, and `tierForPath` categorize files.
*   **Models:** `codeHotspotReport` executes the arch-check tool to analyze code size.

## Exports

The module exports the following key interfaces and functions from `src/index.ts`:

*   **Agents:** `runAll`, `runAgentCheck`, `runReviewAgent`, `runAgentSmoke`.
*   **Commit:** `runCommit`, `runCommitWorkflow`, `runPrecommit`, `parseCommitFlags`.
*   **Docs:** `runDocsApi`, `runDocsReadme`, `runWikiGenerate`, etc.
*   **Commands:** `configure`, `executeAction`, `listActions`, `listModels`, `validateModels`.
*   **Types:** `RepoActionExecutionResult`, `RepoActionDefinition`, `CommitWorkflowResult`, `AgentCheckFlags`.
