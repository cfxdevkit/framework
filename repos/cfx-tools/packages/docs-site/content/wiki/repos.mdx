# repos

# Repos Module Overview

The `repos` module serves as the centralized codebase for the Conflux DevKit ecosystem, orchestrating foundational libraries, on-chain automation, cryptographic trust boundaries, and developer tooling. It enforces a strict vertical architecture where lower-tier primitives power higher-level domain logic and frontend surfaces, while cross-cutting tooling maintains workspace integrity and CI/CD automation.

## Architectural Governance
The entire ecosystem is structured and validated by `[cfx-meta](cfx-meta.md)`, which defines unidirectional dependency tiers and publishes `@cfxdevkit/arch-rules` for runtime constraint checking. This module ensures that core libraries, domain logic, and UI components operate strictly within their designated tiers. Validation scripts in `cfx-meta` verify `workspace:*` consistency, directory layouts, and cross-package dependency graphs during CI, preventing dependency leaks and enforcing semver policies across the monorepo.

## Runtime Integration & Data Flow
Runtime packages communicate through well-defined trust and client boundaries:
- **Client & RPC Layer:** `@cfxdevkit/cdk` from `[cfx-core](cfx-core.md)` provides the unified `ConfluxDevkitClient`, address/chain utilities, and typed error handling. All downstream packages route chain interactions through this client.
- **Cryptographic Trust Boundary:** `[cfx-keys](cfx-keys.md)` (`@cfxdevkit/services`) isolates private key material behind `KeystoreProvider`. Downstream consumers receive `Signer` objects for transaction preparation without ever accessing raw secrets or crossing into `CryptoError` territory.
- **Contract Execution Surface:** `[cfx-solidity](cfx-solidity.md)` exposes `@cfxdevkit/abis` for type-safe EVM interfaces and `@cfxdevkit/contracts` for typed read/write operations and Core/eSpace bridge helpers. These interfaces are consumed directly by automation strategies and UI swap widgets.
- **On-Chain Automation:** `[cfx-domain](cfx-domain.md)` (`@cfxdevkit/automation`) orchestrates long-lived worker processes that evaluate trading conditions via `PriceChecker` and `TWAPEvaluator`, manage job state through `executionRepository`, and submit transactions using the `Signer` abstraction. It enforces resilience via `retryQueue` and comprehensive audit logging.
- **Frontend & Tooling:** `[cfx-ui](cfx-ui.md)` (`@cfxdevkit/defi-react`) consumes client state and keystore hooks to render DeFi primitives and swap interfaces. `[cfx-tools](cfx-tools.md)` (`@cfxdevkit/llm-agents`) bridges CI/CD pipelines with LLM completion, leveraging architecture rules from `cfx-meta` and contract/client definitions from `cfx-core` and `cfx-solidity` to generate OpenSpec artifacts, plan validations, and automate repository workflows.

## Sub-Modules
- `[cfx-meta](cfx-meta.md)` — Architecture rules, tier enforcement, and workspace validation
- `[cfx-core](cfx-core.md)` — Foundational primitives, RPC client, and dev node tooling
- `[cfx-keys](cfx-keys.md)` — Keystore abstraction, cryptographic boundary, and embedded wallet management
- `[cfx-domain](cfx-domain.md)` — Resilient on-chain job execution and trading automation
- `[cfx-solidity](cfx-solidity.md)` — Unified Solidity pipeline, ABI definitions, and typed contract interfaces
- `[cfx-tools](cfx-tools.md)` — LLM-driven repository automation and CI/CD workflow agents
- `[cfx-ui](cfx-ui.md)` — React primitives, DeFi widgets, and data-fetching hooks
