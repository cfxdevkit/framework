# PI Agent & Provider Integration

## Overview

The `pi-agent` module is the bridge between the `cfx-tools` developer toolchain and the PI coding agent shell (`@earendil-works/pi-coding-agent`). It registers chat commands, exposes LLM-backed repo tools, manages provider configuration, and launches PI subprocesses. The module has two distinct surfaces:

- **Repo commands** (`/repo-*`) — LLM-driven workflows for validation, action execution, and commits
- **CDK commands** (`/cdk`) — Purely deterministic wrappers over `@cfxdevkit/cli` functions, no LLM involved

## Entry Points

### `registerPiAgentProjectExtension(pi: ExtensionAPI)`

The top-level registration function called by the PI extension loader. It wires everything together:

1. Resolves the provider bridge from config/environment and registers it with PI
2. Registers all `/repo-*` commands and `/cdk` commands
3. Registers the four LLM-backed tools (`repo_agent_check`, `repo_action_catalog`, `repo_run_action`, `repo_commit_workflow`)
4. Attaches `session_start` and `session_tree` event handlers that refresh the status bar UI
5. Attaches a `session_shutdown` handler that clears all widgets

The scope (monorepo unit) is read from `CFXDEVKIT_PI_SCOPE` via `resolvePiScopeFromEnv()` and propagated to all config and provider resolution calls.

### `registerPiAgentProjectExtension` vs. direct function exports

When integrating programmatically (e.g. from `tooling-cli`), you can import individual pieces:

```ts
import { runPiInteractive, runPiRpc, runPiCommit } from '@cfxdevkit/pi-agent';
import { resolveRuntimeBridgeState } from '@cfxdevkit/pi-agent';
```

## Command Registration

### Repo Commands (`commands.ts`)

Registered by `registerPiRepoCommands`. All handlers are async and catch errors, routing them to `ctx.ui.notify`.

| Command | Description |
|---|---|
| `/repo-status` | Shows current provider, model, scope, and action count |
| `/repo-actions [deterministic\|exploratory]` | Lists available actions from the llm-agents registry |
| `/repo-run <action> [--quick] [--model <id>] [prompt]` | Executes a named action via `executePiRepoAction` |
| `/repo-check [--dry-run] [--create-branch] [--quick]` | Runs the full validation → OpenSpec planning pipeline |
| `/repo-commit [--quick] [--model <id>] [prompt]` | Runs the interactive commit workflow without leaving the PI session |

All results are emitted as `repo-agent-summary` custom messages with a tone of `info`, `success`, `warning`, or `error`. The registered renderer renders these as a styled `Box` containing a title and up to 4 lines (expandable).

### CDK Commands (`commands-cdk.ts`)

Registered by `registerPiCdkCommands`. These handlers are **purely deterministic** — no LLM calls, no agent loop.

| Command | Underlying function |
|---|---|
| `/cdk status [--chain <id\|name>] [--rpc <url>]` | `executeCdkStatus` → `runStatus` |
| `/cdk derive --mnemonic "<phrase>" \| --generate [options]` | `executeCdkDerive` → `runDerive` |
| `/cdk generate [--strength 128\|256]` | `executeCdkGenerate` → `runGenerate` |
| `/cdk contracts extract [--artifacts <dir>] [--out <dir>]` | `executeCdkContractsExtract` → `runContractsExtract` |

Argument parsing uses `parseArgs` from `@cfxdevkit/cli`, with helpers from `commands-cdk-flags.ts`:

- `tokenize(raw)` — splits respecting quoted strings
- `str(flags, key)` — extracts a string flag or `undefined`
- `num(flags, key)` — extracts and validates a numeric flag
- `bool(flags, key)` — extracts a boolean flag

Results are emitted as `cdk-result` custom messages with their own renderer.

## Provider Configuration

### Config Layers

Configuration is stored in JSON files and loaded with a two-layer merge strategy:

1. **Base config** — `.pi/providers.json` (or `CFXDEVKIT_LLM_CONFIG_PATH`)
2. **Scoped config** — `artifacts/llm/config/units/<scope>.json` (when a scope is active)

`readPiConfig()` always reads the base first, then merges the scoped config on top using `mergePiConfigLayers`. The agent-pinned env var `CFXDEVKIT_LLM_AGENT_CONFIG_PATH` takes priority over `CFXDEVKIT_LLM_CONFIG_PATH` to prevent session overrides (e.g. temporary commit policy files) from affecting the main provider resolution.

### `PiLlmConfig` Structure

Key fields in `config-types.ts`:

```ts
interface PiLlmConfig {
  provider: 'lemonade' | 'litellm' | 'openai-compat' | 'github-models' | null;
  baseUrl: string | null;
  defaultModel: string | null;
  githubModel: string | null;
  requestTimeoutMs: number;            // default: 600000
  actions: Record<string, string>;     // legacy: action → model
  providerProfiles: Record<string, ProviderProfile>;
  actionPolicies: Record<string, ActionPolicy>;
  harness: {
    defaultMode: 'deterministic' | 'exploratory';
    providerStrategy: 'auto' | 'gateway' | 'direct';
    deterministic: { preserveDeterministicArtifacts, preserveDeterministicSections };
    exploratory: { allowCodeChanges, allowWideChanges };
  };
}
```

Provider profiles and action policies allow per-action model/provider overrides, including per-phase granularity (e.g. `commit.message-generation` can use a different model than `commit.failure-analysis`).

### Policy Resolution (`config-policy.ts`)

`resolveEffectiveActionPolicy(config, { action?, phase? })` implements a three-level lookup:

1. **Phase policy** — `actionPolicies[action].phases[phase]`
2. **Action policy** — `actionPolicies[action]` or legacy `actions[action]`
3. **Default** — top-level `defaultModel`

It returns a `PiEffectiveActionPolicy` containing the resolved model, profile, provider, baseUrl, and the `source` field indicating which level matched.

### Provider Discovery (`provider-discovery.ts`)

`resolveProviderState(config)` determines the active provider in this order:

1. Explicit `config.baseUrl` with `config.provider`
2. `LITELLM_BASE_URL` environment variable
3. `LEMONADE_URL` or `LEMONADE_BASE_URL` environment variable
4. Probe default Lemonade URLs: `localhost:13305`, `127.0.0.1:13305`, Docker/container variants, `127.0.0.1:8000`
5. `OPENAI_BASE_URL` + `OPENAI_API_KEY`
6. `GITHUB_TOKEN` → GitHub Models endpoint

Model discovery (`discoverProviderModels`) attempts `/api/v1/models`, `/v1/models`, and `/models` paths with a 3-second timeout. The `PiLlmModel` type carries `id`, `checkpoint`, `recipe`, `labels`, `suggested`, and `size` fields.

### Provider Bridge (`provider-bridge.ts`)

`createPiProviderBridge(scope?)` is the main assembly point. It returns a `PiProviderBridge` combining:
- The full resolved config
- Discovered models and default model
- A `PiCliInvocation` with the correct provider, model, and environment variables for spawning PI subprocesses
- The `providerStrategy` from harness config

`registerPiProviderBridge(pi, bridge)` calls `pi.registerProvider('openai', ...)` to inject the provider into the PI shell. All local/lemonade/litellm providers are presented as OpenAI-compatible.

## LLM Tools

### Registered Tools (`tools.ts`)

`registerPiRepoTools` installs four tools into the PI session:

**`repo_agent_check`** — Runs validation and OpenSpec change planning. Returns structured result including validation status, change names, and plan summary. Parameters: `dryRun`, `createBranch`, `quick`.

**`repo_action_catalog`** — Lists available actions from the llm-agents registry. Optional `mode` filter (`deterministic`/`exploratory`).

**`repo_run_action`** — Executes a named action. Parameters: `action` (required), `prompt`, `quick`, `model`. Validates the action name against the registry before calling.

**`repo_commit_workflow`** — Runs `executePiCommitSession` and surfaces gate results, approval state, and commit preview.

### LLM Agents Runtime (`llm-agents-runtime.ts`)

This module provides lazy dynamic imports of `@cfxdevkit/llm-agents`. At startup it checks whether the compiled `dist/index.js` exists; if not, it falls back to the source `src/index.js`. This enables both installed and development usage.

The three primary functions:

- `executePiAction(args)` → `executeAction` in llm-agents
- `executePiAgentCheck(options)` → `runAgentCheck` in llm-agents  
- `executePiCommitWorkflow(args, options)` → `runCommitWorkflow` with `approvalMode: 'defer'`

`getPiActionDefinitions()` returns the static action registry used by both the catalog command and the action name validator in `executePiRepoAction`.

### Commit Session (`tools-commit.ts`)

`executePiCommitSession` handles the per-invocation policy resolution for commit workflows:

1. Reads config and resolves policies for `commit`, `commit.message-generation`, and `commit.failure-analysis` phases independently
2. If a commit profile specifies a different provider/baseUrl, writes a temporary JSON config file to `$TMPDIR` and sets `CFXDEVKIT_LLM_CONFIG_PATH` for the duration of the call (`withScopedCommitPolicy`)
3. Suppresses console output during execution via `withCapturedConsole`
4. Always cleans up the temp file and restores the environment variable

## Runtime: Spawning PI Subprocesses (`runtime.ts`)

`runPiInteractive`, `runPiPrint`, and `runPiRpc` launch the PI CLI binary as a subprocess with full stdio inheritance. The binary is resolved relative to the package's own `node_modules/.bin/pi`.

Key environment variables injected into the subprocess:
- `CFXDEVKIT_LLM_CONFIG_PATH` — scoped config path
- `CFXDEVKIT_PI_SCOPE` — active scope name
- `OPENAI_API_KEY` — provider credential
- `PATH` — prepended with `~/.pi/agent/bin` and the PI binary directory

`runPiCommit` is a convenience wrapper that calls `runPiInteractive` with a structured prompt asking the agent to start a commit session and run `/repo-commit`.

Terminal phase hooks (`PiTerminalPhaseHooks`) allow callers to run setup/teardown around the subprocess without interleaving with PI's TUI:

```ts
await runPiInteractive({
  terminalPhases: {
    beforeStart: async () => { /* Inquirer prompts, etc. */ },
    afterExit: async () => { /* post-session cleanup */ },
  }
});
```

## UI Utilities (`ui.ts`, `ui-cdk.ts`)

All UI state is represented as `PiOperatorUiState`:

```ts
interface PiOperatorUiState {
  statusText: string;    // shown in the status bar
  widgetKey: string;     // identifies the widget slot
  widgetLines: string[]; // content lines
}
```

Factory functions:

| Function | Produces |
|---|---|
| `createPiRuntimeUiState` | Provider, model, scope, action count, active OpenSpec changes |
| `createPiAgentCheckUiState` | Validation status, step counts, planned change names |
| `createPiRepoActionUiState` | Action name, mode, model, response preview |
| `createPiCommitWorkflowUiState` | Status, phase, gate summaries, commit preview, sha |
| `createPiGateUiState` | Gate pass/fail with per-result breakdown |
| `createCdkStatusUiState` | Chain connectivity results |
| `createCdkDeriveUiState` | Derived addresses (optional private keys/mnemonic) |
| `createCdkGenerateUiState` | Generated mnemonic and word count |
| `createCdkContractsUiState` | Extraction directory paths and count |

`clearPiOperatorWidgets` clears all five known widget slots. `setPiWorkflowProgress` / `clearPiWorkflowProgress` control the working indicator shown during long-running operations.

## Tooling Runtime Facade (`tooling-runtime.ts`)

Provides a stable API for external consumers (e.g. `tooling-cli`) that need provider information without directly importing the internals:

- `resolveRuntimeBridgeState(scope?, { action?, phase? })` — returns provider type, URL, model list, strategy, and effective policy in one call
- `resolveProvider()` — returns a `PiToolingProvider` with `discoverModels()` and `chooseModel()` methods
- `resolveProviderModel(provider, preferred?)` — picks the best model: exact match → suggested → first available → fallback
- `readConfig()` / `writeConfig(config)` / `defaultConfig()` — thin wrappers over config storage

## Adding a New Command

**Deterministic command** (no LLM): add a case in `registerPiCdkCommands`, implement the handler calling a `tools-cdk.ts` executor, add a UI state factory in `ui-cdk.ts`.

**LLM-backed command**: add a handler in `registerPiRepoCommands`, call `executePiRepoAction` or a new `llm-agents-runtime` function, build UI state in `ui.ts`, emit with `emitPiOperatorMessage`.

**New tool** (callable by the agent): define with `defineTool` from `@earendil-works/pi-coding-agent`, register in `registerPiRepoTools`. Keep tool execution focused — update the UI state bar, then return structured `details` alongside the text `content`.
