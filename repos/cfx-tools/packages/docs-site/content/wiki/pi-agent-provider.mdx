# PI Agent & Provider Integration Module

## Overview

The PI Agent & Provider Integration module provides the core infrastructure for the Conflux development CLI (`cfxdevkit`) to interact with LLM-powered agent workflows. It bridges the chat interface with repository operations, LLM provider selection, and deterministic/LLM-executed tooling.

**Key responsibilities:**
- Register slash commands (`/cdk`, `/repo-*`) for developer workflows
- Manage LLM provider configuration and discovery
- Execute agent workflows (validation, commit, actions)
- Provide deterministic CDK utilities (status, derive, generate, contracts)
- Render UI state in the PI chat interface

## Key Components

### Commands (`commands.ts`, `commands-cdk.ts`)

Slash command handlers that map user input to tool execution.

#### Repository Commands

| Command | Handler | Description |
|---------|---------|-------------|
| `/repo-status` | `registerPiRepoCommands` | Show current scope, provider, and model context |
| `/repo-actions` | `registerPiRepoCommands` | List available shared repo workflows |
| `/repo-run` | `registerPiRepoCommands` | Run a shared repo action with prompt/model options |
| `/repo-check` | `registerPiRepoCommands` | Run validation → OpenSpec change planning pipeline |
| `/repo-commit` | `registerPiRepoCommands` | Run interactive repository commit workflow |

#### CDK Commands

| Command | Handler | Description |
|---------|---------|-------------|
| `/cdk status` | `handleCdkStatus` | Query chain status with optional `--chain` and `--rpc` |
| `/cdk derive` | `handleCdkDerive` | Derive addresses from mnemonic with optional `--mnemonic` or `--generate` |
| `/cdk generate` | `handleCdkGenerate` | Generate new mnemonic with `--strength` option |
| `/cdk contracts extract` | `handleCdkContracts` | Extract contract artifacts with `--artifacts` and `--out` |

**Flag parsing** is handled by `tokenize()` from `commands-cdk-flags.ts`, which supports quoted strings and space-separated tokens. Type conversion helpers (`str`, `num`, `bool`) extract typed values from parsed flags.

### Configuration (`config*.ts`)

Configuration management for LLM providers and action policies.

#### Core Functions

```typescript
// Load and normalize configuration
readPiConfig(configPath?) → Promise<PiLlmConfig>
writePiConfig(config: PiLlmConfig, configPath?) → Promise<void>

// Configuration resolution
resolvePiConfigPath(scope?) → string
resolveEffectiveActionPolicy(config, {action?, phase?}) → PiEffectiveActionPolicy
resolveNamedProviderProfile(config, profileName?) → PiEffectiveActionPolicy['profile']

// Layer merging
mergePiConfigLayers(baseConfig, scopedConfig) → PiLlmConfig
normalizePiConfig(value) → PiLlmConfig
```

#### Configuration Priority

1. **Base config**: `.pi/providers.json` in workspace root
2. **Scoped config**: `artifacts/llm/config/units/<scope>.json`
3. **Environment overrides**: `CFXDEVKIT_LLM_CONFIG_PATH`, `CFXDEVKIT_LLM_AGENT_CONFIG_PATH`

The `CFXDEVKIT_LLM_AGENT_CONFIG_PATH` environment variable takes precedence for agent-pinned sessions (e.g., `--github` or `--local` PI sessions), ensuring provider configuration survives session overrides.

### Provider Bridge (`providers.ts`, `provider-*.ts`)

Manages LLM provider discovery, connection, and model resolution.

#### Provider Discovery Flow

```typescript
createPiProviderBridge(scope?) → Promise<PiProviderBridge>
  └─> resolveProviderState(config)
        ├─> Check config.baseUrl
        ├─> Check LITELLM_BASE_URL env
        ├─> Check LEMONADE_URL env
        ├─> Scan defaultBaseUrls (localhost:13305, etc.)
        ├─> Check OPENAI_BASE_URL + OPENAI_API_KEY
        └─> Check GITHUB_TOKEN
```

#### Provider Types

| Type | Environment Variables | Default Base URL |
|------|----------------------|------------------|
| `lemonade` | `LEMONADE_URL`, `LEMONADE_BASE_URL` | `http://localhost:13305/` |
| `litellm` | `LITELLM_BASE_URL` | `http://localhost:13305/` |
| `openai-compat` | `OPENAI_BASE_URL`, `OPENAI_API_KEY` | N/A |
| `github-models` | `GITHUB_TOKEN` | `https://models.inference.ai.azure.com` |

#### Model Resolution

```typescript
resolvePiModel(defaultModel, models) → string | null
```

Selects the model by priority:
1. Explicit `defaultModel` from config
2. First available model from provider catalog
3. `null` if no models found

### Runtime (`runtime.ts`)

Session execution and PI CLI spawning.

#### Execution Functions

```typescript
runPiInteractive(options?) → Promise<void>
runPiPrint(options) → Promise<void>
runPiRpc(options?) → Promise<void>
runPiCommit(options?) → Promise<void>
```

All functions call `runPiCli()` which:
1. Resolves workspace root via `findWorkspaceRoot()`
2. Locates PI binary (`pi` or `pi.cmd`)
3. Spawns `pnpm exec` with appropriate environment variables
4. Manages terminal phases (`beforeStart`, `afterExit`)

#### Environment Variables Set

| Variable | Purpose |
|----------|---------|
| `CFXDEVKIT_LLM_CONFIG_PATH` | Provider configuration path |
| `CFXDEVKIT_PI_SCOPE` | Agent scope name |
| `OPENAI_API_KEY` | Provider authentication |

### Tools (`tools.ts`, `tools-cdk.ts`, `tools-commit.ts`)

Tool definitions and executors for the PI agent framework.

#### Repository Tools

| Tool | Description |
|------|-------------|
| `repo_agent_check` | Run validation → OpenSpec change planning |
| `repo_action_catalog` | List available repo workflows |
| `repo_run_action` | Execute a specific repo workflow |
| `repo_commit_workflow` | Run interactive commit workflow |

#### CDK Tools

| Tool | Description |
|------|-------------|
| `executeCdkStatus` | Query chain status |
| `executeCdkDerive` | Derive addresses from mnemonic |
| `executeCdkGenerate` | Generate new mnemonic |
| `executeCdkContractsExtract` | Extract contract artifacts |

**CDK tools are PURELY DETERMINISTIC** — they never call LLM agents. They wrap `@cfxdevkit/cli` functions directly.

#### Commit Session Policy Override

`executePiCommitSession()` supports scoped configuration overrides via temporary files:

```typescript
resolveCommitRuntimePolicy(explicitModel?) → { modelPolicies, profileOverride }
  └─> withScopedCommitPolicy(profileOverride, work)
        ├─> Create temp config file
        ├─> Set CFXDEVKIT_LLM_CONFIG_PATH
        ├─> Execute work
        └─> Cleanup temp file
```

#### Message Rendering

Custom message types are registered via `pi.registerMessageRenderer()`:
- `cdk-result` (`commands-cdk.ts`) — CDK command output
- `repo-agent-summary` (`commands.ts`) — Repository command output

Both render boxed messages with tone-based coloring (`success`, `warning`, `error`, `info`).


### Provider Discovery Flow

```typescript
createPiProviderBridge(scope?)
  ↓
resolvePiConfigPath(scope)
  ├─> Check CFXDEVKIT_LLM_AGENT_CONFIG_PATH (agent-pinned)
  ├─> Check CFXDEVKIT_LLM_CONFIG_PATH (scoped)
  └─> Default: .pi/providers.json
  ↓
readPiConfig(configPath)
  ├─> Load base config
  ├─> Load scoped config (if different)
  └─> mergePiConfigLayers(base, scoped)
  ↓
resolveProviderState(config)
  ├─> Check config.baseUrl
  ├─> Check env vars (LITELLM, LEMONADE, OPENAI, GITHUB)
  ├─> Scan default URLs
  └─> discoverProviderModels(providerType, baseUrl)
  ↓
resolvePiModel(defaultModel, models)
  └─> Return model identifier
```

## Development Guidelines

### Adding New Commands

1. Define handler in `commands.ts` or `commands-cdk.ts`
2. Parse args using `tokenizeArgs()` and flag helpers
3. Call appropriate tool executor
4. Create UI state with `create*UiState()` functions
5. Emit message via `emit*Message()`
6. Add error handling with `ctx.hasUI` checks

### Adding New Tools

1. Define tool schema in `tools.ts` using `defineTool()`
2. Implement `execute` function that calls tool executor
3. Register with `pi.registerTool()`
4. Add UI state creation in `ui.ts`

### Configuration Changes

1. Update `PiLlmConfig` interface in `config-types.ts`
2. Add normalization logic in `config-normalize.ts`
3. Update `defaultPiConfig()` for new defaults
4. Test with `mergePiConfigLayers()` for override scenarios

### Provider Support

1. Add provider type to `PiLlmProviderType` union
2. Add discovery logic in `resolveProviderState()`
3. Add model catalog endpoint in `discoverProviderModels()`
4. Update `resolvePiCliInvocation()` for environment setup
