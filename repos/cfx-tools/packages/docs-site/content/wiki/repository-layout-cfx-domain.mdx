# Repository Layout — cfx-domain

# cfx-domain

**Tier 2 — Vertical Concerns**

The `cfx-domain` repository houses vertical-specific logic that may eventually be split into independent repositories. It serves as a carve-out target per [ADR-0003](../../docs/adr/0003-multi-repo-split.md).

## Repository Boundaries

To maintain architectural integrity, this module adheres to strict dependency rules:

| Dependency Type | Allowed | Forbidden |
|-----------------|---------|-----------|
| **Internal SDKs** | `@cfxdevkit/cdk`, `@cfxdevkit/wallet` | — |
| **UI/Tools** | — | `cfx-ui`, `cfx-tools` |

## Package: `@cfxdevkit/automation`

**Scope:** Off-chain automation strategies executable by `framework/executor`.

### Responsibilities
- **Strategy Interface:** Defines `evaluate`, `buildTx`, and `verify` methods.
- **Built-in Strategies:** Limit order, DCA, TWAP, condition-based swaps.
- **Persistence:** Job and execution state via `JobRepository` and `ExecutionRepository`.
- **Safety:** Pre-execution checks via `SafetyConfig` and `SafetyGuard`.
- **Backtesting:** Price history evaluation via `PriceChecker`.

### Architecture

The automation system operates on a polling model managed by the `Keeper`.

<Mermaid chart={`flowchart TD
    Keeper[Keeper] -->|Poll| JobRepo[JobRepository]
    JobRepo -->|Active Jobs| Keeper
    Keeper -->|Evaluate| Strategy[StrategyEvaluator]
    Strategy -->|Check| PriceSource[PriceSource]
    PriceSource -->|Price Data| Strategy
    Strategy -->|Triggered?| Executor[SwapExecutor]
    Executor -->|Tx| Chain[Blockchain]
    Executor -->|Record| ExecRepo[ExecutionRepository]
    ExecRepo -->|Failures| RetryQueue[RetryQueue]
    RetryQueue -->|Retry| Keeper
`} />

### Key Components

#### Strategies
Strategies define *when* and *how* to execute a job. Each implements `StrategyEvaluator<TJob>`.

- **`LimitOrderStrategy`**: Executes when price meets threshold.
- **`DCAStrategy`**: Executes recurring buys/sells based on time intervals.
- **`TWAPStrategy`**: Executes trades over a duration to average price.
- **`SwapStrategy`**: Condition-based swaps (e.g., `BELOW`/`ABOVE` threshold).

#### Execution Flow
1. **Polling:** `Keeper` queries `JobRepository` for active jobs.
2. **Evaluation:** `StrategyEvaluator` runs `evaluate()` using `PriceSource`.
3. **Construction:** If triggered, `buildTx()` prepares calldata.
4. **Execution:** `SwapExecutorClient` submits transaction via `AutomationManager`.
5. **Recording:** `ExecutionRepository` logs outcome; `RetryQueue` handles failures.

#### Safety & Reliability
- **`SafetyConfig`**: Defines thresholds (e.g., `maxSlippageBps`, `maxPriceDeviationBps`).
- **`SafetyGuard`**: Enforces constraints before execution.
- **`RetryQueue`**: Manages transient failures with exponential backoff.

### Usage

#### Create a Limit Order Job
```ts
import {
  LimitOrderStrategy,
  MemoryJobRepository,
  SwappiPriceSource,
  DEFAULT_SAFETY_CONFIG,
} from '@cfxdevkit/automation';

const priceSource = new SwappiPriceSource({
  routerAddress: SWAPPI_ADDRESSES.router,
  quoteReader: new SwappiQuoteReader(),
});

const jobRepo = new MemoryJobRepository();
const strategy = new LimitOrderStrategy(new PriceChecker(priceSource, DEFAULT_SAFETY_CONFIG));

const job = {
  id: 'limit-0x123',
  type: 'limit_order',
  params: {
    tokenIn: '0x...',
    tokenOut: '0x...',
    amountIn: 1_000_000n,
    amountOutMin: 950_000n,
    deadline: BigInt(Math.floor(Date.now() / 1000) + 3600),
  },
  strategy: strategy,
};

await jobRepo.add(job);
```

#### Run the Keeper
```ts
import { Keeper } from '@cfxdevkit/automation';

const keeper = new Keeper({
  jobRepo,
  executionRepo: new MemoryExecutionRepository(),
  retryQueue: { maxRetries: 3, baseDelayMs: 1000 },
  priceSource,
  safetyConfig: DEFAULT_SAFETY_CONFIG,
});

keeper.start();
```

## Package: `@cfxdevkit/game-engine`

**Scope:** Engine-agnostic, on-chain game primitives.

### Responsibilities
- **State Machine:** `GameEngine`, `GameState`, `Transition`.
- **Turn Management:** `dispatchAction`, `nextTurn`, `roundState`.
- **Entities:** `Inventory`, `Character`, `Item`.
- **Chain Bridge:** Syncs chain events to local state (`createStoreAdapter`).
- **Renderer Integration:** Interfaces for React, Phaser, etc. (`RendererAdapter`).

### Architecture

The engine separates core logic from rendering and chain interaction.

<Mermaid chart={`flowchart LR
    Chain[Blockchain Events] -->|Sync| ChainBridge[Chain Bridge]
    ChainBridge -->|State Update| Engine[GameEngine]
    Engine -->|State| Store[Local Store]
    Store -->|Subscribe| Adapter[RendererAdapter]
    Adapter -->|Render| UI[UI / Phaser]
    UI -->|Action| Engine
`} />

### Key Components

#### Core Engine
- **`GameEngine`**: Manages state transitions.
- **`GameState`**: Holds `currentRound`, `currentPlayerIndex`, `status`.
- **`Transition`**: Pure function `(state: GameState) => GameState`.

#### Entities
- **`Character`**: Contains `inventory` and `stats`.
- **`Inventory`**: Manages `Item` lists (`addItem`, `removeItem`).
- **`Item`**: Defined by `id`, `name`, `quantity`.

#### Adapters
- **`createStoreAdapter`**: Bridges Zustand-style stores to the engine.
- **`RendererAdapter`**: Interface for rendering `GameState`.
- **`RendererContext`**: Provides `engine` and `adapter` to renderers.

### Usage

#### Initialize Engine
```ts
import { GameEngine, createStoreAdapter } from '@cfxdevkit/game-engine';

const engine = new GameEngine();
const adapter = createStoreAdapter({
  subscribe: (listener) => engine.subscribe(listener),
  getState: () => engine.getState(),
  dispatch: (action) => engine.dispatch(action),
});
```

#### Dispatch Action
```ts
import { dispatchAction } from '@cfxdevkit/game-engine';

dispatchAction({ type: 'ATTACK', targetId: 'enemy-1' });
```

## Development Guidelines

### Adding a New Strategy (`automation`)
1. Create a new directory under `src/strategies/`.
2. Implement `StrategyEvaluator` interface.
3. Define Zod schema in `schema.ts`.
4. Export via `src/strategies/index.ts`.
5. Update `API.md` with new symbols.

### Adding a New Game Entity (`game-engine`)
1. Define interface in `src/entities/`.
2. Add state selectors in `src/state/selectors.ts`.
3. Ensure no game-specific rules (e.g., damage formulas) are included; keep logic engine-agnostic.

### Splitting a Package
When a package gains a second consumer:
1. Create a new repository in the `cfxdevkit` org.
2. Move `packages/<name>` to the new repo root.
3. Update `cfx-domain` to re-export from the new package (temporary) or remove dependency.
4. Update `ADR-0003` with the split record.
