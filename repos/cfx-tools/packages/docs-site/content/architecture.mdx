# Architecture

cfxdevkit follows a **four-tier layered architecture**. Dependencies flow downward only — no upward or circular imports are allowed.

## Tier model

```
┌──────────────────────────────────────────────────────────────────┐
│  Tier 3 — Projects & Apps                                        │
│  projects/cas  │  projects/chainbrawler  │  projects/electro     │
├──────────────────────────────────────────────────────────────────┤
│  Tier 2 — Domain Logic                                           │
│  repos/cfx-domain                                                │
├──────────────────────────────────────────────────────────────────┤
│  Tier 1 — Developer Platform                                     │
│  repos/cfx-tools                                                 │
├──────────────────────────────────────────────────────────────────┤
│  Tier 0 — Framework                                              │
│  cfx-solidity │ cfx-core │ cfx-ui │ cfx-keys                     │
└──────────────────────────────────────────────────────────────────┘
             ▲             ▲             ▲
        tools/       infrastructure/   docs/   ← cross-cutting
```

## Key principles

**One-way dependency graph** — Tiers cannot create cycles. A tier-0 package must never import from tier-1 or higher.

**Explicit public surfaces** — Every publishable package exposes an `exports` map in `package.json`. Deep imports outside the declared sub-paths are not supported.

**No circular dependencies** — Enforced by Moon task ordering and `arch-check` CI gate.

**Config packages are not tiers** — Shared config packages (`biome-config`, `tsconfig`, `moon-config`, `arch-rules`) are build-time utilities excluded from the tier model.
